<ul>
  <li>
    대표적인 학회는 RecSys(ACM Conference on Recommender Systems)이다.
  </li>
  <li>
    다른 학회로는 SIGIR(Special Interest Group on Information Retrieval), KDD(knowledge Discovery and Data Mining) 그리고 WSDM(Web Search and Data Mining)이 있다.
  </li>
</ul>

<h1>2. 편향</h1>
<h2>2-1. 선택 편향</h2>
<ul>
  <li>
    <strong>선택 편향(Selection Bias)</strong>은 사용자가 모든 아이템에 평갓값을 부여하는 것이 아니라 <strong>자신의 기호에 따라 평가 대상을 선택</strong>하기 때문에 발생하는 편향이다.
  </li>
    <ul>
      <li>
        이 경우 평갓값이 부여된 아이템이 한쪽으로 치우쳐 평갓값 분포가 왜곡된다.
      </li>
    </ul>
  <li>
    이에대한 대책으로는 다음과 같은 것들이 있다.
  </li>
    <ul>
      <li>
        <strong>Data imputation</strong>: 결손된 사용자의 아이템에 대한 평갓값을 <strong>예측해서 채운다</strong>.
      </li>
      <li>
        <strong>경향 점수 활용</strong>: 아이템의 평갓값이 부여될 확률의 <strong>경향 점수</strong>를 사용해 아이템별 평갓값의 분포 왜곡을 보정한다.
      </li>
    </ul>
</ul>

<br>

<h2>2-2. 동조 편향</h2>
<ul>
  <li>
    <strong>동조 편향(Conformity Bias)</strong>이란 <strong>다른 사용자가 부여한 평갓값의 영향</strong>을 받아 자신이 부여하는 평갓값이 바뀌는 편향이다.
  </li>
    <ul>
      <li>
        즉, 사용자가 부여한 평갓값이 반드시 <strong>사용자 본인의 기호를 반영했다고 단정할 수 없다</strong>.
      </li>
    </ul>
  <li>
    동조 편향에 대한 대안으로는 다음과 같은 방법들이 있다.
  </li>
    <ul>
      <li>
        아이템의 평갓값을 <strong>인기도에 따라 보정</strong>한다.
      </li>
      <li>
        학습하거나 예측할 때 사용자의 선호도, 사용자의 평갓값 부여 상황 등 <strong>사용자 자신의 특징을 알 수 있는 보조 정보</strong>를 사용한다.
      </li>
    </ul>
</ul>

<br>

<h2>2-3. 노출 편향</h2>
<ul>
  <li>
    <strong>노출 편향(Exposure Bias)</strong>은 사용자에게 표시되는 아이템이 추천 후보 아이템 중 <strong>일부인 것</strong>에 기인한 편향이다.
  </li>
    <ul>
      <li>
        이 경우 사용자의 기호에 맞지만 노출되지 않은 아이템이 있을 수 있기에 주의해야 한다.
      </li>
    </ul>
  <li>
    노출 편향에 대한 대응으로는 다음과 같은 방법들이 있다.
  </li>
    <ul>
      <li>
        <strong>평가를 위한 대책</strong>: 아이템을 추천하는 <strong>확률의 경향 점수</strong>를 사용해 평가 지표에 가중치를 부여한다.
      </li>
      <li>
        <strong>학습을 위한 대책</strong>: 서비스에 대한 도메인 지식 등을 구사해 휴리스틱(Heuristic)으로 <strong>학습 손실에 가중치</strong>를 부여한다. 혹은 사용자나 아이템의 보조 정보 등을 활용해 휴리스틱으로 <strong>다운 샘플링</strong> 한다.
      </li>
    </ul>
</ul>

<br>

<h2>2-4. 위치 편향</h2>
<ul>
  <li>
    <strong>위치 편향(Position Bias)</strong>란 사용자가 <strong>상위 순위의 아이템</strong>에 더 많은 행동을 보이는 편향이다.
  </li>
  <li>
    위치 편향에 대한 대안으로는 다음과 같은 방법들이 있다.
  </li>
    <ul>
      <li>
        <strong>클릭 모델</strong>을 사용하여 클릭에 관한 행동을 모델링 한다.
      </li>
      <li>
        순위에서 <strong>위치와 관련된 아이템의 표시 확률에 관한 경향 점</strong>를 활용해 보정한다. 표시 확률을 산출할 때에는 아이템을 사용자에게 무작위로 표출시키는 <strong>결과 무작위화(Result Randomization)</strong>가 단순하면서 효과적인 것으로 알려져 있다.
      </li>
    </ul>
</ul>

<br>

<h2>2-5. 인지적 편향</h2>
<ul>
  <li>
    <strong>인지적 편향(Popularity Bias)</strong>은 <strong>인기 아이템</strong>이 다른 아이템에 비해 추천될 기회가 많아지는 편향이다.
  </li>
  <li>
    인지적 편향에 대한 대안으로는 다음과 같은 방법들이 있다.
  </li>
    <ul>
      <li>
        학습에 인지도 편향에 관한 <strong>정규화항</strong>을 삽입한다.
      </li>
      <li>
        <strong>니치(niche)한 아이템의 추천 기회</strong>를 늘리는 것을 목표로 하여 학습을 수행한다.
      </li>
    </ul>
</ul>

<br>

<h2>2-6. 불공평성</h2>
<ul>
  <li>
    <strong>불공평성(Unfairness)</strong>은 다른 집단에 비해 <strong>특정 개인이나 개인 그룹이 유리해지거나 불리해지도록</strong> 부당하게 구분하는 편향이다.
  </li>
  <li>
    불공평성에 대한 대안으로는 다음과 같은 방법들이 있다.
  </li>
    <ul>
      <li>
        학습 데이터 라벨이나 추천 결과의 <strong>다양성에 대한 균형</strong>을 조정한다.
      </li>
      <li>
        학습에 <strong>공평성의 기준에 관한 항목</strong>을 삽입한다.
      </li>
      <li>
        인종이나 성별 등 <strong>민감한 속성은 가능한 한 사용하지 않도록 학습</strong>한다.
      </li>
    </ul>
</ul>

<br>

<h2>2-7. 피드백 루프에 의한 편향 증폭</h2>
<ul>
  <li>
    추천 시스템의 각 프로세스에서 편향은 <strong>서로 영향</strong>을 미치며 편향을 증폭시키기도 한다.
  </li>
  <li>
    피드백 루프에 의한 편향 증폭은 다음과 같은 대안이 존재한다.
  </li>
    <ul>
      <li>
        추천 대상 아이템 일부를 <strong>무작위로 선택</strong>하여 편향 루프를 줄인다.
      </li>
      <li>
        <strong>강화 학습 프레임워크</strong>를 활용해 효율적으로 편향이 없는 아이템 탐색을 사용자에게 수행하도록 한다.
      </li>
    </ul>
</ul>

<br><br>

<h1>3. 상호 추천 시스템</h1>
<ul>
  <li>
    아이템을 서비스 사용자에게 일반적으로 추천하는 방식을 <strong>Item-to-User</strong> 형식의 서비스라한다. 반대로 서비스 내의 사용자를 상호 추천하는 서비스를 <strong>User-to-User</strong>이라 한다.
  </li>
    <ul>
      <li>
        User-to-User 형식 서비스에서의 추천 시스템을 상호 추천 시스템을 <strong>상호 추천 시스템(Reciprocal Recommender Systems)</strong>이라 한다.
      </li>
    </ul>
  <li>
    상호 추천 시스템에서 가장 큰 특징은 해당 시스템의 <strong>추천 성공 여부 기준</strong>이다.
  </li>
    <ul>
      <li>
        상호 추천 시스템은 한 사용자는 선호이지만 반대로 다른 사용자는 비선호가 될 수 있으며 이 경우가 애매하기 때문이다.
      </li>
    </ul>
</ul>

<br>

<h2>3-1. 기존 추천 시스템과 상호 추천 시스템 비교</h2>
<h3>서비스 내 사용자의 역할</h3>
<ul>
  <li>
    기존 추천 시스템에서 사용자는 항상 호의를 보내는 입장이고 아이템은 항상 호의를 받는 입장이다.
  </li>
  <li>
    상호 추천 시스템에서는 양측이 모두 사용자이기 때문에 양측의 호의를 모두 고려해야 한다.
  </li>
</ul>

<h3>서비스상의 사용자 프로필이나 선호에 관한 정보</h3>
<ul>
  <li>
    기존 추천 시스템에서는 사용자 프로필 정보가 필수가 아니라는 점, 사용자도 자신의 기호를 잘 모를 수도 있다는 점 등으로 인해 제대로된 추천이 어려울 수 있다.
  </li>
  <li>
    반면 상호 추천 시스템의 서비스는 자신의 프로필과 기호를 적극적으로 제공하는 경향이 있다. 단, 자신의 더 부풀리는 경향이 있기에 주의해야 한다.
  </li>
  <li>
    따라서 사용자가 명시적으로 제공하는 프로필이나 선호도 관련 정보와 <strong>실제 선호도 사이에 차이</strong>가 있음을 이해한 상태에서 시스템을 설계하는 것이 중요하다.
  </li>
</ul>

<br>

<h3>성능 평가와 성공 기준</h3>
<ul>
  <li>
    기존 추천 시스템에서는 아이템을 추천 받은 사용자에만 집중했지만 상호 추천 시스템에서는 호의를 보내는 측, 호의를 받는 측 혹은 둘 모두를 고려한 평가 기준을 활용할 수도 있다.
  </li>
  <li>
    기존 추천 시스템에서는 어느 정도 정밀도가 낮아도 새로운 아이템 등에 대한 기회가 될 수 있으나 상호 추천 시스템은 지속적인 호의 거절로 이어질 수 있기에 낮은 정밀도를 주의해야 한다.
  </li>
</ul>

<br><br>

<h1>4. 업리프트 모델링</h1>
<ul>
  <li>
    추천을 하지 않아도 구입되는 상품이 추천을 통해 구입으로 이어지는 노이즈가 발생할 수 있다. 이 경우 업리프트(uplift)를 활용할 수 있다.
  </li>
  <li>
    업리프트 패턴은 다음과 같다.
  </li>
    <ul>
      <li>
        <strong>True Uplift(TU)</strong>: 추천하면 구입하고, 추천하지 않으면 구입하지 않는다.
      </li>
      <li>
        <strong>False Uplift(FU)</strong>: 추천 여부에 관계없이 구입한다.
      </li>
      <li>
        <strong>True Drop(TD)</strong>: 추천하지 않으면 구입하고, 추천하면 구입하지 않는다.
      </li>
      <li>
        <strong>False Drop(FD)</strong>: 추천 여부와 관계없이 구입하지 않는다.
      </li>
    </ul>
  <li>
    업리프트 평가 지표에 대한 기본적인 사고 방식은 '<strong>Uplift Score = 추천 시 평균 구입 확률 - 비추천 시 평균 구입 확률</strong>'이다.
  </li>
  <li>
    추천 시 평균 구입 확률과 비추천 시 평균 구입 확률을 추천 모델 출력으로부터 얻어지는 로그로 측정하면 <strong>모델의 편향에 따라 치우침</strong>이 발생할 수 있는데 이는 <strong>인과 추론 프레임인 경향 점수</strong>를 통해 보정하도록 대책을 세울 수도 있다.
  </li>
  <li>
    업리프트의 최종 목표는 <strong>TU의 비율</strong>을 높이는 것이다.
  </li>
  <li>
    단, TU, FU, TD, FD는 실제로 관측이 불가하기에 다음과 같이 <strong>추천 수행 여부</strong>와 <strong>구입 발생 여부</strong>라는 두 개의 측으로 라벨링해 세분화 한다.
  </li>
    <ul>
      <li>
        R-P(Recommend-Purchase): 추천해서 구입이 발생한 경우. (TU, FU)
      </li>
      <li>
        R-NP(Recommend-Not Purchase): 추천했으나 구입이 발생하지 않은 경우. (TD, FD).
      </li>
      <li>
        NR-P(Not Recommend-Purchase): 추천하지 않았으나 구입이 발생한 경우(FU, TD).
      </li>
      <li>
        NR-NP(NotRecommend-Not Purchase): 추천하지 않았고 구매도 발생하지 않은 경우. (TU, FD)
      </li>
    </ul>
  <li>
    R-P, R-NP, NR-P, NR-NP 클래스에 맞추어 +, - 라벨을 부여한다.
  </li>
    <ul>
      <li>
        + 라벨은 구입이 발생한 경우로 R-P와 NR-P가 해당한다. 단, 업리프트에서는 TU를 포함하는 클래스인 R-P와 NR-NP가 +가 된다.
      </li>
      <li>
        NR-NP 데이터는 다른 클래스에 비해 매우 많기에 학습이 잘 안될 우려가 있다. 따라서 NR-NP 데이터로부터 샘플링 확률을 줄인 상태에서의 학습 등을 고려할 수 있다.
      </li>
    </ul>
</ul>

<h1>5. 도메인에 따른 특징과 과제</h1>
<h2>5-1. 음악 추천</h2>
<h3>끝 없는 재생(Endless Play)</h3>
<ul>
  <li>
    하나의 음악을 다 들었을 때 새로운 음악을 들으려 하지만 몇 곡을 들을지는 정해두지 않음.
  </li>
</ul>

<h3>카탈로그 문제(Catalog Problem)</h3>
<ul>
  <li>
    음악 메타 데이터에 잘못된 정보나 누락이 포함될 수 있기에 데이터 정형에 큰 노력이 필요하다.
  </li>
</ul>

<h3>사용자 선호도 바영</h3>
<ul>
  <li>
    음악 취향은 변할 수 있기에 사용자의 선호도를 빠르고 정확하게 찾는 것이 과제이다.
  </li>
</ul>

<h3>반복되는 소비</h3>
<ul>
  <li>
    어떤 음악이 언제 여러 차례 소비되는지 특정하는 것은 어려우며 음악 특유의 과제에 해당한다.
  </li>
</ul>

<h3>니치한 음악에 대한 수요</h3>
<ul>
  <li>
    음악은 다소 니치한(소수 취향이지만 열정적인 팬들) 경우가 많기에 단순 인기만을 기준으로 하면 사용자 만족도가 낮아질 수 있다.
  </li>
</ul>

<h3>콘텍스트 의존성</h3>
<ul>
  <li>
    음악은 콘텍스트(주변 상황 혹은 분위기)에 의존하기에 사용자의 콘텍스트를 파악해 두는 게 중요하다
  </li>
</ul>

<h3>사회적 영향</h3>
<ul>
  <li>
    어떤 음악을 듣는가는 사용자가 있는 환경적인 영향 또한 받는다.
  </li>
</ul>

<h2>5-2. 패션 추천</h2>
<h3>인플루언서를 경유하여 구입</h3>
<ul>
  <li>
    SNS의 인플루언서를 통한 패션 구매로 이어질 수 있다.
  </li>
</ul>

<h3>유사 상품 구입</h3>
<ul>
  <li>
    패션의 경우 구매에 비해 너무 많은 아이템이 있기에 콘텐츠 자체의 데이터를 최대한 활용한다.
  </li>
  <li>
    특히 이미지 데이터를 활용한 추천이 활발히 이루어지는 편이다.
  </li>
</ul>

<h3>상품 크기나 착용감을 고려하여 상품 선택</h3>
<ul>
  <li>
    의류 상품을 온라인으로 열람하는 경우 사이즈를 정확히 알기 어렵다는 문제가 있기에 주의해야 한다.
  </li>
</ul>

<br>

<h2>5-3. 뉴스 추천</h2>
<h3>콜드 스타트 문제</h3>
<ul>
  <li>
    뉴스 서비스에서는 특히 처음 방문하는 사용자에게 적합한 기사를 선출하여 추천하기 어렵다는 문제가 존재한다.
  </li>
</ul>

<h3>시의성</h3>
<ul>
  <li>
    뉴스는 기사가 무엇이냐에 따라 시간이 지나면서 가치가 달라진다. 따라서 시간에 따른 뉴스의 가치를 모델링할 필요가 있다.
  </li>
</ul>

<h3>구조화되지 않은 문서</h3>
<ul>
  <li>
    콘텐츠의 내용을 깊이 이해할수록 제대로 추천할 수 있지만 언제나 뉴스 기사가 완전한 형식으로 작성되는 것은 아님을 주의해야 한다.
  </li>
</ul>

<h3>중복 해제</h3>
<ul>
  <li>
    추천 시스템은 같은 내용의 기사를 다른 기사로 인식하고 추천을 수행할 수도 있다는 것을 주의해야 한다.
  </li>
</ul>

<h3>페이크 뉴스(fake news)나 클릭베이트(clickbait)</h3>
<ul>
  <li>
    뉴스에는 악의를 갖고 허위 정보로 작성된 disinformation이 있을 수 있으며 또 확인 부족 등 부정확한 misinformation이 있을 수 있음을 주의해야 한다.
  </li>
  <li>
    이 경우에는 추천을 받은 사용자에게 피드백을 받아 수정할 수 있도록 설계를 해두는 것이 중요하다.
  </li>
</ul>

<br>

<h2>5-4. 동영상 추천</h2>
<h3>그룹 시청</h3>
<ul>
  <li>
    그룹 시청의 경우 시청을 하는 사람이 주기적으로 달라지기 때문에 특정 시청 패턴을 확인하여 현재 시청하고 있는 사람에 적합한 추천이 되도록  모델링하는 것이 필요하다.
  </li>
</ul>

<h3>설명의 중요성</h3>
<ul>
  <li>
    동영상을 보는 데에는 적지 않은 시간이 할애되기에 '당신에게 추천' 등의 문구로 사용자가 자신에게 적합한 아이템임을 인지하도록 하는 것이 좋다.
  </li>
</ul>

<h3>개인화</h3>
<ul>
  <li>
    개인에게 맞춘 섬네일 이미지로 추천을 진행할 수 있다.
  </li>
  <li>
    리포콘 등의 검색으로 조작을 하는 경우 더 적은 조작만으로 적합한 아이템이 노출되도록 추천을 구현할 수도 있다.6
  </li>
</ul>
