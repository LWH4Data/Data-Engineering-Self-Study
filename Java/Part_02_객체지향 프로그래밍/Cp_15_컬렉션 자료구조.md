<h1>1. 컬렉션 프레임워크</h1>
<ul>
  <li>
    자바는 널리 알려져 있는 <strong>자료구조(Data Structure)</strong>를 사용할 수 있도록 <strong>java.util 패키지</sstrong>에 포함시켜 두었으며 이들을 총칭하여 <strong>컬렉션 프레임워크(Collection Framework)</strong>라 부른다.
  </li>
  <li>
    컬렉션 프레임워크에서는 몇 가지 인터페이스를 제공하는데 주요 인터페이스로는 List, Set, Map이 있으며 각각 컬력션 객체를 제공한다. (p642 참고).
  </li>
</ul>

<br><br>

<h1>2. List 컬렉션</h1>
<ul>
  <li>
    List 컬렉션은 객체를 <strong>인덱스</strong>로 관리하며 객체를 저장하면 <strong>인덱스가 부여</strong>되고 <strong>인덱스로 객체를 검색, 삭제할 수 있는 기능</strong>을 제공한다.
  </li>
  <li>
    List 컬렉션에서는 ArrayList, Vector, LinkedList 등을 제공하며 자세한 인터페이스 메소드들은 p643을 참고.
  </li>
</ul>

<br>

<h2>2-1. ArrayList</h2>
<ul>
  <li>
    ArrayList는 객체를 추가하면 <strong>내부 배열</strong>에 객체가 저장된다.
  </li>
  <li>
    일반 배열과의 차이점은 ArrayList는 <strong>제한 없이 객체를 추가</strong>할 수 있다는 점이다.
  </li>
  <li>
    List 컬렉션은 객체 자체를 저장하는 것이 아니라 <strong>객체의 번지</strong>를 저장한다.
  </li>
    <ul>
      <li>
        동일한 객체를 중복 저장할 때에도 값이 아닌 <strong>동일한 번지</strong>가 저장된다.
      </li>
      <li>
        <strong>null</strong> 또한 저장이 가능하다.
      </li>
    </ul>
  <li>
    ArrayList에 객체를 추가하는 경우 인덱스 <strong>0부터 차례로 저장</strong>된다.
  </li>
    <ul>
      <li>
        특정 인덱스 객체를 제거하면 <strong>바로 뒤 인덱스 ~ 마지막 인덱스</strong>까지 모두 1씩 당겨진다.
      </li>
      <li>
        특정 인덱스에 객체를 추가하면 <strong>해당 인덱스 ~ 마지막 인덱스</strong>까지 모두 1씩 밀려난다.
      </li>
    </ul>
  <li>
    ArrayList는 객체를 삽입 및 삭제할 때 밀리거나 당겨지기 때문에 <strong>잦은 변환</strong>이 있을 때에는 적합하지 않다. 이때에는 <strong>LinkedList</strong>를 사용하는 것이 좋다.
  </li>
</ul>

```java
// 1. ArrayList 컬렉션 생성.
// 1-1. E에 지정된 타입의 객체만 지정.
//   - 파라미터 E에 저장하고 싶은 객체 타입을 지정.
List<E> list = new ArrayList<E>();
// 1-2. E에 지정된 타입의 객체만 지정.
//   - 파라미터 E에 저장하고 싶은 객체 타입을 지정.
//   - List에 지정한 객체 타입과 동일하면 E 생략 가능.
List<E> list = new ArrayList<>();
// 1-3. 모든 타입의 객체를 저장.
//   - 모든 종류의 객체 저장 가능.
LIst list = new ArrayList();
```

<br><br>

<h2>2-2. Vector</h2>
<ul>
  <li>
    Vector는 ArrayList와 동일한 내부 구조를 갖지만, Vector는 <strong>동기화된(synchronized) 메소드</strong>로 구성되어 있다.
  </li>
    <ul>
      <li>
        따라서 멀티 스레드가 <strong>동시에 Vector() 메소드</strong>를 실행할 수 없다.
      </li>
    </ul>
</ul>

```java
// 1. Vector 생성.
// 1-1. E에 지정된 타입의 객체만 저장.
//   - E에는 Vector에 저장하고 싶은 객체 타입을 지정.
List<E> list = new Vector<E>();
// 1-2. E에 지정된 타입의 객체만 저장.
//   - E에는 Vector에 저장하고 싶은 객체 타입을 지정.
//   - List에 지정한 객체 타입과 동일하다면 객체 타입 생략 가능.
List<E> list = new Vector<>();
// 1-3. 모든 타입의 객체를 저장.
List list = new Vector();
```

<br>

<h2>2-3. LinkedList</h2>
<ul>
  <li>
    ArrayList는 내부 배열에 객체를 저장하지만, LinkedList는 <strong>인접 객체를 체인처럼 연결</strong>하여 관리한다.
  </li>
  <li>
    LinkedList는 객체의 삽입, 삭제에 있어 <strong>앞뒤의 링크만 변경</strong>하면 되기에 빈번한 객체 삭제와 삽입이 일어나는 곳에서는 ArrayList보다 성능이 좋다.
  </li>
</ul>

```java
// 1. LinkedList 초기화.
// 1-1. E에 지정된 타입의 객체만 저장.
List<E> list = new LinkedList<E>();
// 1-2. E에 지정된 타입의 객체만 저장.
List<E> list = new LinkedList<>();
// 1-3. 모든 타입의 객체를 저장.
List list = new LinkedList();
```

<br><br>

<h1>3. Set 컬렉션</h1>
<ul>
  <li>
    Set 컬렉션은 <strong>저장 순서</strong>가 유지되지 않는다.
  </li>
  <li>
    또한 객체를 <strong>중복</strong>해서 저장할 수 없으며 <strong>단 하나의 null</strong>만 저장할 수 있다.
  </li>
  <li>
    Set은 객체를 <strong>저장할 때</strong>와 <strong>찾을 때</strong>의 <strong>순서</strong>가 다를 수도 있다.
  </li>
  <li>
    Set 컬레션에는 대표적으로 HashSet, LinkedHashSet, TreeSet 등이 있다.
  </li>
    <ul>
      <li>
        Set의 인터페이스 메소드는 p652 참고. 
      </li>
      <li>
        인덱스를 사용하지 않기 때문에 인덱스를 매개변수로 갖는 메소드가 없다.
      </li>
    </ul>
</ul>

<br>

<h2>3-1. HashSet</h2>
<ul>
  <li>
    Set 컬렉션 중 가장 많이 사용된다.
  </li>
  <li>
    HashSet은 주소가 같은 동일 객체가 아닌 <strong>값이 같은 동등 객체</strong>를 중복 저장하지 않는다.
  </li>
    <ul>
      <li>
        HashSet은 다른 객체라도 <strong>hashCode() 메소드의 리턴 값</strong>이 같고
        <br>→ <strong>equals() 메소드가 true를 리턴</strong>하면 동일한 객체라고 판단한다.
      </li>
    </ul>
  <li>
    Set 컬렉션은 인덱스가 없기에 <strong>객체를 한 개씩 반복해서 가젹오기</strong> 위해서는 두 가지 방법이 있다.
  </li>
    <ul>
      <li>
        for 문을 사용해 가져오기.
      </li>
      <li>
        Set 컬렉션의 iterator() 메소드로 반복자(iterator)를 얻어 사용하기.
      </li>
        <ul>
          <li>
            <strong>hasNext()</strong>: 반환 타입은 boolean으로 가져올 객체가 있다면 true를, 없다면 false를 리턴한다.
          </li>
          <li>
            <strong>next()</strong>: 반환 타입은 E(선언한 타입)으로 컬렉션에서 하나의 객체를 가져온다.
          </li>
          <li>
            <strong>remove()</strong>: 반환 타입은 void로 next()로 가져온 객체를 Set 컬렉션에서 제거한다.
          </li>
        </ul>
    </ul>
</ul>

```java
// 1. HashSet 초기화.
// 1-1. 타입 파라미터 E에는 HashSet에 저장하려는 객체 타입을 지정한다.
Set<E> set = new HashSet<E>();
// 1-2. Set에 지정한 타입과 동일하다면 HashSet<>처럼 객체 타입을 생략할 수 있다.
Set<E> set = new HashSet<>();
// 1-3. 객체 타입을 모두 생략하면 모든 종류의 객체를 저장할 수 있다.
Set set = new HashSet();


// 2. HashSet의 객체를 하나씩 가져오기.
// 2-1. for loop
Set<E> set = new HashSet<>();
for(E e : set) {
  ...
}
// 2-2. Set 컬렉션의 iterator() 메소드.
Set<E> set = new HashSet<>();
Iterator<E> iterator = set.iterator();
whlie(iterator.hasNext()) {
  E e = iterator.next();
}
```

<h1>4. Map 컬렉션</h1>
<ul>
  <li>
    Map 컬렉션은 <strong>키(key)</strong>와 <strong>값(value)</strong>으로 구성된 <strong>엔트리(Entry) 객체</strong>를 저장한다.
  </li>
    <ul>
      <li>
        키와 값은 모두 <strong>객체</strong>이다.
      </li>
      <li>
        키는 <strong>중복 저장</strong>할 수 없지만, 값은 <strong>중복 저장</strong>이 가능하다.
      </li>
      <li>
        기존에 저장된 키와 동일한 키로 값을 저장하면 기존의 값은 없어지고 <strong>새로운 값으로 대치</strong>된다.
      </li>
    </ul>
  <li>
    Map 컬렉션에는 HashMap, Hashtable, LinkedHashMap, Properties, TreeMap 등이 있다.
  </li>
  <li>
    Map 컬렉션에서 공통으로 사용 가능한 Map 인터페이스 메소드는 p658을 참고.
  </li>
    <ul>
      <li>
        키로 객체들을 관리하기에 <strong>키를 매개값</strong>으로 갖는 메소드가 많다.
      </li>
    </ul>
  <li>
    K는 key 타입을, V는 value 타입을 의미한다.
  </li>
</ul>

<br>

<h2>4-1. HashMap</h2>
<ul>
  <li>
    키로 사용할 객체가 <strong>hashCode()</strong> 메소드의 리턴값이 같고 <strong>equals()</strong> 메소드가 true를 리턴할 경우 <strong>동일 키</strong>로 보고 중복 저장을 허용하지 않는다.
  </li>
</ul>

```java
// 1. HashMap 초기화.
// 1-1. 타입 지정 생성.
Map<String, Integer> map = new HashMap<String, Integer>();
// 1-2. 동일 타입 생략.
Map<String, Integer> map = new HashMap<>();
// 1-3. 모든 타입 허용(거의 사용하지 않는다).
Map map = new HashMap();
```

<br>

<h2>4-2. Hashtable</h2>
<ul>
  <li>
    Hashtable은 HashMap과 동일한 내부 구조를 가지고 있다. 단, Hashtable은 <strong>동기화된(synchronized) 메소드</strong>로 구성되어 있어 <strong>멀티 스레드가 동시에 실행할 수 없다</strong>.
  </li>
  <li>
    즉, 멀티 스레드 환경에서도 안정적으로 객체를 추가, 삭제할 수 있다.
  </li>
</ul>

```java
// 1. Hashtable 초기화.
Map<String, Integer> map = new Hashtable<String, Integer>();
Map<String, Integer> map = new Hashtable<>();
Map map = new Hashtable();
```

<br>

<h2>4-3. Properties</h2>
<ul>
  <li>
    Properties는 Hashtable의 ><strong>자식 클래스</strong>이며 Hashtable의 특징을 그대로 갖는다.
  </li>
    <ul>
      <li>
        Properties는 키와 값을 <strong>String 타입</strong>으로 제한한 컬렉션이다. 
      </li>
      <li>
        Properties는 주로 <strong>확장자가 .properties</strong>인 프로퍼티 값을 읽을 때 사용한다.
      </li>
        <ul>
          <li>
            프로퍼티 파일이란 키와 값이 <strong>= 기호</strong>로 연결된 텍스트 파일을 의미한다.
          </li>
            <ul>
              <li>
                일반 텍스트 파일과는 다르게 ISO 8859-1 문자셋으로 저장되며, 한글은 \u+유니코드로 표현되어 저장한다.
              </li>
            </ul>
        </ul>
    </ul>
  <li>
    일반적으로 프로퍼티 파일은 <strong>클래스 파일(*.java)</strong>들과 함께 저장된다.
  </li>
    <ul>
      <li>
        따라서 클래스 파일 기준으로 <strong>상대 경로</strong>를 이용해 읽는 것이 편리하다.
      </li>
    </ul>
</ul>

```java
// 1. Properties 객체 생성.
Properties properties  new Properties();
// load() 메소드로 포로퍼티 파일의 내용을 메모리에 로드한다.
properties.load(Xxx.class.getResourceAsStream("database.properties"));
```

<h1>5. 검색 기능을 강화시킨 컬렉션</h1>
<ul>
  <li>
    컬레션 프레임워크는 검색 기능을 강화시킨 <strong>TreeSet</strong>과 <strong>TreeMap</strong>을 제공한다.
  </li>
    <ul>
      <li>
        TreeSet은 Set 컬렉션이고, TreeMap은 Map 컬레션이다.
      </li>
    </ul>
</ul>

<br>

<h2>5-1. TreeSet</h2>
<ul>
  <li>
    TreeSet은 <strong>이진 트리(binary tree)</strong>를 기반으로 한 Set 컬렉션이다.
  </li>
    <ul>
      <li>
        이진 트리란 <strong>여러 개의 노드(node)</strong>가 <strong>트리 형태</strong>로 연결된 구조이다.
      </li>
      <li>
        <strong>루트 노드(root node)</strong>라 불리는 하나의 노드에서 시작하여 <strong>각 노드에 최대 2개</strong>의 노드를 연결할 수 있는 자료 구조를 갖는다.
      </li>
    </ul>
  <li>
    TreeSet에 객체를 저장하면 <strong>자동으로 정렬</strong>된다.
  </li>
    <ul>
      <li>
        부모 노드와 비교하여 <strong>작은 것</strong>은 왼쪽 자식 노드, <strong>큰 것</strong>은 오른쪽 자식 노드로 정렬된다.
      </li>
    </ul>
  <ul>
    <li>
      TreeSet이 갖고 있는 검색 메소드들은 p667을 참고.
    </li>
  </ul>
</ul>

```java
// 1. TreeSet 컬렉션 생성.
TreeSet<E> treeSet = new TreeSet<E>();
TreeSet<E> treeSet = new TreeSet<>();
```

<br>

<h2>5-2. TreeMap</h2>
<ul>
  <li>
    TreeMap은 이진 트리를 기반으로 한 <strong>Map 컬레션</strong>이다.
  </li>
    <ul>
      <li>
        TreeSet과의 차이점은 키와 값이 저장된 <strong>Entry</strong>를 저장한다는 것이다.
      </li>
      <li>
        TreeMap에 Entry를 저장하면 <strong>키를 기준</strong>으로 자동 정렬된다.
      </li>
        <ul>
          <li>
            부모 키와 비교하여 작은 것은 왼쪽, 큰 것은 오른쪽.
          </li>
        </ul>
    </ul>
  <li>
    Map 타입이 아닌 TreeMap을 굳이 사용하는 이유는 <strong>검색 관련 메소드</strong>가 TreeMap에만 정의되어 있기 때문이다.
  </li>
    <ul>
      <li>
        검색 관련 메소드는 p671 참고.
      </li>
    </ul>
</ul>

```java
// 1. TreeMap 컬렉션 생성.
TreeMap<K, V> treeMap = new TreeMap<K, V>();
TreeMap<K, V> treeMap = new TreeMap<>();
```

<br>

<h2>5-3. Comparable과 Comparator</h2>
<ul>
  <li>
    TreeSet과 TreeMap에 저장되는 키 객체는 저장과 동시에 오름차순 정렬되는데 이는 객체가 <strong>Comparable 인터페이스</strong>를 구현하고 있기 때문이다.
  </li>
    <ul>
      <li>
        Integer, Double, String 타입은 모두 기본 구형되어 있기에 <strong>사용자 정의 객체</strong>를 저장할 때에 주의해야 한다.
      </li>
    </ul>
  <li>
    Comparable 인터페이스의 <strong>compareTo() 메소드</strong>를 재정의하여 사용자 정의 클래스에서 비교 결과를 <strong>정수 값으로 리턴</strong>해야한다.
  </li>
    <ul>
      <li>
        주어진 객체와 같으면 0, 적으면 음수, 크면 양수를 리턴.
      </li>
    </ul>
  <li>
    원칙적이지 않지만 <strong>비교 기능이 없는 Comparable 비구현 객체</strong>를 저장하고 싶다면 TreeSet과 TreeMap을 생성할 때 <strong>비교자(Comparator)를 제공</strong>하는 방법이 있다.
  </li>
    <ul>
      <li>
        비교자란 Comparator 인터페이스를 구현한 객체이다.
      </li>
        <ul>
          <li>
            Comparator 인터페이스에는 compare() 메소드가 정의되어 있다.
          </li>
          <li>
            비교자는 비교 결과를 정수 값으로 리턴한다.
          </li>
          <li>
            메소드는 int compare(T o1, To2) 이다.
          </li>
        </ul>
    </ul>
</ul>

```java
// 1. 비교 기능이 없는 Comparable 비구현 객체 저장.
//   - 비교자를 제공하는 방법. (ComparatorImpl() 사용).
TreeSet<E> treeSet = new TreeSet<E>( new ComparatorImpl() );
TreeMap<K, V> treeMap = new TreeMap<K, V>( new ComparatorImpl());
```

<h1>6. LIFO와 FIFO 컬렉션</h1>
<ul>
  <li>
    컬렉션 프레임워크에는 <strong>후입선출(LIFO, Last In First Out)을 구현한 스택(Stack)</strong>과 <strong>선입선출(FIFO, First In First Out)을 구현한 큐(Queue)</strong>가 있다.
  </li>
    <ul>
      <li>
        스택의 대표적인 예는 JVM의 스택 메모리가 있다.
      </li>
      <li>
        큐의 대표적인 예로는 스레드풀(EcecutorService)의 작업 큐가 있다.
      </li>
    </ul>
</ul>

<br>

<h2>6-1. Stack</h2>
<ul>
  <li>
    Stack 클래스는 LIFO 자료구조를 구현한 클래스이다.
  </li>
  <li>
    메소드는 두 가지로 <strong>push(E item)</strong>을 통해 주어진 객체를 스택에 넣는 것과 <strong>pop()</strong>을 통해 스택의 맨 위 객체를 빼내는 것이 있다.
  </li>
</ul>

```java
// 1. Stack 자료구조 생성.
Stack<E> stack = new Stack<E>();
Stack<E> stack = new Stack<>();
```

<br>
<h2>6-2. Queue</h2>
<ul>
  <li>
    Queue 인터페이스는 <strong>FIFO 자료구조</strong>에서 사용되는 메소드를 정의하고 있다.
  </li>
  <li>
    Queue 인터페이스에는 다음 두 가지 메소드가 정의되어 있다.
  </li>
    <ul>
      <li>
        <strong>boolean offer(E e)</strong>는 주어진 객체를 큐에 넣는다.
      </li>
      <li>
        <strong>E poll()</strong> 메소드는 큐에서 객체를 빼낸다.
      </li>
    </ul>
  </li>
</ul>

```java
// 1. LinkedList 객체에 Queue 메소드 활용.
Queue<E> queue = new LinkedList<E>();
Queue<E> queue = new LinkedList<>();
```

<h1>7. 동기화된 컬렉션</h1>
<ul>
  <li>
    컬렉션의 프레임워크의 대부분의 클래스는 <strong>싱글 스레드 환경</strong>을 전제로 설계되었다.
  </li>
    <ul>
      <li>
        따라서 스레드가 동시에 접근한다면 의도치 않게 요소가 변경될 수 있는 불완전한 상태가 된다.
      </li>
      <li>
        Vector와 Hashtable은 동기화된(synchronized) 메소드로 구성되어 있어 안정적이다.
      </li>
    </ul>
  <li>
    경우에 따라 ArrayList, HashSet, HashMap을 멀티 스레드 환경에서 사용하고 싶은 경우 <strong>Collections의 synchronizedXXX() 메소드</strong>를 사용한다.
  </li>
    <ul>
      <li>
        비동기화된 메소드를 <strong>동기화된 메소드로 래핑</strong>하는 역할을 한다.
      </li>
      <li>
        비동기화된 컬레션을 대입하면 <strong>동기화된 컬렉션을 리턴</strong>한다.
      </li>
    </ul>
</ul>

```java
// 1. 메소드 활용.
List<T> list = Collections.synchronizedList(new ArrayList<T>());
Set<E> set = Collections.synchronizedSet(new HashSet<E>());
Map<K, V> map = Collections.synchronizedMap(new HashMap<K, V>());
```

<h1>8. 수정할 수 없는 컬렉션</h1>
<ul>
  <li>
    <strong>수정할 수 없는(unmodifiable) 컬렉션</strong>이란 요소를 추가, 삭제할 수 없는 컬렉션을 의미한다.
  </li>
    <ul>
      <li>
        컬렉션 생성 시 저장된 요소를 변경하고 싶지 않을 때 유용하다.
      </li>
    </ul>
  <li>
    수정할 수 없는 컬렉션을 생성하는 방법은 대표적으로 List, Set, Map 인터페이스의 정적 메소드인 <strong>of()</strong>로 생성하는 방법이 있다.
  </li>
    <ul>
      <li>
        두 번째 방법은 List, Set, Map 인터페이스의 정적 메소드인 <strong>copyOf()</strong>을 이용해 <strong>기존 컬렉션을 복사</strong>하여 수정할 수 없는 컬렉션을 만드는 방법이다.
      </li>
      <li>
        세 번째 방법은 <strong>배열</strong>로부터 수정할 수 없는 List 컬렉션을 만드는 것이다.
      </li>
    </ul>
</ul>

```java
// 1. of()를 통해 생성하기.
List<E> immutableList = List.of(E.. elements);
Set<E> immutableSet = Set.of(E.. elements);
Map<K, V> immutableMap = Map.of( K k1, V v1, K k2, V v2, ...);

// 2. copyOf()를 통해 생성.
List<E> immutableList = List.copyOf(Collection<E> coll);
Set<E> immutableSet = Set.copyOf(Collection<E> coll);
Map<K, V> immutableMap = Map.copyOf(Map<K, V> map);

// 3. 배열을 통해 생성.
String[] arr  { "A", "B", "C" };
List<String> immutableList = Arrays.asList(arr);
```
