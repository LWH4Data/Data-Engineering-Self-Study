<h1>1. 라이브러리</h1>
<ul>
  <li>
    <strong>라이브러리(library)</strong>는 프로그램 개발 시 활용할 수 있는 <strong>클래스와 인터페이스들을 모아놓은 것</strong>을 의미한다.
  </li>
  <li>
    일반적으로 <strong>JAR(Java ARchive) 압축 파일(*.jar)</strong> 형태로 존재한다.
  </li>
    <ul>
      <li>
        Jar 파일에는 클래스와 인터페이스의 <strong>바이트코드 파일(*.class)</strong>들이 압축되어 있다.
      </li>
    </ul>
  <li>
    특정 클래스와 인터페이스가 여러 응용프로그램을 개발할 때 공통으로 자주 사용된다면 JAR 파일로 압축해서 라이브러리로 관리하는 것이 좋다.
  </li>
  <li>
    프로그램 개발 시 라이브러리를 이용하려면 라이브러리 JAR 파일을 <strong>ClassPath(class를 찾기위한 경로)에 추가</strong>해야 한다.
  </li>
    <ul>
      <li>
        <strong>콘솔(명령 프롬프트 or 터미널)</strong>에서 프로그램을 실행할 경우
      </li>
        <ul>
          <li>
            java 명령어를 실행할 때 -classpath로 제공
          </li>
          <li>
            CLASSPATH 환경 변수에 경로를 추가
          </li>
        </ul>
      <li>
        <strong>이클립스</strong> 프로젝트에서 실행할 경우
      </li>
        <ul>
          <li>
            프로젝트의 Build Path에 추가
          </li>
        </ul>
    </ul>
  <li>
    이후는 라이브러리 생성에 대한 내용인데 현제 VSCode를 사용하고 있기에 정리하지 않음. VSCode의 방식은 검색을 통해서 쉽게 찾을 수 있다.
  </li>
</ul>

<br><br>

<h1>2. 모듈</h1>
<ul>
  <li>
    모듈(module)은 Java 9부터 지원되었으며 <strong>패키지 관리 기능</strong>까지 포함된 라이브러리이다.
  </li>
    <ul>
      <li>
        일반 라이브러리는 내부에 포함된 모든 패키지에 외부 프로그램이 접근 가능하지만, 모듈은 일부 패키지들은 <strong>은닉</strong>하여 접근할 수 없게 할 수 있다.
      </li>
    </ul>
  <li>
    모듈은 자신이 실행할 때 필요로 하는 의존 모듈을 <strong>모듈 기술자(module-info.java)</strong>에 기술할 수 있어 <strong>모듈 간의 의존 관계</strong>를 파악하기 쉽다.
  </li>
  <li>
    모듈 또한 라이브러리이기에 <strong>JAR 파일 형태</strong>로 배포할 수 있다.
  </li>
  <li>
    대규모 응용프로그램은 기능별로 <strong>모듈화(modulization)</strong>해서 개발할 수 있다.
  </li>
    <ul>
      <li>
        모듈별로 조립하는 방식을 사용하면 <strong>재사용성</strong> 및 <strong>유지보수</strong>에 유리하다.
      </li>
    </ul>
</ul>

<br><br>

<h1>3. 응용프로그램 모듈화</h1>
<ul>
  <li>
    응용프로그램의 <strong>규모가 커질수록</strong> 협업과 유지보수 측면에서 <strong>서브 모듈로 쪼개어 개발</strong>하는 것이 유리하다.
  </li>
    <ul>
      <li>
        이렇게 개발된 모듈들은 다른 응용프로그램에서 <strong>재사용</strong>이 가능하다.
      </li>
    </ul>
</ul>

<h1>5. 패키지 은닉</h1>
<ul>
  <li>
    모듈은 <strong>모듈 기술자(module-info.java)</strong>에 exports 키워드로 내부 패키지 중 <strong>외부에서 사용할 패키지를 지정</strong>한다.
  </li>
    <ul>
      <li>
        exports 되지 않은 패키지들은 외부에서 사용할 수 없게 <strong>은닉</strong>된다.
      </li>
    </ul>
  <li>
    모듈 은익의 이유
  </li>
    <ul>
      <li>
        모듈 사용 방법 통일
      </li>
        <ul>
          <li>
            모듈 외부에서 은닉된 패키지를 사용하지 못하도록 막고, 공개된 패키지로 <strong>사용 방법을 통일</strong>한다.
          </li>
        </ul>
      <li>
        쉬운 수정
      </li>
        <ul>
          <li>
            모델 성능 향상을 위해 은닉 패키지를 수정하여도 <strong>공개된 패키지의 사용 방법은 동일</strong>하다.
          </li>
        </ul>
    </ul>
</ul>

<br><br>

<h1>6. 전이 의존</h1>
<ul>
  <li>
    전이 의존이란 현제 구조에서 <strong>'my_application_two → my_module_a → my_module_b'</strong> 처럼 <strong>참조의 참조</strong> 구조이다.
  </li>
    <ul>
      <li>
        중간 단계인 my_module_a가 my_module_b의 특정 멤버를 사용하지만, my_application_two이 사용하지 않는 경우 my_application_two는 <strong>사용하지도 않는 멤버로 인해 my_module_b에 의존</strong>된다.
      </li>
    </ul>
  <li>
    위와 같은 문제를 해결하기 위해 module-info.java를 작성할 때 <strong>transitive 키워드</strong>를 활용할 수 있다.
  </li>
    <ul>
      <li>
        transitive는 앞으로 나를 필요로하는 의존관계에서는 <strong>나의 의존을 그대로 사용할 수 있게 하라</strong>는 의미이다.
      </li>
    </ul>
</ul>

<br><br>

<h1>7. 집합 모듈</h1>
<ul>
  <li>
    집합 모듈은 여러 모듈을 <strong>모아놓은 모듈</strong>이다.
  </li>
    <ul>
      <li>
        매번 requires할 필요없이 집합 모듈만을 제공하면 된다.
      </li>
    </ul>
  <li>
    집합 모듈은 자체적인 패키지를 가지지 않고 모듈 기술자에 <strong>전이 의존 설정</strong>만 한다.
  </li>
</ul>

```java
// 모듈집합: 자체적인 패키지를 가지지 않고 전이만 하여 사용.
module my_module {
  requires transitive my_module_a;
  requires transitive my_module_b;
}
```

<h1>8. 리플렉션 허용</h1>
<ul>
  <li>
    <strong>리플렉션(reflection)</strong>이란 실행 도중 <strong>타입(클래스, 인터페이스 등)을 검사</strong>하고 <strong>구성 멤버를 조사</strong>하는 것을 의미한다.
  </li>
  <li>
    은닉된 패키지는 기본적으로 다른 모듈에 의해 리플렉션을 허용하지 않는다.
  </li>
  <li>
    필요에 따라 다음과 같이 리플레션을 허용할 수 있다. 
  </li>
  <li>
    opens 지정은 <strong>은닉된 패키지</strong>에만 해당한다. 
  </li>
    <ul>
      <li>
        exports된 패키지는 언제든지 리플렉션이 가능하다.
      </li>
    </ul>
</ul>

```java
// 1. 모듈 전체를 리플렉션 허용
/*
open module 모듈명 {
  ...
}
*/

// 2. 지정된 패키지에 대해 리플렉션 허용
/* 
module 모듈명 {
  ...
  opens 패키지1;
  opens 패키지2;
}
*/

// 3. 지정된 패지키에 대해 특정 외부 모듈에서만 리플렉션 허용.
/*
module 모듈명 {}
  ...
  opens 패키지1 to 외부모듈명, 외부모듈명, ...;
  opens 패키지2 to 외부모듈명;
*/
```

<h1>9. 자바 표준 모듈</h1>
<ul>
  <li>
    자바 프로그램이라면 반드시 <strong>JDK가 제공하는 표준 라이브러리</strong>를 활용해야 한다.
  </li>
    <ul>
      <li>
        Java 9부터 모듈화가 되어 Java 21에 표준 모듈이 완성되었다.
      </li>
    </ul>
  <li>
    java.base 모듈은 최하단 모듈로 requires하지 않아도 사용이 가능하다.
  </li>
  <li>
    java.se는 JDK가 제공하는 모든 모듈의 집합이며 자바 표준 라이브러리를 제한 없이 사용하고자 하는 경우 java.se를 requires하면 된다.
  </li>
  <li>
    <strong>표준 라이브러리를 모듈화</strong>한 이유는 응용프로그램을 실행하는데 필요한 모듈로만 구성된 <strong>작은 사이즈의 자바 실행 환경(JRE)</strong>을 만들기 위해서이다.
  </li>
  <li>
    작은 사이즈의 자바 실행 환경(JRE)를 필요로하는 경우는 다음과 같다.
  </li>
    <ul>
      <li>
        독립 실행형(응용프로그램 + 표준 라이브러리)으로 배포할 경우 표준 라이브러리 크기가 작을수록 <strong>배포 사이즈</strong>가 줄어든다.
      </li>
      <li>
        <strong>제한된 자원</strong>만 가지고 있는 <strong>소형(임베디드) 기기</strong>에는 사이즈가 작은 자바 실행 환경이 필요하다.
      </li>
    </ul>
</ul>
