<h1>1. 인터페이스의 역할</h1>
<ul>
  <li>
    <strong>인터페이스(interface)</strong>는 사전적 의미로 <strong>두 장치를 연결하는 접속기</strong>를 의미한다.
  </li>
  <li>
    두 장치를 개체로 본다면 인터페이스는 <strong>두 객체를 연결</strong>하는 역할을 한다.
  </li>
  <li>
    객체가 서로를 호출할 때 호출되는 객체가 변할 수 있으며 이런 경우의 <strong>다형성을 관리</strong>한다.
  </li>
    <ul>
      <li>
        상속으로 다형성을 관리하는 경우도 있지만 실제로는 인터페이스를 더 많이 사용한다.
      </li>
    </ul>
</ul>

<br><br>

<h1>2. 인터페이스와 구현 클래스 선언</h1>
<ul>
  <li>
    인터페이스는 <strong>'*.java'</strong> 형태의 <strong>소스 파일</strong>로 작성되고, <strong>'*.class'</strong> 형태로 <strong>컴파일</strong> 되기에 물리적 형태는 클래스와 동일하다.
  </li>
  <li>
    단, 소스를 작성할 때 선언하는 방법과 구성 멤버가 클래스와 다르다.
  </li>
</ul>

<br>

<h2>2-1. 인터페이스 선언</h2>
<ul>
  <li>
    인터페이스 선언은 <strong>interface</strong> 키워드를 사용한다.
  </li>
  <li>
    접근 제한자는 class와 동일
  </li>
    <ul>
      <li>
        <strong>default</strong>: 같은 패키지 내에서만 사용 가능.
      </li>
      <li>
        <strong>public</strong>: 패키지와 상관없이 사용 가능.
      </li>
    </ul>
</ul>

```java
// 1. interface의 접근 제한자.
//   - default
interface /** 인터페이스명 { ... } **/
//   - public
public interface /** 인터페이스명 {...} **/
```

```java
// 2. 인터페이스가 갖는 멤버들에는 다음과 같은 종류가 있다.
public interface /** 인터페이스명 **/ {
    // public 상수 필드
    // public 추상 메소드
    // public 디폴트 메소드
    // public 정적 메소드
    // private 메소드
    // private 정적 메소드
}
```

<br>

<h2>2-2. 구현 클래스 선언</h2>
<ul>
  <li>
    <strong>객체 A</strong>가 인터페이스의 추상 메소드를 호출 → <strong>인터페이스</strong>는 객체 B의 메소드를 실행으로 연결된다.
  </li>
    <ul>
      <li>
        이때 객체 B는 인터페이스에 선언된 추상 메소드와 <strong>동일한 선언부</strong>를 가진 메소드를 갖고 있어야 한다.
      </li>
      <li>
        이때 객체 B를 <strong>인터페이스를 구현한(implement) 객체</strong>라 한다.
      </li>
    </ul>
  <li>
    <strong>implements 키워드</strong>는 해당 클래스가 <strong>인터페이스를 통해 사용할 수 있음</strong>과 <strong>추상 메소드를 재정의한 메소드</strong>가 있다는 의미이다.
  </li>
</ul>

```java
// 1. implements를 통해 인터페이스를 구현하고 있음을 선언부에 명시한다.
public class B implements /** 인터페이스명 **/ { ... }
```

<br>

<h2>2-3. 변수 선언과 구현 객체 대입</h2>
<ul>
  <li>
    인터페이스도 하나의 <strong>타입</strong>이다.
  </li>
    <ul>
      <li>
        인터페이스는 <strong>참조 타입</strong>에 속하기에 객체를 참조하지 않고 있다는 의미로 <strong>null</strong>을 대입할 수 있다.
      </li>
    </ul>
  <li>
    인터페이스를 통해 구현 객체를 사용하려면, 인터페이스 변수에 <strong>구현 객체의 번지</strong>를 대입해야 한다.
  </li>
</ul>

```java
// 인터페이스는 참조 타입.
RemoteControl rc;
// 객체를 참조하지 않는다는 의미로 null 대입 가능.
RemoteControl rc = null;

// Television 객체를 생성하고 번지를 대입.
rc = new Televison();

// 대입된 구현 객체를 변수를 통해 인터페이스의 추상 메소드를 호출.
// 이후 메소드 호출 시 인터페이스의 추상 메소드를 호출할 수 있다.
rc.turnOn();

// 변수 선언과 동시에 객체 대입도 가능.
RemoteControl rc = new Television();
```

<br><br>

<h1>3. 상수 필드</h1>
<ul>
  <li>
    인터페이스는 <strong>public static final</strong> 특성을 갖는 <strong>불변의 상수 필드</strong>를 멤버로 가질 수 있다.
  </li>
  <li>
    인터페이스에 선언된 <strong>필드</strong>는 모두 <strong>public static final</strong> 특성을 갖기 때문에 public static final을 생략하여도 자동으로 <strong>컴파일 과정</strong>에서 붙는다.
  </li>
  <li>
    상수명은 <strong>대문자</strong>로 작성하되, 서로 다른 단어로 구성되어 있을 경우 <strong>언더바(_)</strong>로 연결하는 것이 관례이다.
  </li>
  <li>
    상수는 구현 객체와 관련 없는 <strong>인터페이스 소속 멤버</strong>이기에 <strong>인터페이스로 바로 접근</strong>해 상수값을 읽을 수 있다.
  </li>
</ul>

<br><br>

<h1>4. 추상 메소드</h1>
<ul>
  <li>
    인터페이스는 구현 클래스가 재정의해야하는 <strong>public 추상 메소드(abstract method)</strong>를 멤버로 가질 수 있다.
  </li>
    <ul>
      <li>
        추상 메소드란 <strong>리턴타입, 메소드명, 매개변수</strong>만 기술되고 중괄호({})</strong>를 붙이지 않는 메소드를 의미한다.
      </li>
    </ul>
  <li>
    추상 메소드는 객체 A가 인터페이스를 통해 어떻게 메소드를 호출할 수 있는지 <strong>호출 방법</strong>을 알려주는 역할을 한다.
  </li>
  <li>
    구현 클래스에서 <strong>추상 메소드를 재정의</strong>할 때에는 <strong>public보다 더 낮은 접근 제한</strong>으로 재정의할 수가 없다.
  </li>
    <ul>
      <li>
        재정의되는 모든 메소드는 public이 추가되어 있다.  
      </li>
    </ul>
  <li>
    인터페이스로 구현 객체를 사용할 때에는 <strong>인터페이스 변수</strong>를 선언하고, <strong>구현 객체</strong>를 대입해야 한다.
  </li>
    <ul>
      <li>
        인터페이스 변수는 <strong>참조 타입</strong>이기 때문에 구현 객체가 대입되면 <strong>구현 객체의 번지</strong>를 저장한다.
      </li>
    </ul>
  <li>
    구현 객체가 대입되면 인터페이스 변수로 추상 메소드를 호출할 때 <strong>어떤 구현 객체가 대입</strong>되었는가에 따라 실행 내용이 달라진다.
  </li>
</ul>

```java
// 추상 메소드 활용.
/** public abstract 리턴타입 메소드명(매개변수, ...); **/

// 인터페이스 변수 선언 및 구현 객체 대입
RemoteControl rc;
rc = new Television();
RemoteControl rc = new Audio();
```

<br><br>

<h1>5. 디폴트 메소드</h1>
<ul>
  <li>
    인터페이스에는 <strong>완전한 실행 코드를 갖는 디폴트 메소드</strong> 선언이 가능하다.
  </li>
    <ul>
      <li>
        추상 메소드에는 실행부({})가 없지만, 디폴트 메소드에는 <strong>실행부</strong>가 있다. 
      </li>
      <li>
        선언 방법은 클래스 메소드와 동일하지만, <strong>default 키워드</strong>가 리턴 타입 앞에 붙는다.
      </li>
    </ul>
  <li>
    디폴트 메소드의 실행부({})에는 상수 필드를 읽거나 추상 메소드를 호출하는 코드를 작성할 수 있다.
  </li>
  <li>
    디폴트 메소드는 <strong>구현 객체</strong>가 필요한 메소드이다.
  </li>
    <ul>
      <li>
        따라서 메소드를 호출하기 위해서는 구현 객체(Television)를 <strong>인터페이스 변수에 대입한 뒤</strong>에 호출해야 한다.
      </li>
    </ul>
  <li>
    구현 클래스는 <strong>디폴트 메소드</strong>를 재정의해서 자신에게 맞게 수정할 수도 있다.
  </li>
    <ul>
      <li>
        재정의를 할 때에는 <strong>public 접근 제한자</strong>를 반드시 붙이고, <strong>default 키워드를 생략</strong>해야 한다.
      </li>
    </ul>
</ul>

```java
// 디폴트 메소드 선언
/** [public] defualt 리턴타입 메소드명(매개변수, ...) { ... } **/
```

<br><br>

<h1>6. 정적 메소드</h1>
<ul>
  <li>
    인터페이스에는 <strong>정적 메소드</strong> 선언이 가능하며 <strong>구현 객체가 없어도</strong> 인터페이스만으로 호출이 가능하다.
  </li>
  <li>
    인터페이스에 선언된 정적 메소드는 구현 객체 없이 <strong>인터페이스명</strong>으로 접근해서 호출할 수 있다.
  </li>
    <ul>
      <li>
        템플릿: 인터페이스명.정적 메소드명()
      </li>
    </ul>
  <li>
    정적 메소드의 실행부({})를 작성할 때 주의할 점은 <strong>상수 필드</strong>를 제외한 추상 메소드, 다폴트 메소드, private 메소드 등을 호출할 수 없다.
  </li>
    <ul>
      <li>
        정적 메소드는 <strong>구현 객체</strong>가 필요한 <strong>인스턴스 메소드</strong>이기 때문이다.
      </li>
    </ul>
</ul>

```java
// 정적 메소드 선언 방법.
/** [public / private] static 리턴타입 메소드명(매개변수, ...) { ... } **/

```

<br><br>

<h1>7. private 메소드</h1>
<ul>
  <li>
    인터페이스의 상수 필드, 추상 메소드, 디폴트 메소드, 정적 메소드는 모두 public 접근 제한이 기본이지만, 인터페이스 외부에서 접근할 수 없는 <strong>private 메소드 선언</strong>도 가능하다.
  </li>
    <ul>
      <li>
        <strong>private 메소드</strong>는 <strong>디폴트 메소드</strong> 안에서만 호출이 가능하다.
      </li>
      <li>
        반면 <strong>private 정적 메소드</strong>는 <strong>디폴트 메소드</strong> 뿐만 아니라 <strong>정적 메소드</strong> 안에서도 호출이 가능하다.
      </li>
    </ul>
  <li>
    private 메소드의 용도는 디폴트 메소드와 정적 메소드들의 <strong>중복 코드</strong>를 줄이기 위해서이다.
  </li>
</ul>

<br><br>

<h1>8. 다중 인터페이스 구현</h1>
<ul>
  <li>
    구현 객체는 여러 개의 인터페이스를 implements할 수 있다. 
  </li>
  <li>
    다중 인터페이스를 구현할 때에는 implements뒤에 <strong>각 인터페이스를 나열</strong>한 뒤 모든 인터페이스가 갖는 <strong>추상 메소드를 재정의</strong>한다. 
  </li>
  <li>
    구현 객체가 <strong>어떤 인터페이스 변수</strong>에 대입되냐에 따라 변수를 통해 호출할 수 있는 추상 메소드가 결정된다.
  </li>
</ul>

```java
// 다중 인터페이스 형식
/** public class 구현클래스명 implements 인터페이스A, 인터페이스B {} **/

// 구현 객체는 각 인터페이스 타입의 변수에 각각 대입될 수 있다.
// 인터페이스A 변수 = new 구현클래스명();
// 인터페이스B 변수 = new 구현클래스명();
```

<br><br>

<h1>9. 인터페이스 상속</h1>
<ul>
  <li>
    인터페이스는 다른 인터페이스를 <strong>상속</strong>할 수 있으며 클래스와 달리 <strong>다중 상속</strong> 또한 지원한다.
  </li>
    <ul>
      <li>
        다중 상속을 할 때에는 <strong>extends</strong> 키워드 뒤에 상속할 인터페이스르 나열한다.
      </li>
    </ul>
  <li>
    자식 인터페이스의 구현 클래스는 <strong>자식과 부모 인터페이스의 모든 추상 메소드</strong>를 재정의해야 한다.
  </li>
  <li>
    구현 객체는 <strong>자식 및 부모 인터페이스 변수</strong>에 대입될 수 있다.
  </li>
  <li>
    자식 인터페이스 변수의 경우 <strong>'자식 + 부모'</strong> 추상 메소드 호출이 가능하나, 부모 인터페이스의 경우 <strong>'부모'</strong>의 추상 메소드만 호출이 가능하다.
  </li>
</ul>

```java
// 다중 상속 형태
/** public interface 자식인터페이스 extends 부모인터페이스1, 부모 인터페이스2, { ... } **/
```

<br><br>

<h1>10. 타입 변환</h1>
<ul>
  <li>
    인터페이스 변수에 구현 객체를 대입하면 구현 객체는 <strong>인터페이스 타입</strong>으로 <strong>자동 타입 변환</strong>된다.
  </li>
  <li>
    반대로 인터페이스 타입을 <strong>구현 클래스 타입</strong>으로 변환 시키기 위해서는 <strong>강제 타입 변환</strong>이 필요하다.
  </li>
</ul>

<br>

<h2>10-1. 자동 타입 변환</h2>
<ul>
  <li>
    자동 타입 변환은 <strong>'인터페이스 변수 = 구현 객체;'</strong>일 때 구현 객체를 인터페이스 객체로 변환한다.
  </li>
  <li>
    부모 클래스가 인터페이스를 구현하고 있다면 <strong>자식 클래스</strong>도 <strong>인터페이스 타입</strong>으로 자동 타입 변환될 수 있다.
  </li>
</ul>

<br>

<h2>10-2. 강제 타입 변환</h2>
<ul>
  <li>
    강제 타입 변환은 <strong>캐스팅(casting) 기호</strong>를 사용하여 인터페이스 타입을 <strong>구현 클래스 타입</strong>으로 변환하는 것을 의미한다.
  </li>
  <li>
    구현 객체가 인터페이스 타입으로 자동 변환되면 <strong>인터페이스에 선언된 메소드</strong>만 사용 가능하다.
  </li>
    <ul>
      <li>
        따라서 자동 타입 변환 후 <strong>구현클래스의 메소드</strong>를 호출하기 위해서는 <strong>강제 타입 변환</strong>이 필요하다.
      </li>
    </ul>
</ul>

<br><br>

<h1>11. 다형성</h1>
<ul>
  <li>
    인터페이스 또한 <strong>다형성</strong>을 구현하는 주된 기술로 사용되며 현업에서는 상속보다 <strong>인터페이스</strong>릃 활용하는 경우가 더 많다.
  </li>
  <li>
    다형성이란 <strong>사용 방법은 동일</strong>하지만 <strong>다양한 결과</strong>가 나오는 성질을 의미한다.
  </li>
  <li>
    인터페이스는 <strong>메소드 재정의</strong>와 <strong>자동 타입 변환</strong>을 통해 다형성을 구현한다.
  </li>
</ul>

<br>

<h2>11-1. 필드의 다형성</h2>
<ul>
  <li>
    필드 타입으로 타이어 인터페이스를 선언하면 자동 타입 변환으로 인해 필드값으로 객체를 대입할 수 있다. 
  </li>
</ul>

<br>

<h2>11-2. 매개변수의 다형성</h2>
<ul>
  <li>
    매개변수 타입을 <strong>인터페이스</strong>로 선언하면 메소드 <strong>호출 시 다양한 구현 객체</strong>를 대입할 수 있다.
  </li>
  <li>
    메소드를 호출할 때 인터페이스를 구현하는 어떠한 객체라도 <strong>매개값</strong>으로 줄 수 있다.
  </li>
    <ul>
      <li>
        어떤 객체를 매개값으로 주는가에 따라 메소드의 실행 결과가 다르게 나온다. (구현 객체에서 재정의된 메소드가 다르기 때문).
      </li>
    </ul>
</ul>

<br>

<h2>11-3. 요약</h2>
<ul>
  <li>
    <strong>필드</strong>만을 활용한 다형성의 경우 객체를 계속 바꾸어주어야하기 때문에 <strong>즉시 변경하여 반영이 어려움</strong>.
  </li>
  <li>
    <strong>매개변수</strong>만을 활용한 다형성의 경우 객체라면 단순히 반복하면 되는데 <strong>매번 매개변수로 전달</strong>해야한다는 문제가 있음.
  </li>
</ul>

<h1>12. 객체 타입 확인</h1>
<ul>
  <li>
    상속에서 객체 타입을 확인하기 위해 <strong>instanceof</strong> 연산자를 사용하는 것과 동일하게 인터페이스도 <strong>instanceof</strong>로 객체 확인이 가능하다.
  </li>
  <li>
    메소드의 매개변수가 인터페이스 타입인 경우, 메소드 호출 시 <strong>매개값</strong>은 해당 <strong>인터페이스를 구현하는 모든 객체</strong>가 될 수 있다.
  </li>
    <ul>
      <li>
        매개값이 <strong>특정 구현 객체</strong>일 경우에만 <strong>강제 타입 변환</strong>을 하고 싶다면 <strong>instanceof 연산자</strong>를 사용해서 <strong>매개값 타입 검사</strong>를 해야한다.
      </li>
      <li>
        단, Java 12부터는 instanceof 연산의 결과가 <strong>true</strong>인 경우, 우측 타입 변수를 사용할 수 있기 때문에 강제 타입 변환이 필요 없다. (실행문이 없어도 <strong>조건문이 필요</strong>하다).
      </li>
    </ul>
</ul>

```java
// 1. instanceof 연산자를 통해 Bus 객체인지 확인.
if( vehicle instanceof Bus ) {
  // vehicle에 대입된 객체가 Bus인 경우 실행.
}

// 2. vehicle에 대입된 객체가 Bus일 경우에만 Bus로 강제 타입 변환.
public void method( Vehicle vehicle ) {
  if(vehicle instanceof Bus) {
    Bus bus = (Bus) vehicle;
    // bus 변수 사용
  }
}

// 3. Java 12 이후부터는 조건문이 true이면 우측 타입 변수 사용 가능.
//   - 실행문이 없어도 되지만 실행문은 있어야 함.
if(vehicle instanceof Bus bus) {
  // bus 변수 사용
}
```

<h1>13. 봉인된 인터페이스</h1>
<ul>
  <li>
    Java 15부터는 무분별한 인터페이스 생성을 방지하기 위해 <strong>봉인된(sealed) 인터페이스</strong>를 사용할 수 있다.
  </li>
    <ul>
      <li>
        봉인된 인터페이스로 선언하여 지정한 자식 인터페이스만 허용한다.
      </li>
    </ul>
  <li>
    <strong>sealed 키워드</strong>를 사용하면 <strong>permits 키워드 뒤</strong>에는 상속 가능한 <strong>자식 인터페이스</strong>를 지정해야 한다.
  </li>
    <ul>
      <li>
        봉인된 인터페이스를 상속하는 인터페이스는 <strong>non-sealed 키워드</strong>로 선언하거나 sealed 키워드를 통해 <strong>또 다른 봉인 인터페이스로 선언</strong>해야 한다.
      </li>
        <ul>
          <li>
            non-sealed는 <strong>봉인 해제</strong>의 의미로 상속되는 인터페이스는 <strong>다른 자식 인터페이스</strong>를 만들 수 있다.
          </li>
        </ul>
    </ul>
</ul>

```java
// 1. 봉인된 인터페이스 형식
public sealed interface InterfaceA permits InterfaceB { ... }

// 2. sealed InterfaceA를 상속받는 InterfaceB
public non-sealed interface InterfaceB extends InterfaceA { ... }

// 3. non-sealed로 선언된 InterfaceB는 다른 자식 인터페이스 생성 가능.
public interface InterfaceC extends InterfaceB { ... }
```
