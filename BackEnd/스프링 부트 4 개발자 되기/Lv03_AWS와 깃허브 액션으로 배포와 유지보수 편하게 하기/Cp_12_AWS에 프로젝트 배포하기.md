<h1>1. 사전 지식 : AWS</h1>
<h2>1-1. AWS란?</h2>
<ul>
  <li>
    실제 서버를 배포하는 것에는 서버용 PC를 구매하는 것과 AWS와 같은 클라우드 컴퓨팅 서비스를 이용하는 방법이 있다.
  </li>
</ul>

<br>

<h2>1-2. AWS 서비스로 우리의 서비스 배포하기</h2>
<ul>
  <li>
    EC2, 오토 스케일링 그룹, RDS를 사용한다.
  </li>
  <li>
    AWS에서는 정말 많은 서비스를 제공하기에 모든 사용 방법을 파악하기 어렵다. 이를 위해 <strong>일래스틱 빈스토크(elatsic beanstalk)<strong>라는 서비스를 제공한다.
  </li>
</ul>

<h3>1-2-1. 가상의 PC 또는 EC2</h3>
<ul>
  <li>
    EC2는 AWS에서 제공하는 클라우드 컴퓨팅 서비스로 <strong>가상의 PC(서버 한 대)를 임대</strong>하는 서비스이다.
  </li>
</ul>

<h3>1-2-2. 유동적으로 EC2를 관리해주는 오토 스케일링 그룹</h3>
<ul>
  <li>
    사용자의 요청이 많아지면 이를 커버하기 위해 <strong>스케일 아웃</strong>을 하게된다.
  </li>
  <li>
    여러 서버가 대기만을 할 수는 없기 때문에 <strong>오토 스케일링 그룹(auto scailing group)</strong>을 활용한다.
  </li>
    <ul>
      <li>
        오토 스케일링 그룹은 서비스의 <strong>사용자의 요청 횟수</strong>에 따라 EC2를 늘이거나 줄인다.
      </li>
    </ul>
</ul>

<h3>1-2-3. 요청을 분산시켜주는 로드 밸런서</h3>
<ul>
  <li>
    요청이 하나의 경로(서버)로만 들어오면 과부화하 올 수 있어 이를 적절히 분산해야한다.
  </li>
  <li>
    요청들을 분산시키는 역할을 <strong>로드 밸런서(load balancer)</strong>가 한다.
  </li>
    <ul>
      <li>
        요청을 어디로 분산시킬지 정한 그룹을 <strong>대상 그룹(Target Group)</strong>이라 한다.
      </li>
    </ul>
</ul>

<h3>1-2-4. 데이터 저장소 RDS</h3>
<ul>
  <li>
    데이터베이스도 클라우드에서 제공하는 서비스를 사용한다.
  </li>
    <ul>
      <li>
        AWS에는 RDS, Redshift, DocumentDB, ElastiCache 등이 있다.
      </li>
    </ul>
</ul>

<br>

<h2>1-3. 설정이 간편한 일래스틱 빈스토크</h2>
<ul>
  <li>
    일래스틱 빈스토크는 AWS의 서비스를 한 번에 설정하는 서비스이다.
  </li>
  <li>
    서비스 코드만 올리면 로드 밸런싱, 오토 스케일링, 모니터링, 배포 등의 기능을 일래스틱 빈스토크에 <strong>직접 구성하거나 설정 파일로 자동 처리</strong>할 수 있다.
  </li>
  <li>
    일래스틱 빈스토크 자체에는 비용이 없으며 사용하는 서비스에 대한 비용만 지불하면 된다.
  </li>
</ul>

<h3>1-3-1. 일래스틱 빈스토크 사용하는 과정</h3>
<ul>
  <li>
    애플리케이션을 생성하고 소스를 번들 형태로 애플리케이션 버전을 일래스틱 빈스토크에 업로드한다.
    <br>→ 일래스틱 빈스토크가 자동으로 환경을 실행하고 코드 실행에 필요한 AWS 리소스를 생성 및 구성한다.
    <br>→ 이후에는 환경을 직접 관리하거나 필요한 경우 버전 업데이트를 할 수 있다.
  </li>
</ul>

<br><br>

<h1>2. AWS 계정 생성하기</h1>
<ul>
  <li>
    이미 프리티어를 다 소진하였고 EC2와 RDS 등 기본적인 서비스를 통해 프로젝트를 진행한 적이 있어 skip.
  </li>
  <li>
    일래스틱 빈스토크가 편리하다 생각하지만 직접 금액을 통제하기 힘들다 생각되어 도서 내용을 학습만 한다.
  </li>
</ul>

<br><br>

<h1>3. 일래스틱 빈스토크로 서버 구축하기</h1>
<h2>3-1. 일래스틱 빈스토크 서비스 생성하기</h2>
<ul>
  <li>
    서버의 위치를 빠른 응답 속도를 위해 서울로 설정한다.
  </li>
</ul>

<br>

<h2>3-2. 일래스틱 빈스토크에서 RDS 생성하기</h2>
<ul>
  <li>
    RDS 설정을 일래스틱 빈스토크에서 하면 설정한 환경 속성 값이 properties.yml 파일의 설정값 역할을 하기에 <strong>기존 서버의 application.yml에서 DB 설정을 제거</strong>해야 한다.
  </li>
</ul>

<br>

<h2>3-3. 로컬에서 RDS 연결하기</h2>
<ul>
  <li>
    클라우드 DB인 RDS를 연결하기 위해 DB 식별자를 누르고 VPC 보안 그룹 설정의 설정을 완료한다.
  </li>
  <li>
    EC2 → RDS로의 인바운드 규칙을 추가하여 접근 가능하도록 한다.
  </li>
    <ul>
      <li>
        인바운드 규칙을 허가하면 해당 규칙에 대한 아웃바운드 규칙은 허용된다.
      </li>
    </ul>
  <li>
    참고로 인텔리제이에서 DB를 연결하는 건 유료 기능이기에 무료로 사용하려면 플러그인을 설치한다.
  </li>
    <ul>
      <li>
        추가한 플러그인에서 SQL문으로 DB 테이블을 생성하고 조회할 수 있다.
      </li>
    </ul>
  <li>
    AWS RDS에서 설정한 DB에 맞게 서버에 의존성을 추가한다.
  </li>
</ul>

<br><br>

<h1>4. 일래스틱 빈스토크에 우리의 서비스 배포하기</h1>
<ul>
  <li>
    인텔리제이를 실행하고 [Gradle] → [Tasks → build → build]로 빌드를 진행한다.
    <br>→ "build/libs"에 결과로 나온 파일 중 "-버전"으로 끝나는 파일을 기억하기 쉬운 위치에 복사한다.
    <br>→ 일래스틱 빈스토크로 돌아가 생성된 환경의 이름을 선택하고 [업로드 및 배포]를 누른다.
    <br>→ jar 파일을 선택한다.
    <br>→ 일래스틱 빈스토크의 [구성] → [업데이트, 모니털이 로깅]의 편집을 선택하고 <strong>포트, .env에 들어갈만한 민감 정보 등</strong>을 설정한다.
    <br>→ 일래스틱 빈스토크에서 제공하는 URL로 들어가 로그인을 진행해 본다.
    <br>→ 이때 OAuth 서비스에서 승인된 URL에 일래스틱 빈스토크 서버 URL을 추가한다.
  </li>
</ul>

<br>

<h2>4-1. OAuth 서비스에 승인된 URI 추가하기</h2>
<ul>
  <li>
    구글 클라우드 콘솔(console.cloud.google.com) 콘솔에 접속하여 [API 및 서비스 → 사용자 인증 정보 → OAuth 2.0 클라이언트 ID → 추가되어 있는 클라이언트 ID → OAuth 클라이언트 수정 → 승인된 리디렉션 URI]에 일래스틱 빈스토크 서버의 URL과 "login/oauth2/code/google"을 붙여 추가한다.
  </li>
  <li>
    이전에는 서버가 로컬에서 떠서 로컬로 연결하지만 현재는 일래스틱 빈스토크에서 뜨기 때문에 경로를 수정해야 하는 것이다.
  </li>
</ul>

<br>

<h2>4-2. 파일 업로드 서비스를 S3로 변경하기</h2>
<ul>
  <li>
    파일을 서버의 저장소에 저장하면 서버를 재시작하며 유실될 가능성이 있기 때문에 S3(Simple Storage Service)와 같은 클라우드 저장소를 활용한다.
  </li>
  <li>
    AWS에 S3를 검색하고 [버킷 만들기]를 눌러 새로운 버킷을 생성한다.
  </li>
    <ul>
      <li>
        버킷은 S3의 저장소의 단위이다.
      </li>
    </ul>
  <li>
    S3 설정을 위해 "build.gradle", "application.yml", "config/S3Config.java", "service/AwsS3Service.java"
  </li>
</ul>

<h3>4-2-1. store() 메서드</h3>
<ul>
  <li>
    컨트롤러로부터 전달받은 파일과 원본 파일명 UUID를 이용해 고유한 파일명을 생성하고 파일 스트림을 S3에 저장한다.
  </li>
  <li>
    "service/LocalFileStorageService.java" 코드를 수정한다.
    <br>→ SpringBootDeveloperApplication을 클릭하고 [구성 편집]을 선택한다.
    <br>→ 활서화된 프로파일을 찾아 "local"로 변경한다.
    <br>→ 로컬에서 실행하면 로컬 PC로, 배포 환경에서는 S3에 저장되도록 설정한 것.
  </li>
  <li>
    [build → build]를 선택
    <br>→ jar 파일 생성.
    <br>→ AWS에서 [업로드 및 배포]를 선택.
    <br>→ 새로운 jar 파일을 업로드
    <br>→ [배포] 클릭.
    <br>→ 에러 발생
    <br>→ 엘라스틱 빈스토크의 [로그 → 전체]를선택하고 로그 확인. (후에는 ELK Stack, AWS CloudWatch 등으로 로그를 쉽게 볼 수 있음).
    <br>→ IAM에 S3 버킷 권한이 없는 문제 해결.
  </li>
</ul>
