<h1>1. 서버와 클라이언트</h1>
<h2>1-2. 클라이언트란?</h2>
<ul>
  <li>
    클라이언트(client)는 <strong>서버로 요청</strong> 하는 모든 프로그램을 의미한다.
  </li>
</ul>

<br>

<h2>1-2. 서버란?</h2>
<ul>
  <li>
    서버는(server)는 클라이언트의 <strong>요청을 받아 처리하는 주체</strong>를 의미한다.
  </li>
    <ul>
      <li>
        웹 브라우저에 주소를 입력하는 건 <strong>새로운 화면을 그리기 위한 데이터를 달라는 요청</strong>에 해당한다.
      </li>
    </ul>
</ul>

<br><br>

<h1>2. 데이터베이스</h1>
<ul>
  <li>
    데이터베이스(database)는 여러 사람이 <strong>데이터를 한 군데</strong>에 모아놓고 <strong>여러 사람이 사용할 목적</strong>으로 관리하는 데이터 저장소이다.
  </li>
  <li>
    MySQL, Oracle, PostgreSQL 등은 엄밀히 데이터베이스는 아니고 이를 관리하는 시스템이나 묶어서 통칭하는 경우가 많다.
  </li>
  <li>
    클라이언트가 SQL로 데이터베이스를 조작하기 위한 언어를 <strong>데이터베이스 관리 시스템에 요청</strong>하면 데이터베이스 관리 시스템은 데이터를 꺼내 응답한다.
  </li>
</ul>

<br>

<h2>2-1. RDB란?</h2>
<ul>
  <li>
    RDB는 Relational Database의 약자로 관계형 데이터베이스를 의미한다.
  </li>
    <ul>
      <li>
        관계형 데이터베이스가 아닌 데이터베이스를 NoSQL 또는 NewSQL로 구분한다.
      </li>
    </ul>
  <li>
    관계형 데이터베이스는 <strong>행(row)과 열(colume)로 이루어진 테이블</strong>로 관리하며 <strong>기본키(primary key)</strong>를 사용해 <strong>각 행을 식별</strong>한다.
  </li>
    <ul>
      <li>
        또한 각 테이블 간에 관계를 지을 수 있다.
      </li>
    </ul>
</ul>

<h2>2-2. SQL이란?</h2>
<ul>
  <li>
    SQL이란 Structured Query Language의 약자로 데이터 검색을 하는 언어이다.
  </li>
  <li>
    ANSI 표준 SQL이 있으나 데이터베이스 관리 시스템별로 조금씩 차이가 있다.
  </li>
  <li>
    NoSQL 또한 SQL 엔진을 도입하여 데이터를 좀 더 편하게 질의하도록 지원하는 경우가 많다.
  </li>
</ul>

<br>

<h2>2-3. NoSQL이란?</h2>
<ul>
  <li>
    NoSQL은 SQL을 안 쓴다는 의미도 있지만 보통 Not Only SQL 의미로 많이 사용된다. 즉, SQL만 사용하지는 않는다는 의미이다.
  </li>
  <li>
    RDB는 데이터 저장, 질의, 수정, 삭제는 용이하나 성능을 올리기 쉽지 않다.
  </li>
    <ul>
      <li>
        스케일 업은 장비를 업그레이드 하거나, 스케일 아웃을 통해 여러 대의 장비로 분리해야 한다.
      </li>
      <li>
        트랜잭션을 사용하면 성능이 떨어진다.
      </li>
    </ul>
  <li>
    NoSQL은 RDB의 성능 저하 문제를 완화할 수 있으며 데이터 모델링에 따라 다이나모디비, 카우치베이스, 몽고디비 등 다양한 NoSQL이 있다.
  </li>
</ul>

<br><br>

<h1>3. 아이피와 포트</h1>
<ul>
  <li>
    아이피(IP)는 <strong>인터넷에서 컴퓨터 또는 기기들이 서로 식별하고 통신하기 위한 주소</strong>로 이를 통해 서버를 찾을 수 있다.
  </li>
  <li>
    IP가 서버를 찾기 위한 번호라면 <strong>Port</strong>를 통해 서버에서 운용되는 <strong>서비스</strong>를 찾아야한다.
  </li>
  <li>
    웹 브라우저 또한 도메인 네임에 가려져 있지만 기본적으로 IP와 Port 정보이다.
  </li>
    <ul>
      <li>
        <strong>"https://"</strong>는 <strong>443번 포트</strong>를 이용하기 위한 입력이다.
      </li>
    </ul>
  <li>
    각 애플리케이션의 포트 넘버는 수정이 가능하다.
  </li>
</ul>

<br><br>

<h1>4.라이브러리와 프레임워크</h1>
<h2>4-1. 라이브러리란?</h2>
<ul>
  <li>
    라이브러리(library)는 애플리케이션 개발에 필요한 기능인 <strong>클래스, 함수</strong> 등을 모아놓은 <strong>코드의 모음</strong>을 말한다.
  </li>
    <ul>
      <li>
        다른 사람이 미리 개발해 둔 것들을 가져다 사용하여 개발 속도를 향상 시킬 수 있다.
      </li>
      <li>
        각 라이브러리들은 독립적이기에 서로 영향을 크게 주지 않는다.
      </li>
    </ul>
  <li>
    개발을 할 때에 <strong>필요한 기능을 구현</strong>하기 위해 사용한다.
  </li>
</ul>

<br>

<h2>4-2. 프레임워크란?</h2>
<ul>
  <li>
    프레임워크는 소프트웨어 개발을 수월하게 하기 위한 <strong>소프트웨어 개발 환경</strong>이다.
  </li>
    <ul>
      <li>
        틀(frame)과 일하다(work)의 합성어로 일하기 위한 틀을 제공한다.
      </li>
    </ul>
  <li>
    애플리케이션 개발에 있어 <strong>전체저인 구조</strong>를 잡기 위해 사용한다.
  </li>
</ul>

<br><br>

<h1>5. 백엔드 개발자의 업무</h1>
<ul>
  <li>
    백엔드 개발자의 주된 업무는 서버 측 애플리케이션을 개발하는 것이다.
  </li>
    <ul>
      <li>
        서버에 대한 지식, 프로그래밍 지식, 배포 및 보안에 대한 지식을 활용한다.
      </li>
    </ul>
  <li>
    대게 백엔드 개발자의 업무 순서는 <strong>"과제 할당 → 과제 분석 → 개발 → 테스트(리뷰) → QA 및 버그 수정 → 배포 → 유지보수"</strong> 순으로 진행된다.
  </li>
    <ul>
      <li>
        <strong>과제 할당</strong>: 다양한 부서에서 개발 과제를 할당 받는다.
      </li>
      <li>
        <strong>과제 분석</strong>
      </li>
        <ul>
          <li>
            주어진 과제를 코드로 옮기는 과정을 설계한다 (상태, 알고리즘, 등).
          </li>
          <li>
            제공할 성능과 기술을 검토하고 문제가 있다면 해결 방법 또한 분석한다.
          </li>
        </ul>
      <li>
        <strong>개발</strong>
      </li>
        <ul>
          <li>
            개발을 진행하며 대부분 버그를 수정하는 일을 하게 된다.
          </li>
          <li>
            다른 개발자들과의 리뷰를 통해 완성도를 높여간다.
          </li>
        </ul>
      <li>
        <strong>테스트</strong>
      </li>
        <ul>
          <li>
            실행 여부, 로직, 성능 한계, 코드의 가독성 등을 검토한다.
          </li>
          <li>
            조직이나 개발자에 따라 단위 테스트를 작성하는 경우도 있다.
          </li>
          <li>
            개발용 코드보다 테스트용 코드를 먼저 작성하는 테스트 주도 개발 기법을 사용하기도 한다.
          </li>
        </ul>
      <li>
        <strong>QA</strong>: 개발과 테스트가 어느정도 완료 되었다면 품질을 위해 QA(Quality Assurance)를 신청한다.
      </li>
      <li>
        <strong>배포와 유지보수</strong>
      </li>
        <ul>
          <li>
            배포 방법에는 롤링 배포, 블루 그린 배포, 카나리 배포 등 여러 가지가 존재한다.
          </li>
          <li>
            개발자는 테스트, 배포, 모니터링을 반복한다.
          </li>
          <li>
            빌드부터 배포까지의 과정을 자동화 하는 것을 <strong>CI(Continuout Integration)/CD(Continuous Delivery & Continuous Deployment)</strong>라 한다.
          </li>
            <ul>
              <li>
                CI는 지속적인 통합으로 새로운 코드 변경 사항이 <strong>정기적으로 빌드 및 테스트</strong>되어 <strong>저장소에 통합</strong>되는 것을 의미한다.
              </li>
              <li>
                CD는 지속적인 배포를 의미한다. 즉, <strong>프로덕션 환경까지 변경된 코드가 반영</strong>된다.
              </li>
            </ul>
        </ul>
    </ul>
</ul>

<h1>7. 자바 애너테이션</h1>
<ul>
  <li>
    자바 <strong>애너테이션(annotation)</strong>은 작성한 코드에 추가하는 표식으로 <strong>@기호</strong>를 사용하며 보통 <strong>메타 데이터</strong>로 사용되는 경우가 많다.
  </li>
    <ul>
      <li>
        @Override: 선언된 메소드가 오버라이드 되어 있음.
      </li>
      <li>
        @Deprecated: 더 이상 사용되지 않음.
      </li>
      <li>
        @SuppressWarning: 컴파일 경고를 무시함.
      </li>
    </ul>
</ul>
