<h1>1. 스프링과 스프링 부트</h1>
<h2>1-1. 스프링의 등장</h2>
<ul>
  <li>
    <strong>엔터프라이즈 애플리케이션</strong>이란 <strong>대규모의 복잡한 데이터를 관리</strong>하는 애플리케이션을 의미한다.
  </li>
    <ul>
      <li>
        소프트웨어 분야가 발전하면서 엔터프라이즈 애플리케이션이 점점 복잡해지는 문제 발생.
      </li>
      <li>
        개발을 하며 서버 성능과 안정성, 보안 등을 고려하면서 개발하는 것이 힘들어짐.
      </li>
    </ul>
  <li>
    2003년 6월에 등장한 스프링 부트는 위의 문제점을 개선하고자 등장 하였으며 <strong>서버의 성능, 보안, 안정성</strong>을 높은 수준으로 제공한다.
  </li>
</ul>

<br>

<h2>1-2. 스프링을 더 쉽게 만들어주는 스프링 부트</h2>
<ul>
  <li>
    스프링 부트는 <strong>복잡한 스프링의 설정을 보완</strong>하고자 2013년 4월 처음 공개된 도구이다.
  </li>
    <ul>
      <li>
        톰캣, 제티, 언더토우 등과 같은 <strong>웹 애플리케이션 서버(web application server, WAS)를 내장</strong>하고 있다.
      </li>
      <li>
        <strong>빌드 구성을 단순화</strong>하는 <strong>스프링 부트 스타터</strong>를 제공한다.
      </li>
      <li>
        XML을 설정 하지 않고 <strong>자바 코드로 모두 작성</strong>할 수 있다.
      </li>
      <li>
        JAR를 이용해 <strong>자바 옵션만으로도 배포</strong>가 가능하다.
      </li>
      <li>
        애플리케이션의 모니터링 및 관리 도구인 <strong>스프링 액츄에이터(spring actuator)</strong>를 제공한다.
      </li>
    </ul>
  <li>
    스프링 부트는 스프링에 속한 도구이다.
  </li>
</ul>

<h3>1-2-1. 차이점1: 구성의 차이</h3>
<ul>
  <li>
    <strong>스프링</strong>은 애플리케이션 개발에 필요한 환경을 <strong>수동으로 구성</strong>해야 한다. 반면 <strong>스프링 부트</strong>는 스프링 코어와 스프링 MVC의 모든 기능을 <strong>자동으로 로드</strong>해준다.
  </li>
</ul>

<h3>1-2-2. 차이점2: 내장WAS의 유무</h3>
<ul>
  <li>
    <strong>스프링</strong>은 일반적으로 <strong>톰캣과 같은 WAS</strong>에 배포가 된다. 반면 <strong>스프링 부트</strong>는 <strong>내장된 WAS(톰캣, 제티, 언더토우 등)</strong>가 있어 이를 활용할 수 있다.
  </li>
</ul>

<h1>2. 스프링 콘셉트 공부하기</h1>
<h2>2-1. 제어의 역전과 의존성 주입</h2>
<h3>2-1-1. IoC란?</h3>
<ul>
  <li>
    <strong>IoC(Inversion of Control)</strong>이란 객체를 직접 생성하지 않고 <strong>외부에서 관리하는 객체</strong>를 가져와 사용하는 것을 의미한다.
  </li>
  <li>
    스프링은 <strong>스프링 컨테이너</strong>가 객체를 관리, 제공하는 역할을 한다.
  </li>
  <li>
    기존에 코드가 직접 객체를 제어한 것에서 스프링이 객체를 제어하는 것으로 역전된 것이다.
  </li>
</ul>

```java
// 1. IoC 사례.
// 1-1. 기존의 방식(객체를 생성).
public class A {
  // 클래스 A에 new 키워드로 클래스 B의 객체 생성.
  b = new B();
}
// 1-2. 스프링 컨테이너를 활용한 IoC 활용.
public class A {
  // 코드에서 객체를 생성하지 않고 외부에서 받아온 객체를 b에 할당.
  private B b;
}
```

<h3>2-1-2. DI란?</h3>
<ul>
  <li>
    <strong>DI(Dependency Injection)</strong>란 의존성 주입으로 <strong>IoC를 구현</strong>하기 위해 사용하는 방법이다.
  </li>
  <li>
    의존성 주입이란 객체가 직접 필요한 객체를 만들지 않고, <strong>외부(스프링)가 대신 넣어주는 것</strong>을 의미한다.
  </li>
</ul>

```java
// 1. IoC/DI 사례.
public class A {
  // A클래스는 B를 주입 받음.
  //   - Autowired는 스프링 컨테이너에 있는 빈을 주입한다.
  //   - 빈은 스프링 컨테이너에서 관리하는 객체이다.
  //   - 즉, 스프링 컨테이너가 B 객체를 생성하여 A에 주입한다.
  @Autowired
  B b;
}
```

<h2>2-2. 빈과 스프링 컨테이너</h2>
<h3>2-2-1. 스프링 컨테이너란?</h3>
<ul>
  <li>
    스프링 컨테이너는 <strong>빈이 생성되고 소멸되기까지의 생명주기</strong>를 관리한다.
  </li>
  <li>
    개발자가 @Autowired 같은 애너테이션을 사용해 빈을 주입 받을 수 있게 <strong>DI를 지원</strong>한다.
  </li>
</ul>

<h3>2-2-2. 빈이란?</h3>
<ul>
  <li>
    커피콩(bean)처럼 작고 재사용 가능한 단위로 빈이라는 이름이 붙었다.
  </li>
  <li>
    빈은 IoC를 구현할 때 <strong>스프링 컨테이너가 제공하는 객체</strong>를 의미한다.
  </li>
  <li>
    빈을 등록하는 방법은 XML 파일 설정, 애너테이션 추가 등으로 다양하다.
  </li>
</ul>

```java
// 1. @Component 애너테이션을 통해 MyBean 클래스를 빈으로 등록.
@Component
public class MyBean {
}
```

<h2>2-3. 관점 지향 프로그래밍</h2>
<ul>
  <li>
    관점 지향 프로그래밍(AOP, Aspect Oriented Programming)은 프로그래밍에 대한 관심을 <strong>핵심 관점, 부가 관점</strong>으로 나누어 <strong>관심 기준으로 모듈화</strong>하는 것을 의미한다.
  </li>
    <ul>
      <li>
        <strong>부가 관점의 로직을 모듈화</strong>하여 프로그래머는 <strong>핵심 관점 코드에만 집중</strong>하도록 한다.
      </li>
      <li>
        프로그램의 변경과 확장에도 용이하다.
      </li>
    </ul>
</ul>

<br>

<h2>2-4. 이식 가능한 서비스 추상화</h2>
<ul>
  <li>
    이삭 가능한 서비스 추상화(PSA, Protable Service Abstraction)은 스프링에서 제공하는 다양한 기술들을 <strong>추상화</strong>해 <strong>개발자가 쉽게 사용하도록 하는 인터페이스</strong>를 의미한다.
  </li>
  <li>
    <strong>특정 기술에 종속되지 않고, 동일한 방식으로 다양한 기술을 사용</strong>할 수 있게 해주는 것을 의미한다.
  </li>
</ul>

<h2>3-2. 자동 구성</h2>
<ul>
  <li>

  <li>
    이를 통해 개발자가 특정 기술을 실행할 때마다 설정할 필요없이 간단하게 실행이 가능하다.

<h3>3-3-3. 레코드</h3>
<ul>
  <li>
    레코드는 <strong>데이터 전달을 목적으로 하는 객체</strong>를 더 빠르고 간편하게 만들기 위한 기능이다.
  </li>
  <li>
    레코드는 <strong>상속이 불가</strong>하고 파라미터에 정의된 필드는 <strong>private final</strong>로 정의된다.
  </li>
  <li>
    <strong>Getter를 자동으로 생성</strong>하기 때문에 애너테이션이나 메소드로 getter를 정의하지 않아도 된다.
  </li>
</ul>

```java
// record 선언
record Item(String name, int price) {
// 파라미터가 private final로 정의됨.
}

// 1. 17 버전.
Item juice = new Item("juice", 3000);
juice.price();  // 3000

// 2. 25 버전
if (obj instanceof Item(String name, int price)) {
  // name, price를 바로 사용 가능.
}
```

<h2>3-3-4. 패턴 매칭</h2>
<ul>
  <li>
    패턴 매칭은 타입 확인을 위해 사용하던 instanceof 키워드를 좀 더 쉽게 사용할 수 있게 해준다.
  </li>
    <ul>
      <li>
        instanceof 키워드와 형변환 코드를 조합할 필요 없이 <strong>바로 형변환</strong>을 할 수 있다.
      </li>
    </ul>
  <li>
    25버전부터는 <strong>하나의 switch 문</strong>으로 <strong>여러 타입에 대한 분기 처리</strong>를 표현할 수 있다.
  </li>
</ul>

```java
// 11 버전.
if (o instanceof Integer) {
  Integer i = (Integer) o;
  ''' 생략 '''
}

// 17 버전.
if (o instanceof Integer i) {
  ''' 생략 '''
}

// 25 버전.
switch (o) {
  case Integer i -> {
    // i 사용.
  }
  case String s -> {
    // s 사용.
  }
  default -> {
    // 기차 타입 처리
  }
}
```

<h3>3-3-5. 가상 스레드</h3>
<ul>
  <li>
    기존 플랫폼 스레드는 CPU 코어와 1:1 매핑을 하였지만, 자바 25버전의 <strong>가상 스레드(Virtual Threads)</strong>는 <strong>N:1 매핑</strong>이 가능하다.
  </li>
  <li>
    가상 스레드는 소수의 OS 스레드 풀 위에서 <strong>동시성</strong> 제공하며 JVM 가상 스레드의 스케줄을 관리한다.
  </li>
    <ul>
      <li>
        플랫폼 스레드는 1:1 매핑을 사용하는 병렬성 처리이다.
      </li>
    </ul>
</ul>

```java
// 17 버전.
ExecutorService executor = Executors.newFixedTreadPool(10);
executor.submit(() -> {
  // 작업 처리.
});

// 25 버전.
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
  executor.submit(() -> {
    // 작업 처리.
  });
}
```

<h2>3-5. HTTP Interface 사용 가능.</h2>
<ul>
  <li>
    스프링 부트 2까지는 HTTP 클라이언트를 사용하기 위해 RestTemplate이나 WebClient와 같이 <strong>명령형 방식으로 HTTP 호출을 구현</strong>해야 했다.
  </li>
  <li>
    스프링 부트 3부터는 HTTP 인터페이스를 통한 <strong>선언형 HTTP 클라이언트</strong>가 기본 지원된다.
  </li>
</ul>

```java
// 스프링 부트 2
@Service
public class MilestoneService {
  private final RestTemplate restTemplate;

  public MilestoneService(RestTemplate restTemplate) {
    this.restTemplate = restTemplate;
  }

  public List<Milestone> getMilestones(String org, String repo) {
    return restTemplate.exchange(
      "/repos/{org}/{repo}/milestones",
      HttpMethod.GET,
      null,
      new ParameterizedTypeReference<List<Milestone>>() {},
      org,
      repo
    ).getBodt();
  }
}

// 스프링 부트 4
public interface MilestoneService {
  @GetExchange("/repos/{org}/{repo}/milestones")
  List<Milestone> getMilestones(@PathVariable String org, @PathVariable Stgring repo);
}
```

<h1>4. 스프링 부트 4 코드 이해하기</h1>
<ul>
  <li>
    실습 코드의 주석과 함께 진행.
  </li>

<br>

<h3>4-a. @SpringBootConfiguration</h3>
<ul>
  <li>
    <strong>스프링 부트 관련 설정</strong>을 나타내는 애너테이션이다.
  </li>
  <li>
    @Configuration을 상속하여 만든 애너테이션이다.
  </li>
  <li>
    사용자가 직접 사용하는 애너테이션은 아니다.
  </li>
</ul>

<h3>4-b. @ComponentScan</h3>
<ul>
  <li>
    <strong>사용자가 등록한 빈</strong>을 읽고 등록한다.
  </li>
    <ul>
      <li>
        @Component라는 애너테이션을 갖는 클래스를 찾아 빈으로 등록한다.
      </li>
    </ul>
  <li>
    모든 빈에 @Component만 사용하는 것은 아니다. @Component를 감싸는 애너테이션이 있으며 실제 개발에서는 <strong>용도에 따라 다른 애너테이션</strong>을 사용한다.
  </li>
    <ul>
      <li>
        @Configuration: 설정 파일 등록.
      </li>
      <li>
        @Repository: ORM 매핑.
      </li>
      <li>
        @Controller, @RestController: 라우터.
      </li>
      <li>
        @Service: 비즈니스 로직.
      </li>
    </ul>
</ul>

<h3>4-c. EnableAutoConfiguration</h3>
<ul>
  <li>
    스프링 부트에서 <strong>자동 구성을 활성화</strong>하는 애너테이션이다.
  </li>
    <ul>
      <li>
        스프링 부트 서버가 실행될 때 스프링 부트의 <strong>파일을 읽고</strong> 정의된 <strong>설정들을 자동으로 구성</strong>한다.
      </li>
      <li>
        자동 구성에서 본 spring.factories의 클래스들이 모두 @EnableAutoConfiguration을 사용할 때 자동 설정된다.
      </li>
    </ul>
</ul>

<h2>4-1. 테스트 컨트롤러 살펴보기</h2>
<ul>
  <li>
    @Controller와 @ResponseBody 애너테이션이 함께 있고, 이 둘이 합쳐진 결과물이 RestController이다.
  </li>
    <ul>
      <li>
        @RestController 애너테이션 구현 파일인 RestController.java에는 @Controller를 포함하고 있다.
      <li>
        @Controller 애너테이션 구현 파일인 Controller.java에 @Component가 포함되어 있다.
      </li>
      <li>
        따라서 둘 모두 @Component 계열로 Bean 등록된다.
      </li>
    </ul>
  <li>
    다른 몇 가지 애너테이션들도 @Component 애너테이션을 가지고 있으나 <strong>빈이 무슨 역할을 하는지 명확하게 구분</strong>하기 위해 다른 이름으로 덮어 둔 것이다.
  </li>
  <li>
    @Component 애너테이션이 있는 클래스는 빈으로 등록된다.
  </li>
    <ul>
      <li>
        @Controller, @RestController, @Configuration, @Repository, @Service 모두 @Component에 해당한다.
      </li>
    </ul>
</ul>
