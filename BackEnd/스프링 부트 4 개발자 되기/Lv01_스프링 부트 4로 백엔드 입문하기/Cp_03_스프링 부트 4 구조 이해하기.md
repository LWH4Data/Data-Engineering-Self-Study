<h1>0. 그림으로 이해하는 프로젝트</h1>
<ul>
  <li>
    <strong>TestController 클래스</strong>: 클라이언트의 <strong>요청</strong>을 받아 분기 처리한다. (어떤 요청인지 판단).
  </li>
  <li>
    <strong>TestService 클래스</strong>: TestController에게 전달받은 요청에 맞는 <strong>작업을 수행</strong>한다.
  </li>
  <li>
    <strong>Member 클래스, MemberRepository 클래스</strong>: <strong>데이터베이스</strong>를 위한 클래스이다.
  </li>
    <ul>
      <li>
        Member 클래스로 구현한 클래스를 MemberRepository 클래스가 실제 테이블과 연결한다.
      </li>
    </ul>
</ul>

<br><br>

<h1>1. 스프링 부트 4 구조 살펴보기</h1>
<ul>
  <li>
    스프링 부트는 각 계층이 <strong>인접한 계층과 통신</strong>하는 구조를 따른다.
  </li>
    <ul>
      <li>
        계층이란 각자의 <strong>역할과 책임</strong>이 있는 어떤 소프트웨어의 구성 요소를 의미한다.
      </li>
      <li>
        각 계층은 서로 <strong>소통</strong>할 수는 있지만 다른 계층에 <strong>직접 간섭하거나 영향을 미치지 않는다</strong>.
      </li>
    </ul>
</ul>

<br>

<h2>1-1. 카페와 빵집으로 계층 이해하기</h2>
<ul>
  <li>
    커피와 빵집은 독립적으로 운영된다. 커피를 사면 빵을 할인해 줄 수는 있지만(통신) 빵집에서 커피를 파는 행위(영항 및 간섭)는 불가하다.
  </li>
  <li>
    스프링 부트에는 프레젠테이션, 비즈니스, 퍼시스턴스 계층이 서로 통신하며 프로그램을 구성한다.
  </li>
  <li>
    <strong>계층</strong>은 <strong>개념</strong>의 영역이고, <strong>실제 구현</strong>을 위한 영역은 <strong>컨트롤러, 서비스, 리포지터리</strong>이다.
  </li>
</ul>

<h3>1-1-1. 프레젠테이션 계층</h3>
<ul>
  <li>
    HTTP 요청을 받아 비즈니스 계층으로 전송한다.
  </li>
  <li>
    컨트롤러가 프레젠테이션 계층 역할을 한다.
  </li>
    <ul>
      <li>
        앞의 TestController가 해당하며 이외에도 몇 가지가 더 존재한다.
      </li>
    </ul>
</ul>

<h3>1-1-2. 비즈니스 계층</h3>
<ul>
  <li>
    모든 비즈니스 로직을 처리한다.
  </li>
    <ul>
      <li>
        비즈니스 로직이란 서비스를 만들기 위한 로직을 의미한다.
      </li>
    </ul>
  <li>
    서비스가 비즈니스 계층의 역할을 한다.
  </li>
</ul>

<h3>1-1-3. 퍼시스턴스 계층</h3>
<ul>
  <li>
    모든 데이터베이스 관련 로직을 처리한다.
  </li>
  <li>
    이 과정에서 데이터베이스에 접근하는 DAO 객체를 사용할 수도 있다.
  </li>
    <ul>
      <li>
        DAO란 데이터베이스 계층과 상호작용하기 위한 객체라고 이해하면 된다.
      </li>
    </ul>
</ul>

<h2>1-2. 스프링 부트 프로젝트 디렉터리 구성하며 살펴보기</h2>

  <li>

<h2>1-3. main 디렉터리 구성하기</h2>
<ul>
  <li>
    HTML과 같은 뷰 관련 파일을 넣을 resource/templates 디렉터리를 생성한다.
  </li>
  <li>
    이전에 생성한 static 디렉터리에는 JS, CSS와 같은 이미지와 같은 정적 파일을 넣는 용도로 사용한다.
  </li>
  <li>
    application.properties파일을 삭제하고 application.yml 파일을 생성한다.
  </li>
    <ul>
      <li>
        최근에는 YAML 형식을 더 많이 사용한다. YAML 형식은 계층 구조를 들여쓰기로 표현할 수 있어 설정값이 많아질수록 <strong>가독성</strong>이 뛰어나고 중첩된 설정을 <strong>구조적으로 관리하기 용이</strong>하다.
      </li>
      <li>
        application.yml에는 <strong>데이터베이스의 설정 경로, 로깅 설정 정보 등이 들어갈 수 있고 직접 설정을 정의</strong>할 때에도 사용한다.
      </li>
    </ul>
</ul>

<h1>2. 스프링 부트 4 프로젝트 발전시키기</h1>
<ul>
  <li>
    의존성을 추가한 뒤 프레젠테이션 계층, 비즈니스 계층, 퍼시스턴스 계층 순으로 코드를 추가한다.
  </li>
</ul>

<br>

<h2>2-1. build.gradle에 의존성 추가하기</h2>
<ul>
  <li>
    추가되는 패키지들
  </li>
    <ul>
      <li>
        <strong>스프링 데이터 JPA</strong>: 스프링 부트용 JPA이다.
      </li>
      <li>
        <strong>인메모리 데이터베이스 H2</strong>: 로컬 환경과 테스트 환경에서 사용할 인메모리 데이터베이스이다.
      </li>
      <li>
        <strong>롬복</strong>: 반복 메소드 작성 작업을 줄여준다.
      </li>
    </ul>
  <li>
    추가되는 패키지들 앞에는 해당 패키지를 어떻게 관리할지를 지정하는 <strong>키워드</strong>가 붙는다.
  </li>
    <ul>
      <li>
        <strong>implementation</strong>: 프로젝트 코드가 <strong>컴파일</strong>과 <strong>런타임</strong>에 모두 해당 라이브러리를 필요로할 때 사용한다.
      </li>
      <li>
        <strong>testImplementation</strong>: 프로젝트의 <strong>테스트 코드를 컴파일하고 실행</strong>할 때만 필요한 의존성을 설정, 테스트 코드에서만 사용하고 메인 애플리케이션 코드에서는 사용하지 않는다.
      </li>
      <li>
        <strong>runtimeOnly</strong>: <strong>런타임</strong>에만 필요한 의존성을 지정한다. 즉, 컴파일 시에는 필요하지 않지만 애플리케이션을 실행할 때 필요한 라이브러리를 설정한다.
      </li>
      <li>
        <strong>compileOnly</strong>: <strong>컴파일</strong> 시에만 필요한 의존성을 지정한다.
      </li>
      <li>
        <strong>annotationProcessor</strong>: 컴파일 시에 <strong>애너테이션을 처리</strong>할 때 사용하는 도구의 의존성을 지정한다.
      </li>
    </ul>
  <li>
    모든 의존성 지정이 끝났따면 [Gradle] 탭에서 새로 고침을 눌러 추가한 의존성을 다운로드 한다.
  </li>
</ul>

      <li>

      </li>

</ul>

<h2>2-4. 작동 확인하기</h2>
<ul>
  <li>
    실행 테스트를 하기 위해서는 애플리케이션을 실행할 때마다 SQL 문을 실행해 필요한 데이터를 DB에 넣는 작업이 필요하다.ㄴ
  </li>
  <li>
    애플리케이션을 실행할 때 저장할 더미 데이터를 넣을 SQL 파일 생성.
  </li>
    <ul>
      <li>
        resources 디렉터리에 <strong>data.sql 파일</strong>을 생성한다.
      </li>
      <li>
        데이터를 주입하기 위해 <strong>application.yml 파일</strong>을 열어 코드를 변경한다.
      </li>
        <ul>
          <li>
            <strong>show-sql, format_sql</strong>: 애플리케이션이 실행 과정에 데이터베이스에 쿼리할 일이 있으면 <strong>실행 구문을 모두 보여준다</strong>.
          </li>
          <li>
            <strong>defer-datasource-initialization</strong>: 애플리케이션을 실행할 때 <strong>테이블을 생성</strong>하고 data.sql 파일에 있는 <strong>쿼리를 실행</strong>하도록 한다.
          </li>
        </ul>
    </ul>
</ul>

<h1>3. 스프링 부트 요청-응답 과정 한 방에 이해하기</h1>
<ul>
  <li>
    포스트맨에서 톰캣에 /test GET 요청을 한다.
  </li>
  <li>
    스프링 부트의 디스패처 서블릿은 URL을 분석하고, 요청을 처리할 수 있는 컨트롤러를 찾는다.
  </li>
  <li>
    요청은 요청을 처리할 수 있는 메소드와 매칭되고 처리한 데이터를 가져온다.
  </li>
  <li>
    뷰 리졸버는 템플릿 엔진을 사용해 HTML 문서를 만들거나 JSON, XML 등의 데이터를 생성한다.
  </li>
  <li>
    완성된 결과를 return하고 포스트맨에서 확인할 수 있다.
  </li>
</ul>
