<h1>1. 터치와 키 이벤트</h1>
<h2>1-1. 터치 이벤트</h2>
<ul>
  <li>
    앱의 화면에서 발생하는 사용장 이벤트는 <strong>터치(touch)</strong>이다. 화면을 손가락으로 눌렀는지, 떼었는지, 스와이프(swipe) 했는지에 따라 동작한다.
  </li>
  <li>
    앱의 화면에서 발생하는 사용자의 터치 이벤트를 처리하고 싶다면 액티비티 클래스에 터치 이벤트의 콜백 함수 <strong>onTouchEvent()</strong>를 선언한다.
  </li>
    <ul>
      <li>
        콜백 함수란 어떤 이벤트가 발생하거나 시점에 도달했을 때 <strong>시스템에서 자동으로 호출하는 함수</strong>를 의미한다.
      </li>
    </ul>
  <li>
    액티비티에 onTouchEvent() 함수를 <strong>재정의(override)</strong>해서 선언해 두면 사용자가 액티비티 화면을 터치하는 순간 onTouchEvent() 함수가 자동 호출된다.
  </li>
    <ul>
      <li>
        onTouchEvent() 함수의 매개변수는 <strong>MotionEvent 객체</strong>이며 <strong>터치의 종류</strong>와 <strong>발생 지점(좌푯값)</strong>이 담긴다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 터치 이베트 처리.
class MainActivity : AppCompatActivity() {
    (... 생략 ...)
    ovierride fun onTouchEvent(event: MotionEvent?): Boolean {
        return super.onTouchEvent(event)
    }
}
```

<h3>1-1-1. 터치 이벤트의 종류</h3>
<ul>
  <li>
    터치 이벤트 종류에는 다음 세 가지가 있다.
  </li>
    <ul>
      <li>
        <strong>ACTION_DOWN</strong>: 화면을 손가락으로 <strong>누른 순간</strong>의 이벤트
      </li>
      <li>
        <strong>ACTION_UP</strong>: 화면에서 손가락을 <strong>떼는 순간</strong>의 이벤트
      </li>
      <li>
        <strong>ACTION_MOVE</strong>: 화면을 손가락으로 <strong>누른 채 이동하는 순간</strong>의 이벤트
      </li>
    </ul>
  <li>
    단순 터치는 onTouchEvent()가 두번 호출된다(ACTION_DOWN → ACTION_UP). 손가락을 이동하는 경우에는 세 번 호출된다(ACTION_DOWN → ACTION_MOVE → ACTION_UP)
  </li>
  <li>
    <strong>x</strong>는 터치 이벤트가 발생한 <strong>뷰에서의 좌표</strong>를, <strong>rawX</strong>는 <strong>스크린(화면)에서의 좌표</strong>를 의미한다.
  </li>
    <ul>
      <li>
        <strong>액티비티</strong>는 전체 영역이기에 onTouchEvent()를 재정의하면 x와 rawX의 값의 차이가 없다.
      </li>
      <li>
        <strong>특정 뷰</strong>에서 onTouchEvent()를 재정의하면 뷰와 액티비티의 차이로 x와 rawX의 좌표차이가 발생할 수 있다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 터치 이벤트 처리.
override fun onTouchEvent(event: MotionEvent?): Bollean {
    when (event?.action) {
        MotionEvent.ACTION_DOWN -> {
            Log.d("kkang", "Touch down event x: ${event.x}, rawX: ${event.rawX}")
        }
    }
    return super.onTouchEvent(event)
}
```

<h2>1-2. 키 이벤트</h2>
<ul>
  <li>
    키 이벤트는 사용자가 폰의 <strong>키를 누르는 순간</strong> 발생한다.
  </li>
  <li>
    액티비티의 키 이벤트를 처리할 때에는 다음과 같은 콜백 함수를 재정의한다.
  </li>
    <ul>
      <li>
        <strong>onKeyDown</strong>: 키를 누른 순간의 이벤트
      </li>
      <li>
        <strong>onKeyUp</strong>: 키를 떼는 순간의 이벤트
      </li>
      <li>
        <strong>onKeyLongPress</strong>: 키를 오래 누르는 순간의 이벤트
      </li>
    </ul>
  <li>
    키 이벤트 함수의 첫 번째 매개변수는 <strong>키의 코드</strong>로 사용자가 <strong>어떤 키</strong>를 눌렀는지 식별할 수 있다.
  </li>
  <li>
    여기서 의미하는 키는 앱에서 글을 입력할 때 화면에 나타나는 <strong>소프트 키보드(soft keyboard)</strong>를 의미하지 않는다.
  </li>
  <li>
    키 이벤트는 안드로이드 기기에 사용하는 <strong>하드웨어 키보드</strong>나 안드로이드 <strong>시스템 버튼</strong>이 해당한다.
  </li>
  <li>
    키들에는 보통 네비게이션 바(뒤로 가기, 홈, 오버뷰)의 버튼들과 전원 버튼 그리고 볼륨 조절 버튼이 있다.
  </li>
    <ul>
      <li>
        이 중 <strong>뒤로 가기 버튼</strong>과 <strong>볼륨 조절 버튼</strong>은 '키'로 취급하여 키 이벤트로 처리할 수 있다. (제스처 네비게이션을 활용해도 동일).
      </li>
    </ul>
  <li>
    onKeyDown() 혹은 onKeyUp() 함수 대신 onBackPressed() 함수를 활용할 수도 있는데 API Level 33 이후로는 deprecated 되었다.
  </li>
  <li>
    뒤로 가기 버튼을 이벤트에서 별도 처리하지 않으면 기본적으로 <strong>이전 액티비티</strong>로 이동한다.
  </li>
    <ul>
      <li>
        뒤로 가기 버튼의 이벤트를 별도 지정하는 경우는 주로 앱이 종료되지 않고 한번 더 확인하거나 메시지를 출력할 때에 사용한다.
      </li>
    </ul>
</ul>

<h1>2. 뷰 이벤트</h1>
<ul>
  <li>
    뷰의 이벤트도 터치 이벤트로 처리할 수 있지만 이 경우 프로그래밍이 복잡해 진다. 따라서 보통 뷰 이벤트를 별도로 사용한다.
  </li>
</ul>

<br>

<h2>2-1. 뷰 이벤트의 처리 구조</h2>
<ul>
  <li>
    뷰 이벤트는 <strong>일정한 구조</strong>에 따라 처리되며 콜백 함수만 선언해서는 처리할 수 없다.
  </li>
  <li>
    뷰 이벤트 처리는 <strong>이벤트 소스(event source)</strong>와 <strong>이벤트 핸들러(event handler)</strong>로 역할이 나뉘고 <strong>리스너(listener)</strong>로 연결해야 이벤트를 처리할 수 있다.
  </li>
    <ul>
      <li>
        <strong>이벤트 소스</strong>: 이벤트가 발생한 객체
      </li>
      <li>
        <strong>이벤트 핸들러</strong>: 이벤트 발생 시 실행할 로직이 구현된 객체
      </li>
      <li>
        <strong>리스너</strong>: 이벤트 소스와 이벤트 핸들러를 연결해 주는 함수
      </li>
    </ul>
  <li>
    대부분 이벤트 핸들러의 이름 형식은 <strong>"OnXXXListener"인 인터페이스</strong>를 구현해서 만든다.
  </li>
    <ul>
      <li>
        OnClickListener, onLongClickListener, OnItemClickListener 등의 인터페이스가 있다.
      </li>
    </ul>
  <li>
    인터페이스를 구현한 object 클래스를 이벤트 핸들러로 만들 수도 있지만 <strong>액티비티 자체에 인터페이스 구현, 이벤트 핸들러를 별도의 클래스로 만들어 처리, 코틀린의 SAM(single abstract method)</strong>를 활용하는 것도 가능하다.
  </li>
</ul>

```kotlin
// 1. 체크박스 이벤트 처리.
//   - 이벤트 소스: checkbox
//   - 리스터(이벤트 핸들러 등록): setOnCheckedChangedListener
//   - 이벤트 핸들러: object
binding.check.setOnCheckedChangeListener(object: CompoundButton.OnCheckedChangeListener {
    override fun onCheckedChanged(p0: CompoundButton?, p1: Boolean) {
        Log.d("kkang", "체크박스 클릭")
    }
})

// 2. 액티비티에서 인터페이스를 구현한 예.
class MainActivity3 : AppCompatActivity(), CompoundButton.OnCheckedChangeListener {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val binding = ActivityMain3Binding.inflate(layoutInflater)
        setContentView(binding.root)
        binding.checkbox.setOnCheckedChangeListener(this)
    }
    override fun onCheckedChanged(p0: CompoundButton?, p1: Boolean) {
        Log.d("kkang", "체크박스 클릭")
    }
}

// 3. 이벤트 핸들러를 별도의 클래스로 만든 예.
class MyEventHandler : CompoundButton.onCheckedChangeListener {
    override fun onCheckedChanged(p0: CompoundButton?, p1: Boolean) {
        Log.d("kkang", "체크박스 클릭")
    }
}
class MainActivity3 : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val binding = ActivityMain3Binding.inflate(layoutInflater)
        setContentView(binding.root)

        binding.checkbox.setOnCheckedChangeListener(MyEventHandler())
    }
}

// 4. SAM 기법으로 구현한 예.
class MainActivity3 : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState) 

        val binding = ActivityMain3Binding.inflate(layoutInflater)
        setContentView(binding.root)

        binding.checkbox.setOnCheckedChangeListener {
            compoundButton, b ->
            Log.d("kkang", "체크박스 클릭")
        }
    }
}
```

<h2>2-2. 클릭과 롱클릭 이벤트 처리</h2>
<ul>
  <li>
    안드로이드는 앱의 화면 구성에 필요한 <strong>여러 뷰를 제공</strong>하고, 대부분 <strong>뷰에서 자체 이벤트</strong> 또한 제공한다.
  </li>
    <ul>
      <li>
        따라서 뷰가 아무리 많아도 이벤트 처리 구조는 동일하다. (이벤트 소스 ↔ 이벤트 리스너 ↔ 이벤트 핸들러).
      </li>
    </ul>
  <li>
    <strong>ClickEvent</strong>와 <strong>LongClickEvent</strong>는 뷰의 최상위 클래스인 <strong>View</strong>에 정의된 이벤트이며 가장 많이 이용하는 이베늩이다.
  </li>
    <ul>
      <li>
        open fun setOnClickListener(l: View.onClickListener?): Unit
      </li>
      <li>
        open fun setOnLongClickListener(l: View.OnLongClickListener?): Unit
      </li>
    </ul>
  <li>
    ClickEvent는 OnClickListener를 구현한 객체를 이벤트 핸들러로, LongClickEvent는 OnLongClickListener를 구현한 객체를 이벤트 핸들러로 등록해야 한다.
  </li>
  <li>
    코틀린의 <strong>SAM 기법</strong>을 활용하면 코드를 더 간단하게 작성할 수 있다.
  </li>
    <ul>
      <li>
        SAM은 <strong>자바 API</strong>를 코틀린에서 활용할 때 <strong>람다 표현식</strong>으로 쉽게 이용할 수 있게 해주는 기법이다.
      </li>
      <li>
        SAM은 단어 뜻 그대로 <strong>하나의 추상 함수를 포함하는 인터페이스</strong>를 활용하는 방법이다.
      </li>
      <li>
        <strong>인터페이스</strong> 해당 인테페이스를 등록하는 <strong>세터(setter)</strong> 함수가 <strong>자바</strong>에 작성되어 있다면 <strong>코틀린에서 세터 함수</strong>를 통해 인터페이스를 구현한 객체를 등록할 때 <strong>람다 함수</strong>로 쉽게 등록이 가능하다.
      </li>
      <li>
        자바 함수를 코틀린에서 이용하려면 인터페이스를 구현한 객체를 <strong>매개변수로 지정</strong>해야 한다.
      </li>
      <li>
        <strong>추상 함수 하나</strong>를 포함하는 인터페이스만 SAM 기법으로 이용할 수 있다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 버튼의 클릭, 롱클릭 이벤트 처리.
binding.button.setOnClickListener {
    Log.d("kkang", "클릭 이벤트")
}

binding.button.setOnLongClickListener {
    Log.d("kkang", "롱클릭 이벤트")
    true
}
```

    }
