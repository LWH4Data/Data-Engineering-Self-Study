<h1>1.인텐트 이해하기</h1>
<h2>1-1. 인텐트란?</h2>
<ul>
  <li>
    안드로이드 앱은 모두 4개의 컴포넌트로 개발을 하며 이중 핵심이 되는 클래스가 <strong>Intent</strong>이다.
  </li>
  <li>
    Intent는 컴포넌트를 실행하려고 <strong>시스템에 전달하는 메시지</strong>라고 정의할 수 있다.
  </li>
    <ul>
      <li>
        기능을 수행하는 함수를 제공하는 클래스가 아니라 <strong>데이터를 담는 클래스</strong>이다.
      </li>
    </ul>
  <li>
    Intent에서 중요한 점은 시스템에서 <strong>앱의 컴포넌트를 어떻게 실행하는지 원리를 파악</strong>하는 것이다.
  </li>
  <li>
    안드로이드 컴포넌트 클래스는 개발자 코드가 아닌 <strong>시스템이 생성</strong>해서 실행하기 때문에 <strong>개발자 코드로 생명주기</strong>를 관리할 수 없다.
  </li>
    <ul>
      <li>
        시스템에 Intent를 전달하여야 하며 시스템은 Intent 정보를 분석해서 해당하는 컴포넌트를 실행한다.
      </li>
    </ul>
  <li>
    Intent의 중재 역할은 <strong>외부 앱의 컴포넌트와 연동</strong>할 때에도 동일하다.
  </li>
  <li>
    액티비티는 <strong>매니페스트 파일</strong>에 등록해야 하며 <strong>액티비티 클래스 하나당 activity 태그 하나</strong>로 등록한다. 이때 액티비티 클래스 이름을 지정하는 <strong>name 속성</strong>은 생략할 수 없다.
  </li>
    <ul>
      <li>
        액티비티뿐만 아니라 서비스, 브로드캐스트 리시버, 콘텐츠 프로바이더도 매니페스트 파일에 등록한다.
      </li>
    </ul>
  <li>
    안드로이드 컴포넌트를 매니페스트 파일에 등록해야하는 이유는 <strong>시스템에 컴포넌트를 알려야하기 때문</strong>이다.
  </li>
    <ul>
      <li>
        시스템은 <strong>런타임 때 매니페스트 파일의 정보를 참조</strong>하여 앱을 실행한다.
      </li>
    </ul>
</ul>

```xml
<!-- 1. MainActivity와 DetailActivity 등록. -->
<activity
    android:name=".DetailActivity"
    android:exported="true" />

<activity
    android:name=".MainActivity"
    android:exported="true">

    <intent-filter>
        <action android:name="android:intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
```

```kotlin
// 2. 인텐트를 시스템에 전달.
// Intent 객체이 DetailActivity를 실행하라는 정보를 담아 생성한다.
val intent: Intent = Intent(this, DetailActivity::class.java)

// startActivity() 함수가 인텐트를 시서템에 전달(시작)한다.
//   - 매개변수는 시스템에 실행을 요청할 컴포넌트의 정보가 담긴 Intent 객체이다.
startActivity(intent)
```

<h2>1-2. 인텐트 엑스트라 데이터</h2>
<ul>
  <li>
    인텐트에 컴포넌트 실행을 요청할 때 데이터를 함께 전달하려면 <strong>엑스트라 데이터(extra data)</strong>를 이용해야 한다.
  </li>
    <ul>
      <li>
        컴포넌트 객체는 시스템이 생성하기 때문에 개발자 코드로는 직접 접근이 불가하고 따라서 컴포넌트 객체로 어떤 함수를 호출할 수는 없다.
      </li>
    </ul>
  <li>
    인텐트에 엑스트라 데이터를 추가하는 함수는 putExtra() 이다.
  </li>
    <ul>
      <li>
        public intent putExtra(String name, CharSequence value)
      </li>
      <li>
        첫 번째 매개변수는 <strong>데이터의 식별자</strong>이고 두 번째 매개변수가 <strong>전달할 데이터</strong>이다.
      </li>
      <li>
        pubExtra() 함수는 각 타입의 데이터를 담을 수 있도록 <strong>오버로딩</strong>으로 선언되어 있다.
      </li>
    </ul>
  <li>
    인텐트로 실행된 컴포넌트에서 엑스트라 데이터를 가져오려면 <strong>인텐트 객체의 프로퍼티(intent)</strong>를 이용하면 된다. 이후 인텐트 객체의 <strong>getIntExtra() 함수</strong>로 데이터를 가져온다. 데이터를 가져오는 함수는 타입별로 여러 개 제공한다.
  </li>
    <ul>
      <li>
        public int getIntExtra(String name, int defaultValue)
      </li>
      <li>
        public String getStringExtra(String name)
      </li>
      <li>
        public double getDoubleExtra(String name, double defaultValue)
      </li>
    </ul>
</ul>

```kotlin
// 1. 엑스트라 데이터 추가.
val intent: Intent = Intent(this, DetailActivity::class.java)
intent.putExtra("data1", "hello")
intent.putExtra("data2", 10)
startActivity(intent)
```

```kotlin
// 2. 엑스트라 데이터 가져오기.
val data1 = intent.getStringExtra("data1")
val data2 = intent.getIntExtra("data2", 0)
```

<h2>1-3. 액티비티 화면 되돌리기 - ActivityResultLauncher</h2>
<ul>
  <li>
    액티비티는 화면을 구성하는 컴포넌트로 한 액티비티에서 다른 액티비티를 실행하면 화면이 전환된다. 이때 <strong>다시 돌아왔을 때</strong> 사후 처리가 필요한 경우가 있다.
  </li>
  <li>
    사후 처리 여부에 따라 인텐트로 액티비티를 시작하는 방법은 다음 3가지로 나뉜다. 보통 안드로이드 11버전이 나올 즈음부터는 androidx의 <strong>ActivityResultLauncher</strong>를 사용한다.
  </li>
    <ul>
      <li>
        public void startActivity(Intent intent)
      </li>
      <li>
        public void startActivityForResult(Intent intent, int requestCode)
      </li>
      <li>
        ActivityResultLauncher
      </li>
    </ul>
  <li>
    사후 처리가 필요 없을 때에는 startActivity() 함수를 사용한다.
  </li>
  <li>
    대표적으로 무언가를 실행하고 결과를 받는 것에는 <strong>퍼미션 허용 여부를 묻는 다이얼로그</strong>의 결과를 받을 때나 <strong>인텐트로 실행한 액티비티의 결과</strong>를 받을 때가 있다.
  </li>
  <li>
    사용자가 뒤로가기 버튼을 누르지 않고 자동으로 화면을 되돌릴 때에는 <strong>finish() 함수</strong>를 이용한다.
  </li>
    <ul>
      <li>
        finish() 함수는 현재 보이는 액티비티를 종료해 달라고 시스템에 요청한다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 결과와 화면 되돌리기.
// 자신을 실행한 인텐트 객체에 엑스트라 데이터로 데이터를 담을 수 있다.
intent.putExtra("resultData", "world")

// setResult() 함수로는 결과를 어떻게 되돌릴지 지정한다.
//   - RESULT_OK: 요청을 제대로 처리한 경우.
//   - RESULT_CANCELED: 요청을 제대로 처리하지 못한 경우.
setResult(RESULT_OK, intent)
finish()
```

<h3>1-3-1. Contract</h3>
<ul>
  <li>
    ActivityResultLauncher를 이용하려면 Contract 객체가 필요하다. Contract 객체는 ActivityResultLauncher로 실행될 <strong>요청을 처리</strong>하는 역할을 한다.
  </li>
    <ul>
      <li>
        ActivityResultLauncher로 인텐트를 발생시켜 액티비티를 실행할 때 <strong>인텐트를 발생</strong>시키는 역할을 한다.
      </li>
    </ul>
  <li>
    Contract는 <strong>ActivityResultContract</strong>를 상속받은 서브 클래스이며 <strong>직접</strong> 만들거나 <strong>API</strong>에서 제공하는 클래스를 이용하기도 한다. 대표적인 Contract로는 다음이 있다.
  </li>
    <ul>
      <li>
        PickContract: 선택한 연락처의 Uri 획득.
      </li>
      <li>
        RequestPermission: 권한 요청, 허락 여부 파악.
      </li>
      <li>
        RequestMultiplePermissions: 여러 권한을 동시에 요청.
      </li>
      <li>
        StartActivityForResult: 인텐트 발생, 액티비티 실행 결과 획득.
      </li>
      <li>
        TakePicturePreview: 사진 촬영 후 비트맵 획득.
      </li>
      <li>
        TakePicture: 사진 촬영, 저장, 비트맵 획득.
      </li>
    </ul>
</ul>

<h3>1-3-2. ActivityResultLauncher</h3>
<ul>
  <li>
    ActivityResultLauncher는 <strong>registerForActivityResult()</strong> 함수로 만드는 객체이다.
  </li>
  <li>
    함수의 매개변수로 실제 작업자인 <strong>Contract</strong> 객체와 결과를 처리하는 <strong>Callback</strong> 객체를 등록한다.
  </li>
</ul>

<h3>1-3-3. launch</h3>
<ul>
  <li>
    ActivityResultLauncher의 함수이면서 <strong>launch 함수</strong>를 호출하는 순간 ActivityResultLauncher에 등록된 <strong>Contract 객체가 실행</strong>된다.
  </li>
</ul>

```kotlin
// 1. ActivityResultLauncher 생성.
// ActivityResultLauncher를 이용하기 위해 우선 registerForActivityResult() 함수로 ActivityResultLauncher 객체를 생성한다.
val requestLauncher: ActivityResultLauncher<Intent> = registerForActivityResult(

    // Contract 함수 등록.
    //   - 인텐트를 발생시키는 StartActivityForResult 객체를 사용한다.
    ActivityResultContracts.StartActivityForResult())
    
    // Callback 함수 등록.
    { 
        val resultData = it.data?.getStringExtra("result")
        binding.mainResultView.text = "result : $resultData"
    }
```

```kotlin
// 2. ActivityResultLauncher 실행.
val intent: Intent = Intent(this, DetailActivity::class.java)
// 생성한 ActivityResultLauncher 객체를 launch() 함수로 실행한다.
requestLauncher.launch(intent)
```

<h2>1-4. 인텐트 필터</h2>
<ul>
  <li>
    같은 앱의 컴포넌트는 앞서 살펴본 것과 같이 클래스 타입 레퍼런스(DetailActivity::class.java)로 설정할 수 있지만 외부 앱의 컴포넌트는 불가능하기 때문에 설정 방법에 따라 다음 두 가지로 나뉜다.
  </li>
    <ul>
      <li>
        명시적 인텐트: <strong>클래스 타입 레퍼런스 정보</strong>를 활용한 인텐트이다.
      </li>
      <li>
        암시적 인텐트: <strong>인텐트 필터 정보</strong>를 활용한 인텐트이다.
      </li>
    </ul>
  <li>
    <strong>내부 앱</strong>의 컴포넌트를 요청하는 인텐트 객체를 만들 때에는 <strong>명시적 인텐트(explicit intent)</strong>를 사용하고, <strong>외부 앱</strong>의 경우 컴포넌트 클래스 타입 래퍼런스를 활용할 수 없기 때문에 <strong>암시적 인텐트(implicit intent)</strong>를 이용한다.
  </li>
  <li>
    암시적 인텐트는 <strong>매니페스트 파일</strong>에 선언된 <strong>인텐트 필터</strong>를 이용한다.
  </li>
  <li>
    intent-filter는 activity, service, receiver 등 <strong>컴포넌트 등록 태그 하위</strong>에 작성할 수 있으며 <strong>암시적 인텐트로 실행</strong>할 때만 추가하고 반드시 사용할 필요는 없다.
  </li>
    <ul>
      <li>
        이렇게 intent-filter를 추가하면 해당 컴포너트의 클래스명과 인텐트 필터 정보가 시스템에 등록된다. 이후 외부에서 컴포넌트를 실행할 때 등록된 정보에 맞게 인텐트를 설정한다.
      </li>
    </ul>
  <li>
    intent-filter 하위에는 <strong>action, category, data 태그</strong>를 이용해 정보를 설정할 수 있다.
  </li>
    <ul>
      <li>
        action: 컴포넌트의 기능을 나타내는 문자열이다.
      </li>
        <ul>
          <li>
            android:name에 설정하는 값은 개발자가 임의로 지정하지만, 일반적으로 <strong>컴포넌트의 기능</strong>을 나타낼 것을 권장한다.
          </li>
        </ul>
      <li>
        category: 컴포넌트가 포함되는 범주를 나타내는 문자열이다.
      </li>
        <ul>
          <li>
            android:name에 설정하는 문자열은 컴포넌트가 <strong>어느 범주</strong>에 포함되어야 하는지를 의미한다. 개발자가 임의로 설정이 가능하지만 대부분 <strong>플랫폼 API가 제공하는 문자열</strong>을 이용한다.
          </li>
        </ul>
      <li>
        data: 컴포넌트에 필요한 데이터 정보이다.
      </li>
        <ul>
          <li>
            컴포넌트에서 <strong>어떤 성격의 데이터</strong>를 처리하는지를 의미한다.
          </li>
          <li>
            <strong>URL 형식</strong>으로 표현하고 위의 두 태그와 달리 android:scheme(URL 프로토콜명), android:host(URL의 도메인), android:port(URL의 포트), android:mimeType(데이터 타입) 등의 <strong>속성</strong>을 이용한다. (속성은 필요한 만큼만 선언한다).
          </li>
        </ul>
    </ul>
  <li>
    안드로이드의 홈 화면도 결국 <strong>하나의 앱</strong며 우리가 보는 앱의 아이콘은 <strong>인텐트 필터로 설정된 액티비티</strong>이다. 따라서 앱의 아이콘을 터치하면 해당 앱의 텐트 필터를 통해 앱을 실행한다.
  </li>
</ul>

```xml
<!-- 1. 암시적 인텐트 -->
<!-- 매니페스트 파일에 어떤 액티비티를 등록할 때 아래와  name 속성만 지정하면 명시적 인텐트로만 실행할 수 있다. -->
<!-- 안드로이드 시스템은 매니페스트 파일의 내용을 참조한→ name 속성만 있으면 이것만 인지한다. -->
<activity android:name=".OneActivity" />
<activity
    android:name=".TwoActivity"
    android:exported="true">
    
    <!-- intent-filter를 설정하여 외부에서도 인텐트로 실행할 수 있게 한다. -->
    <intent-filter>
        <action android:name="ACTION_EDIT" />
        <category android:name="android.intent.category.DEFAULT" />
    </intent-filter>
</activity>
```

```xml
<!-- 2. 자동으로 만들어지는 메인 액티비티. -->
<activity android:name=".MainActivity"
    android:exported="true" >

    <intent-filter>
        <action android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
```

```xml
<!-- 3. 외부 앱과 연동하는 인텐트 필터 설정(메인 액티비티 외). -->
<activity android:name=".TwoActivity"
    android:exported="true">

    <intent-filter>
        <action android:name="ACTION_EDIT" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:scheme="http" />
    </intent-filter>
</activity>
```

```kotlin
// 4. 인텐트의 프로퍼티를 이용하는 방법.
// 인텐트 정보에 카테고리 정보를 설정하지 않으면 DEFAULT가 자동 할당된다.
// 만약 <category> 태그의 name 속성값의 android.intent.category가 DEAFULT가 아니라면 설정이 필요하다.
val intent = Intent()

// action 정보를 intent의 프로퍼티에 설정.
intent.action = "ACTION_EDIT"

// data 정보를 intent의 프로퍼티에 설정.
intent.data = Uri.parse("http://google.com")
startActivity(intent)
```

```kotlin
// 5. 인텐트의 생성자를 이용하는 방법.
val intent = Intent("ACTION_EDIT", Uri.parse("http://www.google.com"))
startActivity(intent)
```

```xml
<!-- 6. data의 mimeType 설정. -->
<activity android:name=".TwoActivity">
    <intent-filter>
        <action android:name="ACTION_EDIT" />
        <category android:name="android.intent.category.DEFUALT" />
        <data android:mimeType="image/*" />
    </intent-filter>
</activity>
```

```kotlin
// 7. 6번에서 설정한 것에 따라 이를 실행하는 인텐트에도 타입 정보를 설정한다.
val intent = Intent("ACTION_EDIT")
intent.type = "image/*"
startActivity(intent)
```

<h2>1-5. 액티비티 인텐트 동작 방식</h2>
<ul>
  <li>
    명시적 인텐트는 클래스 타입 레퍼런스 정보를 이용하기 때문에 <strong>액티비티가 없거나 여러 개일 수 없다</strong>. 반면 암시적 인텐트는 실행할 액티비티를 action, category, data 등의 문자열 정보로 나타내기 때문에 <strong>없거나 여러 개일 수 있다</strong>.
  </li>
    <ul>
      <li>
        실행할 액티비티가 없을 때: 인텐트를 시작한 곳에 오류가 발생한다.
      </li>
      <li>
        실행할 액티비티가 1개 일 때: 문제 없이 실행된다.
      </li>
      <li>
        실행할 액티비티 n개일 때: 사용자 선택으로 하나만 실행한다.
      </li>
    </ul>
  <li>
    실행할 액티비티가 여러 개라면 다이얼로그를 띄워서 <strong>사용자에게 어떤 앱의 액티비티를 실행할지</strong> 묻는다.
  </li>
    <ul>
      <li>
        사용자에게 묻지 않고 특정 앱의 액티비티를 실행하고 싶다면 앱의 <strong>패키지명</strong>을 지정한다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 해당 액티비티가 없을 때
val intent = Intent("ACTION_HELLO")
startActivity(intent)
// android.content.ActivityNotFoundException: No Activity found to handle Intent { act=ACTION_HELLO } 오류 발생.
```

```kotlin
// 2. 해당 액티비티가 없을 때 예외 처리.
val intent = Intent("ACTION_HELLO")
try {
    startActivity(intent)
} catch (e: Exception) {
    Toast.makeText(this, "no app...", Toast.LENGTH_SHORT).show()
}
```

```kotlin
// 3. 액티비티가 여러 개일 때
val intent = Intent(Intent.ACTION_VIEW, Uri.parse("geo:37.7749, 127.4194"))
startActivity(intent)
```

```kotlin
// 4. 패키지명을 지정하여 지정한 액티비티가 실행되도록 설정.
val intent = Intent(Intent.ACTION_VIEW, Uri.parse("geo:37.7749, 127.4194"))
// 앱의 패키지명을 지정.
intent.setPackage("com.google.android.apps.maps")
startActivity(intent)
```

<h1>2. 액티비티 생명주기</h1>
<h2>2-1. 액티비티의 상태</h2>
<ul>
  <li>
    생명주기(life cycle)란 액티비티가 <strong>생성되어 소멸</strong>하기까지의 과정을 의미한다. 
  </li>
  <li>
    Activity 클래스는 액티비티가 <strong>상태 변화</strong>를 알아차릴 수 있는 여러 <strong>콜백 함수</strong>를 제공한다.
  </li>
    <ul>
      <li>
        콜백 함수는 액티비티의 상태가 바뀔 때마다 앱이 어떻게 동작해야 하는지 구현한다.
      </li>
    </ul>
  <li>
    액티비티의 상태는 크게 다음 세 가지로 구분할 수 있다.
  </li>
    <ul>
      <li>
        활성: 액티비티 <strong>화면이 출력</strong>되고 있고 사용자가 <strong>이벤트를 발생</strong>시킬 수 있는 형태.
      </li>
      <li>
        일시 정지: 액티비티 <strong>화면이 출력</strong>되고 있지만 사용자가 <strong>이벤트를 발생시킬 수 없는 상태</strong>.
      </li>
      <li>
        비활성: 액티비티 화면이 <strong>출력되고 있지 않는 상태</strong>.
      </li>
    </ul>
</ul>

<h3>2-1-1. 활성 상태</h3>
<ul>
  <li>
    활성 상태는 액티비티가 실행되어 화면에 나오고 사용자 이벤트 처리를 할 수 있는 상태 즉 <strong>포커스</strong>를 가지는 상태를 의미한다. <strong>버튼을 클릭</strong>하거나 <strong>에디트 텍스트</strong>에 글을 입력할 수 있다.
  </li>
  <li>
    처음 실행된 액티비티는 <strong>onCreate() → onStart() → onResume()</strong> 함수까지 호출한다. 이후 <strong>setContentView()</strong> 함수로 출력한 내용이 액티비티 화면에 나온다.
  </li>
    <ul>
      <li>
        <strong>onStart()</strong>나 <strong>onResume()</strong> 함수에서 setContentView() 함수를 호출해도 화면이 출력된다.
      </li>
      <li>
        onCreate() 함수는 <strong>최초에 한 번</strong>만 호출되고 onStart()나 onResume() 함수는 <strong>반복해서 호출</strong>할 수 있으므로 일반적으로 setContentView()는 <strong>onCreate()</strong> 함수에서 호출한다.
      </li>
    </ul>
</ul>

<h3>2-1-2. 일시 정지 상태</h3>
<ul>
  <li>
    일시 정지 상태는 <strong>onPause() 함수</strong>까지 호출된 상태이다.
  </li>
  <li>
    일시 정지 상태는 액티비티가 화면에 보이지만 포커스를 잃어 <strong>사용자 이벤트를 처리할 수 없는 상태</strong>를 의미한다.
  </li>
  <li>
    액티비티가 <strong>다시 포커스</strong>를 얻어 사용자 이벤트를 처리할 수 있으면 <strong>onResume() 함수가 자동으로 호출</strong>된다.
  </li>
</ul>

<h3>2-1-3. 비활성 상태</h3>
<ul>
  <li>
    비활성 상태란 액티비티가 종료되지 않고 <strong>화면에만 보이지 않는 상태</strong>를 의미한다.
  </li>
    <ul>
      <li>
        <strong>인텐트</strong>로 다른 액티비티가 실행됬거나 <strong>홈 버튼</strong>을 눌러 런처 화면으로 이동해 액티비티가 보이지 않는 상황이다.
      </li>
      <li>
        안드로이드폰의 홈 버튼은 <strong>런처 앱의 액티비티</strong>를 실행한다. 즉, 홈 버튼은 액티비티가 종료되는 것이 아니라 단지 <strong>비활성 상태</strong>가 되는 것이다.
      </li>
    </ul>
  <li>
    활성 상태에서 비활성 상태가 되면 <strong>onPause() → onStop()</strong> 함수까지 호출된다.
  </li>
    <ul>
      <li>
        다시 액티비티가 화면에 보이면 <strong>onRestart() → onStart() → onResume()</strong> 함수까지 호출되어 <strong>활성 상태</strong>가 된다.
      </li>
    </ul>
  <li>
    액티비티가 <strong>종료</strong>된다는 것은 <strong>onDestroy()</strong>까지 호출되었다는 의미이다.
  </li>
    <ul>
      <li>
        사용자가 뒤로가기 버튼을 눌러 액티비티를 벗어나거나 코드에서 finish() 함수를 호출하는 상황이 해당한다.
      </li>
    </ul>
  <li>
    onCreate()만 사용하지 않고 생명주기 함수를 쓰는 이유는 앱이 화면에 <strong>보이지 않을 때에도 작업이 게속 진행되어</strong> 네트워크 트래픽이 과도하게 발생하거나 배터리가 소모하는 등의 상황을 방지하기 위해서이다.
  </li>
</ul>

<br>

<h2>2-2. 액티비티의 상태 저장</h2>
<ul>
  <li>
    일반적으로 액티비티가 종료되면 <strong>객체가 소멸</strong>되고 데이터가 모두 사라지며 다시 액티비티를 실행하면 <strong>초기 상태</strong>로 나온다. 따라서 액티비티가 종료될 때 유지해야할 데이터가 있다면 <strong>저장해 두었다가 실행할 때 복원</strong>해야 한다.
  </li>
  <li>
    액티비티가 종료되는 상황 중 <strong>finish() 함수를 호출</strong>하거나 액티비티에서 <strong>사용자가 폰의 뒤로가기 버튼</strong>을 누르거나 화면에 오랫동안 나오지 않아 <strong>시스템이 알아서 종료</strong>되는 상황은 액티비티를 다시 실행해도 초기 상태로 나오기 때문에 <strong>상태를 저장했다 복원할 필요가 없다</strong>.
  </li>
  <li>
    <strong>상태를 저장</strong>한다는 것은 액티비티가 종료되어 메모리의 데이터가 사라지더라도 실행할 때 사용자가 저장한 데이터로 <strong>액티비티의 상태를 복원</strong>하겠다는 의미이다.
  </li>
    <ul>
      <li>
        대표적인 예로 회면을 회전하는 경우가 있다. 화면을 회전하면 액티비티가 종료되었다가 나온다.
      </li>
    </ul>
  <li>
    액티비티를 종료할 때 저장했다가 복원해야 할 데이터가 있다면 <strong>Bundle</strong>이라는 객체에 담아주면 된다.
  </li>
    <ul>
      <li>
        다른 생명주기 함수는 매개변수를 가지지 않지만 onCreate(), onSavedInstanceState(), onRestoreInstanceState() 함수는 매개변수를 가지며 모두 <strong>Bundle 객체</strong>이다. 이 Bundle 객체를 통해 데이터를 저장하고 복원한다.
      </li>
    </ul>
  <li>
    onSaveInstanceState()는 onStop() 함수 다음에 호출되기 때문에 해당 함수가 호출된다는 것은 <strong>액티비티가 종료</strong>됨을 의미한다. 따라서 매개변수 <strong>Bundle 객체</strong>에 데이터를 담아 주면 자동으로 <strong>데이터 파일</strong>로 저장해 둔다.
  </li>
    <ul>
      <li>
        putString(), putInt() 등의 함수로 데이터를 저장하며 <strong>자체 캐싱 파일</strong>에 데이터를 저장해둔다.
      </li>
      <li>
        다시 액티비티가 생성되어 실행될 때 캐싱 파일이 있다면 해당 내용을 읽어 번틀 객체에 담아 <strong>onCreate(), onRestoreInstanceState() 함수의 매개변수로 전달</strong>한다.
      </li>
      <li>
        번들 객체에서 데이터를 가져올 때에는 getString(), getInt() 등의 함수를 이용한다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 번들 객체 사용.
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
}

override fun onRestoreInstanceState(savedInstanceState: Bundle) {
    super.onRestoreInstanceState(savedInstanceState)
}

override fun onSaveInstanceState(outState: Bundle) {
    super.onSaveInstanceState(outState)
}
```

```kotlin
// 2. 번들에 데이터 저장.
// outState Bundle 객체에 데이터를 설정하여 저장한다.
override fun onSaveInstanceState(outState: Bundle) {
    super.onSaveInstanceState(outState)

    // putString() 함수를 통해 데이터 저장.
    outState.putString("data1", "hello")

    // putInt() 함수를 통해 데이터를 저장.
    outState.putInt("data2", 10)
}
```

```kotlin
// 3. 번들에 저장된 데이터 가져오기.
override fun onRestoreInstanceState(savedInstanceState: Bundle) {
    super.onRestoreInstanceState(savedInstanceState)
    val data1 = savedInstanceState.getString("data1")
    val data2 = savedInstanceState.getInt("data2")
}
```

<h1>3. 액티비티 ANR 문제와 코루틴</h1>
<h2>3-1. ANR 문제란?</h2>
<ul>
  <li>
    ANR(activity not response)은 <strong>액티비티가 응답하지 않는 오류 상황</strong>을 의미한다. 이를 고려하지 않으면 앱이 수시로 종료될 수 있다.
  </li>
  <li>
    액티비티로 구성한 앱 화면은 액티비티가 사용자 이벤트에 <strong>5초 이내</strong>로 반응하지 않으면 ANR 오류가 발생한다. 
  </li>
    <ul>
      <li>
        사용자 이벤트를 처리하지 못하는 이유는 액티비티를 실행한 시스템에서 발생한 수행 흐름에서 시간이 오래 걸리는 작업이 끝나지 않는 경우이다.
      </li>
    </ul>
  <li>
    시스템에서 <strong>액티비티를 실행하는 수행 흐름</strong>을 <strong>메인 스레드</strong> 또는 화면을 출력하는 수행 흐름이라는 의미에서 <strong>UI 스레드</strong>라 한다.
  </li>
    <ul>
      <li>
        메인 스레드 중간에 이벤트 처리가 오래걸리는 문제로 인한 ANR 오류를 고려해야 한다.
      </li>
    </ul>
  <li>
    액티비티에서 시간이 오래 걸리는 대표적인 작업은 서버와 통신하는 <strong>네트워크</strong>이다. 특히 네트워크가 불안정할 경우 이벤트 처리 시간이 길어져 ANR 오류가 발생할 가능성은 커진다.
  </li>
    <ul>
      <li>
        단, 네트워크의 경우 대부분 네트워크 통신을 지원하는 전문 라이브러리(Volley 또는 Retrofitw2)를 사용하면 이미 ANR 문제를 처리해두어 신경 쓸 필요는 없다.
      </li>
    </ul>
  <li>
    ANR 문제를 해결하는 방법은 액티비티를 실행한 <strong>메인 스레드 이외에 실행 흐름(개발자 스레드)</strong>을 따로 만들어 시간이 오래 걸리는 작업을 담당하게 하면 된다.
  </li>
    <ul>
      <li>
        처리가 오래 걸리는 이벤트를 개발자 스레드에 배정하면 메인 스레드는 영향을 받지 않는다.
      </li>
    </ul>
  <li>
    단, 개발자 스레드를 이용하는 경우 ANR 오류는 해결되지만 <strong>화면을 변경할 수 없다</strong>는 문제가 발생한다.
  </li>
    <ul>
      <li>
        화면 변경은 개발자가 만든 스레드에서는 불가하고 <strong>메인 스레드</strong>에서만 할 수 있기 때문이다.
      </li>
    </ul>
</ul>

<br>

<h2>3-2. 코루틴으로 ANR 오류 해결</h2>
<ul>
  <li>
    화면 변경이 가능하면서 ANR 오류를 해결할 때에는 코틀린 언어가 제공하는 <strong>코루틴(coroutine)이라는 기능</strong>을 사용한다.
  </li>
  <li>
    ANR 문제에 스레드-핸들러나 AsyncTask를 <strong>예전에는 이용</strong>하였으나 두 기술 모두 API 30에서 deprecated 되었으며 코루틴이 복잡한 로직을 처리하기 때문에 <strong>비즈니스 로직에만 집중</strong>할 수 있다는 장점이 있다.
  </li>
</ul>

<h3>3-2-1. 코루틴이란?</h3>
<ul>
  <li>
    코루틴은 한마디로 <strong>비동기 경량 스레드(non-blocking lightweight thread)</strong>라고 요약할 수 있다. 안드로이드 시스템이 아니라 <strong>프로그래밍 언어</strong>에서 제공하는 기능이다.ㄴ
  </li>
    <ul>
      <li>
        coroutine에서 co는 함께를, routine을 작업의 처리 단위를 의미하며 결국 <strong>비동기 처리 방식</strong>과 동일하다.
      </li>
    </ul>
  <li>
    스레드는 성능에 문제가 많고 자유롭게 지어할 수 없거나 구현하기도 복잡하다는 문제가 있다. 반면 코루틴은 스레드보다 가벼우면서 더 많은 기능을 제공한다. (안드로이드도 코루틴을 권장한다).
  </li>
    <ul>
      <li>
        경량이다.
      </li>
      <li>
        메모리 누수가 적다.
      </li>
      <li>
        취소 등 다양한 기능을 지원한다.
      </li>
      <li>
        많은 제트팩 라이브러리에 적용되어 있다.
      </li>
    </ul>
</ul>

<h3>3-2-2. 안드로이드에서 코루틴 이용</h3>
<ul>
  <li>
    안드로이드 앱에서 코루틴을 사용하려면 빌드 그래들 파일의 dependencies 항목에 다음을 등록한다.
  </li>
    <ul>
      <li>
        implementation("org.jetbrains.kotlinx:kotlinx-coroutines-android:1.7.3")
      </li>
    </ul>
  <li>
    코루틴을 구동하려면 먼저 <strong>스코프(scope)</strong>를 준비해야 하며 스코프에서 코루틴을 구동한다.
  </li>
    <ul>
      <li>
        스코프는 <strong>성격이 같은 코루틴을 묶는 개념</strong>으로 이해할 수 있다.
      </li>
      <li>
        한 스코프에 여러 코루틴을, 한 애플리케이션에 여러 스코프를 만들 수 있다.
      </li>
    </ul>
  <li>
    코루틴 스코프는 <strong>CoroutineScope를 구현한 클래스의 객체</strong>이며 직접 구현할 수도 있고 GlobalScope, ActorScope, ProducerScope 등 코틀린 언어가 제공하는 스코프를 이용할 수도 있다.
  </li>
  <li>
    스코프를 만들 때 디스패처(dispatchers)로 어디서 동작해야할지를 나타낸다.
  </li>
    <ul>
      <li>
        Dispatchers.Main: 액티비티의 <strong>메인 스레드</strong>에서 동작하는 코루틴을 만든다.
      </li>
        <ul>
          <li>
            메인 스레드는 사용자 이벤트를 처리하는 곳이기 때문에 빨리 끝나는 작업을 맡기는 것이 좋다.
          </li>
        </ul>
      <li>
        Dispatchers.IO: <strong>파일에 읽거나 쓰기</strong> 또는 <strong>네트워크 작업</strong> 등에 최적화되어 있다.
      </li>
      <li>
        Dispatchers:Default: <strong>CPU</strong>를 많이 사용하는 작업을 백그라운드에서 실행한다.
      </li>
    </ul>
  <li>
    Channel 클래스는 코루틴 값을 전달받을 수 있는 방법을 제공한다.
  </li>
    <ul>
      <li>
        Channel은 큐(queue) 알고리즘과 비슷하며 Channel의 <strong>send() 함수</strong>로 데이터를 전달하면 해당 데이터를 받는 코루틴에서는 <strong>receive()</strong>나 <strong>consumeEach()</strong> 등의 함수로 데이터를 받는다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 시간이 오래 걸리는 작업 예.
var sum = 0L

// 버튼을 클릭하면 시간이 올래 걸리는 작업이 진행된다.
var time = measureTimeMillis {
    for (i in 1..2_000_000_000) {
        sum += i
    }
}

Log.d("kkang", "time: $time")

// 작업이 완료되면 결괏값을 뷰에 출력한다.
binding.resultView.text = "sum : $sum"
```

```kotlin
// 2. 스레드-핸들러 구조로 작성한 코드.
val handler = object. Handler() {
    override fun handleMessage(msg: Message) {
        super.handleMessage(msg)
        bidning.resultView.text = "sum : ${msg.arg1}"
    }
}

// 시간이 오래 걸리는 작업을 개발자 스레드로 작성한다.
thread {
    var sum = 0L
    var time = measureTimeMillis {
        for (i in 1..2000_000_000) {
            sum += i
        }

        val message = Message()
        message.arg1 = sum.toInt()
        handler.sendMessage(message)
    }
    Log.d("kkang", "time : $time")
}
```

```kotlin
// 3. 코루틴으로 작성한 소스. 
// Channel 클래스는 코루틴 값을 전달받을 수 있는 방법을 제공한다.
val channel = Channel<Int>()

// Dispatcher로 해당 스코프에서 구동한 코루틴이 어디에서 동작해야 하는지를 나타낸다.
//   - 현재는 Dispatchers.Default에서 구동된다.
val backgroundScope = CoroutineScope(Dispatchers.Default + Job())
backgroundScope.launch {
    var sum = 0L
    var time = measureTimeMillis {
        for (i in 1..2_000_000_000) {
            sum += i
        }
    }
    Log.d("kkang", "time : $time")
    
    // send() 함수로 데이터를 전달한다.
    channel.send(sum.toInt())
}

// Dispatchers.Main에서 구동된다.
val mainScope = GlobalScope.launch(Dispatchers.Main) {
    
    // channel의 consumeEach() 함수로 데이터를 받는다.
    channel.consumeEach {
        binding.resultView.text = "sum : $it"
    }
}
```
