<ul>

<h2>3-2. 서비스의 백그라운드 제약</h2>
<ul>
  <li>
    서비스는 앱이 <strong>백그라운드 상태일 때 인텐트를 전달</strong>하면 오류가 발생한다. 포그라운드 상황에서 잘실행되면 인텐트도 백그라운드 상황에서는 오류가 발생한다.
  </li>
  <li>
    <strong>포그라운드(foreground)</strong>는 앱의 화면이 <strong>전면에 보이는 상황</strong>이고, <strong>백그라운드(background)</strong>는 앱의 <strong>화면이 안 보이는 상황</strong>이다.
  </li>
  <li>
    서비스가 정상적으로 실행되는 포그라운드 상황은 다음과 같다.
  </li>
    <ul>
      <li>
        액티비티가 시작되든 일시 중지되든 상관없이 <strong>보이는 액티비티</strong>가 있을 때.
      </li>
      <li>
        <strong>포그라운드 서비스</strong>가 있을 때.
      </li>
      <li>
        앱의 서비스에 바인딩하거나 앱으 콘텐츠 프로바이더를 사용해 <strong>또 다른 포그라운드 앱</strong>이 연결되었을 때 
      </li>
    </ul>
  <li>
    위의 포그라운드 상황 외에는 모두 백그라운드 상황으로 간주한다. 백그라운드 상황에서도 다음과 같은 경우 서비스가 정상적으로 작동한다.
  </li>
    <ul>
      <li>
        <strong>우선순위가 높은 파이어베이스 클라우드 메시징(FCM)</strong> 처리.
      </li>
      <li>
        SMS/MMS 메시지와 같은 <strong>브로드캐스트 수신</strong>.
      </li>
      <li>
        알림에서 <strong>PendingIntent 실행</strong>.
      </li>
      <li>
        VPN 앱이 포그라운드로 승격되기 전에 <strong>VpnService</strong> 시작.
      </li>
    </ul>
  <li>
    앱이 백그라운드 상황이더라도 <strong>startForegroundService()</strong> 함수로 인텐트를 시작하면 앱이 <strong>백그라운드 상황에서도 서비스가 실행</strong>된다.
  </li>
    <ul>
      <li>
        단, startForegroundService()로 실행한 서비스는 <strong>결국 오류</strong>가 발생하며 서비스를 <strong>정상으로 유지할 수는 없다</strong>.
      </li>
      <li>
        따라서 startForegroundService()로 실행한 경우 빨리 <strong>startForeground()</strong> 함수를 호출해 <strong>포그라운드 상황</strong>으로 만들어야 한다.
      </li>
        <ul>
          <li>
            startForeground() 함수의 매개변수는 <strong>알림 객체</strong>이다. 따라서 알림을 발생시키며 사용자에게 <strong>앱이 실행되고 있음을 알린다</strong>.
          </li>
          <li>
            startForeground() 함수를 <strong>호출하는 서비스</strong>는 <strong>매니페스트 설정</strong>이 필요한다.
          </li>
            <ul>
              <li>
                service 태그에 foregroundServiceType으로 지정하며 포그라운드에서 어떤 작업을 하는 서비스인지 등록한다.
              </li>
              <li>
                이후 퍼미션이 등록되어 있어야 한다. 퍼미션이 필요한 이유는 앱이 포그라운드 서비스를 가지고 있음을 명시(android.permission.FOREGROUND_SERVICE)하고 서비스의 알림을 발생(android.permission.POST_NOTIFICATION)시키기 위해서이다.
              </li>
            </ul>
        </ul>
    </ul>
</ul>

<h1>4. 잡 스케줄러</h1>
<ul>
  <li>
    앱이 백그라운드 상황일 때 작업 처리에 제약을 받는 문제는 <strong>잡 스케줄러</strong>를 이용해 보완할 수 있다.
  </li>
  <li>
    잡 스케줄러도 개발자가 만드는 서비스이기 때문에 어떤 상황에서 서비스를 실행해야 하는지 <strong>조건을 명시</strong>해주어야 한다.
  </li>
</ul>

<br>

<h2>4-1. 잡 스케줄러의 실행 조건</h2>
<ul>
  <li>
    잡 스케줄러에 조건으로 명시할 수 있는 상황은 다음과 같다.
  </li>
    <ul>
      <li>
        네트워크 타입
      </li>
      <li>
        배터리 충전 상태
      </li>
      <li>
        특정 앱의 콘텐츠 프로바이더 갱신(대표적으로 갤러리 앱)
      </li>
      <li>
        실행 주기
      </li>
      <li>
        최소 지연 시간
      </li>
      <li>
        시스템 재구동 시 현재 조건 유지 여부
      </li>
    </ul>
</ul>

<br>

<h2>4-2. 잡 스케줄러의 3가지 구성 요소</h2>
<ul>
  <li>
    <strong>잡 서비스</strong>: 백그라운드에서 처리할 작업을 <strong>구현한 서비스</strong>이다.
  </li>
  <li>
    <strong>잡 인포</strong>: <strong>잡 서비스 정보</strong>와 <strong>실행될 조건</strong>을 지정한다.
  </li>
  <li>
    <strong>잡 스케줄러</strong>: 잡 인포를 <strong>시스템에 등록</strong>한다.
  </li>
</ul>

<h3>4-2-1. 잡 서비스 - 백그라운드 작업 구현</h3>
<ul>
  <li>
    잡 서비스는 개발자가 만드는 서비스이기 때문에 <strong>매니페스트에 service 태그</strong>로 등록하며 <strong>android.permission.BIND_JOB_SERVICE 퍼미션</strong>도 포함한다.
  </li>
  <li>
    매니페스트 파일에 등록한 후 <strong>JobService</strong>를 상속받아 onCreate(), onStartCommand(), onDestroy()와 같은 <strong>생명주기 함수</strong>를 <strong>재정의한 클래스</strong>를 작성한다.
  </li>
  <li>
    서비스의 생명주기 함수 중 onStartCommand() 함수는 호출되지 않기에 필요가 없다. 반면 <strong>onStartJob()</strong>, <strong>onStopJob()</strong> 함수는 반드시 재정의 해야한다.
  </li>
    <ul>
      <li>
        onStartJob() 함수에는 <strong>백그라운드에서 처리할 작업</strong>을 구현한다. 함수의 반환값은 <strong>Boolean 타입</strong>이다.
      </li>
        <ul>
          <li>
            false: 작업이 완벽하게 종료되었음을 의미한다.
          </li>
            <ul>
              <li>
                시스템은 더 처리할 필요가 없다고 판단하고 onStopJob() 함수를 건너뛰고 바로 <strong>onDestroy() 함수</strong>를 호출해 <strong>서비스를 종료</strong>한다.
              </li>
            </ul>
          <li>
            true: 작업이 아직 끝나지 않았음을 의미한다.
          </li>
            <ul>
              <li>
                아직 처리할 작업이 남아있는 것으로 판단하고 onDestroy() 함수를 호출하지는 않는다.
              </li>
              <li>
                이때에는 오래 걸리는 작업을 스레드 등에서 처리하고 끝날 때 <strong>jobFinished() 함수</strong>를 호출하는 식으로 onStartJob() 함수를 구현한다.
              </li>
            </ul>
        </ul>
    </ul>
  <li>
    onStopJob() 함수가 호출되는 경우는 onStartJob() 함수에서 true를 반환해 서비스가 오랜 시간 살아있는 중 갑자기 잡 스케줄러를 <strong>실행하는 조건</strong>이 바뀌거나 어디선가 <strong>cancel() 함수로 취소</strong>했을 때이다.
  </li>
    <ul>
      <li>
        시스템은 이러한 상황에서 잡 스케줄러가 <strong>비정상으로 종료된 것</strong>으로 인지하고 서비스를 종료하기 전 처리할 로직을 실행하고자 <strong>onStopJob() 함수</strong>를 호출한다.
      </li>
      <li>
        onStopJob() 함수의 반환값도 Boolean 타입이며 해당 작업을 다시 시스템에 등록하거나 취소할지를 결정한다.
      </li>
        <ul>
          <li>
            <strong>false</strong>인 경우 <strong>잡 스케줄러 등록 취소</strong>를, <strong>true</strong>인 경우 잡 <strong>스케줄러에 재등록</strong> 한다.
          </li>
        </ul>
    </ul>
  <li>
    JobInfo.Builder에 지정한 잡 서비스가 실행되는 조건을 명시할 때에는 다음과 같은 세터 함수를 이용한다. (p475 참고).
  </li>
    <ul>
      <li>
        setPersistsed(true): 기기를 재부팅해도 작업 등록을 유지해야 하는지 설정한다.
      </li>
      <li>
        setPeriodic(long intervalMillis): 작업의 실행 주기를 설정한다.
      </li>
      <li>
        setMinimumLatency(long minLatencyMillis): 작업의 실행 지연 시간을 설정한다.
      </li>
      <li>
        ...
      </li>
      <li>
        setRequiresCharing(boolean requiresCharging): 배터리가 충전 상태인지를 설정한다.
      </li>
    </ul>
</ul>

```xml

    android:exnabled="true"
```

```kotlin
    // 등록할 작업의 식별값이다. 후에 cancel() 함수로 작업을 취소할 때 이 값을 사용한다.
    1,

    // 등록할 잡 서비스이다. 해당 서비스를 시스템에 등록하겠다는 의미이다.
    ComponentName(this, MyJobService::class.java)

    ).run {

    // 세터 함수를 이용해 조건을 명시한다.
    setRequiredNetworkType(JobInfo.NETWORK_TYPE_UNMETERED)

    // JobSechduler의 schedule() 함수로 시스템에 등록한다.
    jobScheduler?.schedule(build())
}
```

<h3>4-2-3. 잡 스케줄러 - 잡 서비스 등록 시 데이터 전달</h3>
<ul>
  <li>
    잡 서비스를 JobInfo 객체를 이용해 시스템에 등록하면 조건이 만족할 때 실행이 되며 이때 <strong>잡 서비스에 데이터를 전달</strong>하려면 JobInfo.Builder의 <strong>setExtras()</strong> 함수를 이용한다.
  </li>
</ul>

```kotlin
// 1. 잡 서비스에 데이터 전달.
var jobScheduler: JobScheduler? = getSystemService<JobScheduler>()
val extras = PersistableBundle()

// 위에서 생성한 PersistableBundle() 객체에 전달할 데이터를 담는다.
extras.putString("extra_data", "hello kkang")
val builder = JobInfo.Builder(1, componentName)
builder.setRequiredNetworkType(JobInfo.NETWORK_TYPE_UNMETERED)
builder.setRequiresCharging(true)

// setExtras() 함수를 호출하며 매개변수로 PersistableBundle 타입의 객체를 지정하였다.
builder.setExtras(extras)
val jobInfo = builder.build()
jobScheduler!!.schedule(jobInfo)
```

```kotlin

  jobParameters: JobParameters): Boolean {
    
    // 전달받은 매개변수의 속성(extras)에서 데이터를 얻는다.
    jobParameters.extras.getString("extra_data")
    (... 생략 ...)

}
```
