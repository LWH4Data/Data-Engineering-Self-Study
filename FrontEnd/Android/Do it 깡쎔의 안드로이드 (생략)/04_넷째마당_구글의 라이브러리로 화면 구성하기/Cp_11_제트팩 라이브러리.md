<h1>1. 제트팩과 androidx 소개</h1>
<ul>
  <li>
    구글에서는 안드로이드 앱을 개발하는 데 필요한 다양한 라이브러리 모음을 <strong>제트팩(Jetpack)</strong>이라는 이름으로 제공한다.
  </li>
    <ul>
      <li>
        제트팩이란 안드로이드 플랫폼이 기본으로 제공하는 <strong>플랫폼 API외에 따로 추가된 라이브러리</strong>이다.
      </li>
    </ul>
</ul>

<br>

<h2>1-1. 플랫폼 API</h2>
<ul>
  <li>
    플랫폼 API란 ART(Anroid runtime)에서 제공하는 안드로이 앱의 <strong>핵심 라이브러리</strong>이다.
  </li>
    <ul>
      <li>
        ART는 대부분 <strong>android</strong>나 <strong>java</strong>로 시작하는 패키지명을 사용한다.
      </li>
      <li>
        예를 들어 지금껏 도서에서 살펴본 java.lang.String.util.Date나 android.app.Activity 등이 해당한다.
      </li>
    </ul>
  <li>
    안드로이드 앱은 런타임 때 ART에 의해 실행되기 때문에 ART에서 제공하는 플랫폼 API를 이용해 개발지만 <strong>호환성 문제</strong>가 발생할 수 있으며 실제 앱을 개발할 때에는 더 다양한 기능이나 화면을 구현해야할 필요가 있다.
  </li>
    <ul>
      <li>
        구글에서는 위와같은 문제를 해결하기 위해 2018년 <strong>제트팩</strong>이라는 라이브러리 모음을 발표하였다.
      </li>
    </ul>
</ul>

<br>

<h2>1-2. 제트팩</h2>
<ul>
  <li>
    제트팩은 구글에서 안드로이드 앱 개발용으로 제공하는 다양한 라이브러리 모음으로 <strong>androidx</strong>로 시작하는 패키지명을 사용한다. 크게 세 가지 목적으로 제공한다.
  </li>
    <ul>
      <li>
        앱을 개발하는 데 필요한 <strong>권장 아키텍처</strong>를 제공한다.
      </li>
        <ul>
          <li>
            뷰 모델(ViewModel), 라이브 데이터(LiveData), 룸(Room), 페이징(Paging) 등과 같은 라이브러리가 해당한다.
          </li>
        </ul>
      <li>
        API 레벨의 <strong>호환성 문제를 해결</strong>한다.
      </li>
        <ul>
          <li>
            이전에는 android 버전을 신경써야 했지만 같은 클래스를 androidx(제트팩)으로 사용하면 호환성 문제를 신경쓰지 않아도 된다.
          </li>
        </ul>
      <li>
        플랫폼 API에서 제공하지 않는 <strong>다양한 기능을 제공</strong>한다.
      </li>
        <ul>
          <li>
            대표적으로 컴포즈가 해당한다.
          </li>
        </ul>
    </ul>
</ul>

<br>

<h2>1-3. androidx 라이브러리</h2>
<ul>
  <li>

        <strong>androidx.drawerlayout</strong>: 옆에서 <strong>서랍처럼 열리는 화면</strong>을 구성한다.

<h1>2. appcompat 라이브러리 - API 호환성 해결</h1>
<ul>
  <li>
    appcompat 라이브러리는 안드로이드 <strong>앱의 화면을 구성하는 액티비티</strong>를 만들며 <strong>API 레벨의 호환성 문제</strong>를 해결한다.
  </li>
  <li>
    appcompat 라이브러리를 사용하기 위해서는 그래들 파일의 dependencies 항목(의존성 설정)에 <strong>"implementation(libs.androidx.appcompat)"</strong>을 선언해야 하는데 안드로이드 스튜디오에서 모듈을 만들 때 <strong>자동으로 추가</strong>된다.
  </li>
  <li>
    appcompat 라이브러리를 이용해서 액티비티를 만들 때는 플랫폼 API의 Activity가 아니라 <strong>AppCompatActivity</strong> 클래스를 상속받아 작성한다.
  </li>
</ul>

```kotlin
// 1. appcompat 라이브러리 사용.
import androidx.appcompat.app.AppCompatActivity

(... 생략 ...)

class MainActivity : AppCompatActivity() {
}
```

<h2>2-1. 툴바</h2>
<h3>2-1-1. 툴바와 액션바</h3>
<ul>
  <li>
    <strong>툴바(ToolBar)</strong>는 <strong>화면 위쪽에 타이틀 문자열</strong>이 출력되는 영역을 꾸미기 위한 뷰이다.
  </li>
  <li>
    타이틀 문자열이 나오는 부분을 구성하는 방법은 액티비티 구성 요소인 <strong>액션바</strong>를 이용하는 방법과 <strong>툴바</strong>를 이용하는 방법 두 가지가 있다.
  </li>
  <li>
    <strong>액티비티</strong>가 출력되는 전체 영역을 <strong>창(Window)</strong>라 하고, <strong>setContentView() 함수</strong>가 출력하는 내용이 나오는 부분을 <strong>콘텐츠 영역</strong>이라 한다.
  </li>
  <li>
    액션바는 안드로이드 초창기때부터 제공했던 방식이고 툴바는 안드로이드 21 버전부터 제공하는 방식이다. 현재는 대부분 <strong>툴바</strong>를 사용하고 있다.
  </li>
  <li>
    툴바를 이용할 것인지 액션바를 이용할 것인지에 대한 부분은 <strong>액티비티에 적용되는 테마 설정</strong>에 의해 셜정된다.
  </li>
    <ul>
      <li>
        테마는 <strong>XML</strong>로 설정하며 직접 설정하지 않아도 앱을 만들면 <strong>res/values 폴더</strong>에 테마를 위한 XML 파일이 있고 <strong>기본 테마가 선언</strong>되게 된다.
      </li>
    </ul>
</ul>

```xml
<!-- 1. res/values/themes/themes.xml 기본 구성. -->
<resources xmlns:tools="http://schemas.android.com/tools">
    <!-- 기본 테마 (NoActionBar로 액션바를 사용하지 않음). -->
    <style name="Base.Theme.AndroidLab" parent="Theme.Material3.DayNight.NoActionBar">
        <!-- 여기에 light 테마를 지정. -->
        <!-- <item name="colorPrimary">@color/my_light_primary</item> -->
    </style>
    
    <!-- 위의 style을 상속받아 선언 -->
    <style name="Theme.AndroidLab" parent="Base.Theme.AndroidLab" />
</resources>
```

```xml
<!-- 2. manifests/AndroidManifest.xml에 1번의 theme이 적용됨. -->
<!--   - 가장 하단의 android:theme을 확인. -->
<application
    android:allowBackup="true"
    android:icon="@mipmap/ic_launcher"
    android:lable="@string/app_name"
    android:roundIcon="@mipmap/ic_launcher_round"
    android:supportsRtl="true"
    android:theme="@string/Theme.AndroidLab">
    (... 생략 ...)
</application>
```

<h3>2-1-4. 업 버튼 설정</h3>
<ul>
  <li>
    업(up) 버튼은 액티비티 화면이 앱의 <strong>첫 화면</strong>이 아닐 때 <strong>이전 화면(←)</strong>으로 되돌아가는 기능을 의미한다.
  </li>
    <ul>
      <li>
        안드로이드폰은 뒤로가기 버튼이 있지만 버튼 외에도 뒤로갈 수 있는 옵션이 주어진다는 장점이 있다.
      </li>
    </ul>
  <li>
    업 버튼은 되돌아 가야 하는 <strong>이전 화면의 액티비티</strong>를 등록해야 하며 <strong>매니페스트 파일</strong>에 등록하는 방법과 <strong>액티비티 코드</strong>로 등록하는 방법 두 가지가 있다.
  </li>
  <li>
    업 버튼을 눌렀을 때 바로 이전 화면으로 돌아가는 것이 아니라 <strong>특별한 로직을 실행</strong>하거나 <strong>매니페스트 설정 없이</strong> 코드에서 이전화면으로 돌아가고 싶을 때에는 <strong>onSupportNavigateUp() 함수</strong>를 재정의한다.
  </li>
    <ul>
      <li>
        이렇게 하면 사용자가 업 버튼을 누를 때 onSupportNavigateUp() 함수가 자동으로 호출된다.
      </li>
    </ul>
</ul>

```xml
<!-- 1. 매니페스트 파일에서 업 버튼 설정. -->
<!--   - parentActivityName 속성을 등록하고 툴바에 업 버튼이 나오게 지정한다. -->
<activity
    android:name=".test1.TwoActivity"
    android:exported="false"
    android:parentActivityName=".MainActivity"/>
```

```kotlin
// 2. 1번에서 등록한 속성에 더해 출 바에 업 버튼이 나오게 지정한다.
supportActionBar?.setDisplayHomeAsUpEnabled(true)
```

```kotlin
// 3. 업 버튼 클릭 시 자동으로 호출되는 함수 재정의.
override fun onSupportNavigateUp(): Boolean {
    Log.d("kkang", "onSupportNavigateUp")
    return super.onSupportNaviageUp()
}
```

<h2>2-2. 메뉴 구성</h2>
<ul>
  <li>
    메뉴는 툴바의 중요한 구성 요소이며 <strong>액티비티 화면에서 사용자 이벤트</strong>를 사용할 수 있도록 한다.
  </li>
  <li>
    액티비티에 메뉴를 추가하면 <strong>툴바 오른쪽에 오버플로 버튼</strong>이 나타난다.
  </li>
    <ul>
      <li>
        사용자가 오버플로 버튼을 누르면 메뉴가 <strong>아래로 확장</strong>되어 나타난다.
      </li>
      <li>
        오버플로 메뉴 중 몇몇은 <strong>툴바에 아이콘</strong>으로 나오게 할 수 있다.
      </li>
        <ul>
          <li>
            이런 방식으로 툴바에 노출된 아이콘을 <strong>액션 아이템(혹은 액션 버튼)</strong>이라 한다.
          </li>
        </ul>
      <li>
        액션 아이템과 오버플로 메뉴 모두 메뉴이다.
      </li>
    </ul>
  <li>
    액티비티에 메뉴를 추가하고 싶다면 <strong>onCreateOptionsMenu()</strong>와 <strong>onPrepareOptionsMenu()</strong> 함수를 이용한다. 두 함수는 <strong>액티비티의 메뉴</strong>를 구성할 때 자동으로 호출되는 것은 동일하나 <strong>호출되는 시점</strong>이 다른다.
  </li>
    <ul>
      <li>
        <strong>onCreateOptionsMenu()</strong> 함수는 액티비티가 실행되면서 <strong>처음에 한 번</strong>만 호출된다.
      </li>
        <ul>
          <li>
            <strong>정적인 메뉴</strong>를 구성할 때 사용한다.
          </li>
        </ul>
      <li>
        <strong>onPrepareOptionsMenu()</strong> 함수는 <strong>액티비티가 실행</strong>되면서 한 번 호출된 후 <strong>오버플로 메뉴</strong>가 나타날 때마다 반복해서 호출된다.
      </li>
        <ul>
          <li>
            메뉴가 화면에 나올 때마다 <strong>동적으로 구성</strong>하고 싶은 경우에 선택한다.
          </li>
        </ul>
    </ul>
  <li>
    액티비티의 메뉴는 주로 <strong>사용자 이벤트를 처리</strong>하려고 사용하는 것이기 때문에 대부분 <strong>액티비티 내에 정적</strong>으로 제공하면 되며 따라서 <strong>onCreateOptionsMenu()</strong> 함수를 사용한다.
  </li>
  <li>
    생성한 메뉴를 사용자가 선택했을 때의 이벤트 처리는 <strong>onOptionsItemSelected()</strong> 함수를 이용한다.
  </li>
    <ul>
      <li>
        함수의 매개변수는 이벤트가 발생한 메뉴 객체인 <strong>MenuItem</strong>이다. 
      </li>
        <ul>
          <li>
            MenuItem의 <strong>itemId 속성</strong>으로 이벤트가 발생한 메뉴 객체의 <strong>식별값</strong>을 얻어서 이벤트를 처리한다.
          </li>
        </ul>
    </ul>
</ul>

```kotlin
// 1. 메뉴 구성 함수.
// 함수의 매개변수인 Menu 객체를 메뉴바로 생각하면 된다.
override fun onCreateOptionsMenu(menu: Menu?): Boolean {
    
    // Menu 객체에 메뉴를 추가할 때 add() 함수를 이용한다.
    //   - fun add(groupId: Int, itemId: Int, order: Int, title: CharSequence!): MenuItem
    //     - add() 함수의 두 번째 매개변수는 매뉴의 식별자이다.
    //       - 메뉴는 이벤트 처리가 목적이기에 어떤 메뉴를 클릭했는지 식별할 때 사용한다.
    //     - add() 함수의 네 번째 매개변수는 메뉴의 문자열이다.
    //   - add() 함수의 반환값은 MenuItem 객체이며 해당 객첵라 메뉴 하나를 의미한다.
    val menuItem1: MenuItem? = menu?.add(0, 0, 0, "menu1")
    val menuItem2: MenuItem? = menu?.add(0, 1, 0, "menu2")
    return super.onCreateOptionsMenu(menu)
}
```

```kotlin
// 2. 메뉴 선택 시 이벤트 처리.
// when 문은 단일 표현식 함수를 제공하는 조건문이다.
//   - () 안의 조건식이 참이면 중괄호 안의 구문이 실행되며 결과를 반환한다.
//   - 함수의 매개변숫값에 따라 특정 구문을 실행하고 결과를 반환하는 함수라면 단일 표현식 함수로 작성한는 것이 편리하다.
override fun onOptionsItemSelected(item: MenuItem): Boolean = when (item.itemId) {
    0 -> {
        Log.d("kkang", "menu1 click")
        true
    }
    1-> {
        Log.d("kkang", "menu2 click")
        true
    }
    else -> super.onOptionsItemSelected(item)
}
```

<h3>2-2-1. 리소스로 메뉴 구현</h3>
<ul>
  <li>
    onCreateOptionsMenu() 함수에서 Menu 객체의 add() 함수로 메뉴를 구성할 수도 있지만, 액티비티의 메뉴는 대부부 <strong>정적</strong>으로 제공되기 때문에 코드가 아니라 <strong>리소스 XML 파일</strong>로 구성한다.
  </li>
    <ul>
      <li>
        XML 파일은 <strong>res/menu 디렉터리</strong>에 만든다.
      </li>
    </ul>
  <li>
    메뉴는 기본적으로 <strong>오버플로 메뉴</strong>로 출력된다. 
  </li>
  <li>
    <strong>툴바에 아이콘</strong>으로 메뉴가 나타나게 하려면 <strong>showAsAction 속성</strong>을 이용한다. showAsAction 속성값에는 다음 세 가지 가운데 하나를 지정한다.
  </li>
    <ul>
      <li>
        <strong>never(기본)</strong>: 항상 <strong>오버플로 메뉴<strong>로 출력한다.
      </li>
      <li>
        <strong>ifRoom</strong>: 만약 <strong>툴바에 공간</strong>이 있다면 <strong>액션 아이템</strong>으로, 없다면 <strong>오버플로 메뉴</strong>로 출력한다.
      </li>
      <li>
        <strong>always</strong>: 항상 <strong>액션 아이템</strong>으로 출력한다.
      </li>
    </ul>
</ul>

```xml

    <!--   - title과 icon 속성은 메뉴 문자열과 아이콘을 지정한다. -->

        android:id="@+id/menu1"

    <item

        andorid:icon="@andorid:drawable/ic_menu_search"
        android:title="menu2"
```

<h3>2-2-2. 액션 뷰 이용</h3>
<ul>
  <li>
    <strong>액션 뷰(ActionView)</strong>는 <strong>액션바</strong>에서 특별한 기능을 제공하며 대표적으로 <strong>androidx.appcompat.widget.SearchView</strong>가 있다.
  </li>
    <ul>
      <li>
        SearchView는 액션바에서 <strong>검색 기능</strong>을 제공한다.
      </li>
    </ul>
  <li>
    액션 뷰를 메뉴에 적용할 때에는 <strong>actionViewClass 속성</strong>을 이용한다. 해당 속성에 이용할 뷰 클래스를 등록하면 된다.
  </li>
  <li>
    버튼으로 사용자 이벤트를 처리할 수도 있지만 대부분 스마트폰의 <strong>화면이 작기 때문</strong>에 대부분 <strong>메뉴</strong>로 처리한다.
  </li>
    <ul>
      <li>
        메뉴는 툴바에서 제공하기 때문에 화면의 <strong>메인 콘텐츠와 분리</strong>할 수 있기 때문이다.
      </li>
    </ul>
</ul>

<h1>3. 프래그먼트 - 액티비티처럼 동작하는 뷰</h1>
<ul>
  <li>
    androidx에서 제공하는 라이브러리 가운데 <strong>프래그먼트</strong>와 <strong>뷰 페이저2</strong>는 중요도와 사용 빈도가 높다.
  </li>
    <ul>
      <li>
        뷰 페이저2는 플랫폼 API에서 제공하지 않는 라이브러리이다.
      </li>
    </ul>
  <li>
    프래그먼트는 플랫폼 API에서도 android.app.Fragment로 제공하지만 대부분 <strong>androidx.fragment 라이브러리</strong>를 이용해 구현한다.
  </li>
</ul>

<br>

<h2>3-1. 프래그먼트 소개</h2>
<ul>
  <li>
    프래그먼트는 텍스트 뷰나 버튼 처럼 액티비티 화면을 구성하는 뷰이지만 이 자체만으로는 <strong>화면에 출력되지 않는다</strong>.
  </li>
  <li>
    프로그먼트와 다른 뷰의 차이점은 프래그먼트는 <strong>액티비티처럼 동작</strong>한다는 점이다.
  </li>
    <ul>
      <li>
        다시 말하면 액티비티에 작성할 수 있는 모든 코드는 프래그먼트에도 사용할 수 있다.
      </li>
    </ul>
  <li>
    프래그먼트는 태블릿처럼 <strong>화면이 넓은 기기</strong>에서 동작하는 앱을 개발할 수 있도록 제공되었다.
  </li>
    <ul>
      <li>
        한 화면은 하나의 액티비티 클래스에서 작성하는데 화면이 크면 당연히 액티비티 클래스에 많은 코드를 작성해야 한다.
      </li>
      <li>
        두 개의 창이 필요한 경우 뷰는 둘 모두 출력하기에 문제가 발생하지만 프래그먼트는 각각을 작성하고 보여줄 수 있다. 즉, 하나의 액티비티 클래스를 길게 작성할 필요 없이 <strong>여러 프래그먼트</strong>로 작성하고 활용한다.
      </li>
    </ul>
  <li>
    일반화해서 이야기하면 왼쪽과 오른쪽에 각각 다른 클래스로 나누어 개발을 하지만 <strong>액티비티는 하나</strong>이고 <strong>프래그먼트 클래스를 두 개</strong>를 만드는 구조가 된다.
  </li>
  <li>
    프래그먼트로 구현하는 대표적인 예로는 <strong>탭</strong>과 <strong>뷰</strong>가 있다. 탭과 뷰 페이저 모두 <strong>여러 화면을 제공</strong>하며 모두 하나의 클래스로 구현하기에는 부담스럽기에 탭과 뷰 페이저가 제공하는 한 화면을 <strong>하나의 프래그먼트</strong>로 개발한다.
  </li>
    <ul>
      <li>
        탭 버튼이 4개 있는 화면을 만들 때에는 화면도 4개가 필요하다. 이때 각 화면을 프래그먼트로 개발하고 액티비티는 하나의 탭 버튼에서 프래그먼트만 교체해서 출력하는 방식으로 개발한다.
      </li>
    </ul>
</ul>

<br>

<h2>3-2. 프래그먼트 구현</h2>
<ul>
  <li>
    프래그먼트는 뷰이지만 이 자체로는 화면에 아무것도 출력하기 않기 때문에 프래그먼트 <strong>화면을 구성하는 XML 파일</strong>을 작성해야 한다. 액티비티를 만들 때 레이아웃 XML 파일을 작성했던 것과 차이가 없다.
  </li>
  <li>
    프래그먼트는 <strong>Fragment</strong>를 상속받아 작성하는 클래스이다.
  </li>
    <ul>
      <li>
        프래그먼트 클래스에 최소한으로 작성해야 하는 함수는 <strong>onCreateView()</strong>이다.
      </li>
      <li>
        onCreateView() 함수가 자동 호출되며 반환한 View 객체가 화면에 출력된다.
      </li>
    </ul>
</ul>

```kotlin
// 1. 프래그먼트 구현.
(... 생략 ...)
import androidx.fragment.app.Fragment

class OneFragment : Fragment() {
    lateinit var binding: FragmentOneBinding
    override fun onCreateView(
        inflater: LayoutInflater, 
        container: ViewGroup?, 
        savedInstanceState: Bundle?
    ): View? {
        binding = FragmentOneBinding.inflate(inflater, container, false)
        return binding.root
    }
}
```

<h3>3-2-1. 액티비티에서 프래그먼트 출력</h3>
<ul>
  <li>
    프래그먼트를 출력해서 보여주기 위해서는 <strong>FragmentContainerView</strong>가 제공된다.
  </li>
  <li>
    FragmentContainerView를 <strong>액티비티 레이아웃 XML에 등록</strong>하고 <strong>name 속성</strong>에 출력하고자 하는 프래그먼트 클래스를 지정하면 지정된 프래그먼트가 화면에 출력된다.
  </li>
    <ul>
      <li>
        때로는 XML이 아니라 <strong>코드</strong>에서 직접 프래그먼트 객체를 생성해서 화면에 출력해야할 때도 있는데 이때에도 FragmentContainerView를 이용한다.
      </li>
        <ul>
          <li>
            FragmentContainerView의 name 속성에 프래그먼트를 지정하지 않고 <strong>코드에서 동적으로 </strong>지정해준다.
          </li>
        </ul>
    </ul>
  <li>
    동적으로 프래그먼트를 추가 혹은 제거 하려면 FragmentManager로 만든 <strong>FragmentTransaction 클래스의 함수</strong>를 이용한다.
  </li>
  <li>
    코드에서 동적으로 프래그먼트를 제어하는 함수에는 다음과 같은 함수들이 있다.
  </li>
    <ul>
      <li>
        add(int containerViewId, Fragment fragment): 새로운 프래그먼트를 <strong>화면에 추가</strong>한다.
      </li>
      <li>
        replace(int containerViewId, Fragment fragment): 추가된 프래그먼트를 <strong>대체</strong>한다.
      </li>
      <li>
        remove(Fragment fragment): 추가된 프래그먼트를 <strong>제거</strong>한다.
      </li>
      <li>
        commit(): 화면에 <strong>적용</strong>한다.
      </li>
    </ul>
</ul>

```xml

    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:id="@+id/fragment_content"
    android:name="com.example.test_11.OneFragment">
  
</androidx.fragment.app.FragmentContainerView>
```


<h3>3-2-2. 프래그먼트 생명주기</h3>
<ul>
  <li>
    프래그먼트는 액티비티처럼 동작하는 뷰이고 따라서 액티비티에서 작성하는 코드는 모두 프래그먼트에도 작성할 수 있다. 결과적으로는 프래그먼트도 <strong>액티비티와 생명주기(life cycle)가 같다</strong>.
  </li>
  <li>
    프래그먼트도 액티비티의 생명주기 함수인 onCreate(), onStart(), onResume(), onPause(), onStop(), onDestroy()를 대로 가지고 있으며 호출되는 시점 또한 액티비티와 동일하다.
  </li>
  <li>
    프래그먼트의 생명주기는 크게 초기화(initialized), 생성(created), 시작(started), 재개(resumed), 소멸(destroyed) 다섯 단계로 구분되며 각 단계가 진행되며 자동으로 호출되는 함수들이 있다. (p339 참고).
  </li>
    <ul>
      <li>
        <strong>초기화 단계</strong>는 프래그먼트의 화면을 구성할 <strong>뷰가 준비되지 않은 상태</strong>로 onAttach와 onCreate 함수는 뷰와 관련이 없는 <strong>프래그먼트의 초기화 로직</strong>을 수행한다.
      </li>
      <li>
        <strong>생성 단계</strong>는 프래그먼트의 <strong>화면을 구성할 뷰를 준비</strong>한다. 대부분은 <strong>onCreateView 함수</strong>에서 프래그먼트의 화면을 구성할 뷰 객체를 준비한다.
      </li>
      <li>
        <strong>시작 단계</strong>는 프래그먼트 <strong>화면</strong>이 사용자에게 보여진다.
      </li>
      <li>
        <strong>재개 단계</strong>에서는 포커스를 가지고 <strong>사용자의 이벤트</strong>를 처리할 수 있다.
      </li>
    </ul>
  <li>
    어떤 프래그먼트가 처음 화면에 나오면 <strong>onAttach → onCreate → onCreateView → OnViewCreated → onStart → onResume</strong> 함수가 순대로 호출된다. 이후 다른 프래그먼트로 교체될 때는 <strong>백 스택</strong>을 사용하는지에 따라 생명주기가 다르게 동작한다.
  </li>
    <ul>
      <li>
        백 스택(back stack)이란 프래그먼트가 화면에 보이지 않는 순간 <strong>제거하지 않고 저장</strong>했다가 <strong>다시 이용할 수 있는 기능</strong>을 의미한다.
      </li>
      <li>
        백 스택을 사용하려면 FragmentTransaction의 <strong>addToBackStack()</strong> 함수를 이용해야 한다.
      </li>
      <li>
        백 스택을 사용하지 않으면 프래그먼트로 화면 전환이 불가하며 프래그먼트가 교체될 때 기존의 프래그먼트는 <strong>onDestroy</strong>까지 호출되어 <strong>제거</strong>된다.
      </li>
      <li>
        백 스택을 사용하면 프래그먼트가 제거되지 않고 <strong>onDestroyView</strong> 함수까지만 호출된다. 즉, <strong>onPause → onStop → onDestroyView</strong> 함수가 자동으로 호출된다.
      </li>
        <ul>
          <li>
            다시 준비되어 화면에 출력될 경우 <strong>onCreateView → onViewCreated → onStart → onResume</strong> 함수가 차례로 호출된다.
          </li>
        </ul>
    </ul>
</ul>

```kotlin
// 1. 백 스택 사용 설정.
transaction.addToBackStack(null)
```

<h1>4. 리사이클러 뷰 - 목록 화면 구성</h1>
<h2>4-1. 리사이클러 뷰 기초 사용법</h2>
<ul>
  <li>
    리사이클러 뷰는 <strong>여러 가지 항목을 나열하는 목록 화면</strong>을 만들 때에 사용한다.
  </li>
  <li>
    리사이클러 뷰를 만들 때 사용하는 <strong>RecyclerView 클래스</strong> 또한 클래스 만으로는 <strong>화면에 아무 것도 출력되지 않기 때문</strong>에 다음과 같은 구성 요소가 필요하다.
  </li>
    <ul>
      <li>
        ViewHoler(필수): 각 항목을 구성하는 뷰 객체를 가진다.
      </li>
      <li>
        Adapter(필수): 뷰 홀더에 있는 뷰 객체에 적절한 데이터를 대입하여 항목을 완성한다.
      </li>
      <li>
        LayoutManager(필수): 어댑터가 만든 항목들을 어떻게 배치할지 결정하여 리사이클러 뷰에 출력한다.
      </li>
      <li>
        ItemDecoration(옵션): 항목을 꾸민다.
      </li>
    </ul>
  <li>
    API 레벨 1 버전부터 제공된 리스트 뷰로 목록을 구성할 수도 있지만 androidx의 <strong>리사이클러 뷰</strong>가 리스트 뷰로 만들기 어렵거나 불가능한 목록 화면을 만들 수 있어 거의 모든 상황에 리사이클러 뷰를 사용한다.
  </li>
</ul>

```xml
<!-- 1. 리사이클러 뷰를 레이아웃 XML 파일에 등록한다. -->
<androidx.recycleview.widget.RecyclerView
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:id="@+id/recyclelerView"
    android:layout_width="match_parent"
    android:layout_height="match_parent" />
```

```xml
<!-- 2. 목록에 표시할 항목을 디자인한 레이아웃 XML 파일 item_main.xml 파일을 작성한다. -->
<LinearLayout
    android:id="@+id/item_root"
    xmlns:android="http://schemas.android.com/apk/android"
    android:layout_width="wrap_content"
    android:layout_height="wrap_content"

        android:layout_height="wrap_content"

</LinearLayout>
```

<h3>4-1-1. 뷰 홀더 준비</h3>
<ul>
  <li>
    각 항목에 해당한느 뷰 객체를 가지는 뷰 홀더는 <strong>RecycleView.ViewHolder</strong>를 상속받아 작성한다.
  </li>
  <li>
    원래는 뷰 홀더에 항목들의 뷰 객체를 선언하고 findViewById로 가져와야 하지만 <strong>뷰 바인딩</strong>을 활용하면 대상이되는 레이아웃 XML 파일에 해당하는 <strong>바인딩 객체</strong>만으로 항목들을 가져올 수 있다.
  </li>
</ul>

```kotlin
// 1. 뷰 홀더 준비.
class MyViewHolder(val binding: ItemMainBinding): RecyclerView.ViewHolder(binding.root)
```

<h3>4-1-2. 어댑터 준비</h3>
<ul>
  <li>
    어댑터는 뷰 홀더의 뷰에 데이터를 출력해 각 항목을 만들어 주는 역할을 하며 리사이클러 뷰를 위한 어댑터는 <strong>RecyclerView.Adapter</strong>를 상속받아 작성한다.
  </li>
  <li>
    생성한 어댑터에서 재정의 해야하는 함수는 다음과 같다.
  </li>
    <ul>
      <li>
        getItemCount(): <strong>항목 개수를 판단</strong>하려고 자동으로 호출된다.
      </li>
      <li>
        onCreateViewHolder(): 항목의 뷰를 가지는 <strong>뷰 홀더를 준비</strong>하려고 자동으로 호출된다.
      </li>
        <ul>
          <li>
            항목을 구성할 때 이용할 뷰 홀더 객체를 준비한다.
          </li>
          <li>
            onCreateViewHolder() 함수에서 반환한 뷰 홀더 객체는 자동으로 onBindViewHolder() 함수의 매개변수로 전달된다.
          </li>
        </ul>
      <li>
        onBindViewHolder(): 뷰 홀더의 뷰에 <strong>데이터를 출력</strong>하려고 자동으로 호출된다. 
      </li>
        <ul>
          <li>
            getItemCount() 함수에서 반환된 항목의 개수만큼 호출된다.
          </li>
          <li>
            매개변수로 전달된 뷰 홀더 객체의 뷰에 <strong>데이터를 출력</strong>하거나 <strong>필요한 이벤트를 등록</strong>한다.
          </li>
        </ul>
    </ul>
</ul>

<h3>4-2-3. 리사이클러 뷰 출력</h3>
<ul>

    어댑터 준비가 완료되었다면 리사이클러 뷰에 어댑터와 레이아웃 매니저를 등록해 화면에 출력한다.
  </li>
</ul>

```kotlin
// 1. 리사이클러 뷰 출력.
class RecyclerVieActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val binding = ActivityRecyclerViewBinding.inflate(layoutInflater)
        setContentView(binding.root)
        val datas = mutableListOf<String>()
        for(i in 1..10){
            datas.add("Item $i")
        }
        binding.recyclerView.layoutManager = LinearLayoutManager(this)
        binding.recyclerView.adapter = MyAdapter(datas)
        binding.recyclerView.addItemDecoration(DividerItemDecoration(this, LinearLayoutManager.VERTICAL))
    }
}
```

<h2>4-3. 레이아웃 매니저</h2>
<ul>
  <li>
    레이아웃 매니저는 어댑터로 만든 항목을 <strong>리사이클러 뷰에 배치</strong>한다.
  </li>
  <li>
    레이아웃 매니저는 recyclerView.layoutManager를 상속 받은 클래스로 라이브러리에서 다음처럼 제공한다.
  </li>
    <ul>
      <li>
        LinearLayoutManager: 항목을 <strong>가로</strong>나 <strong>세로</strong> 방향으로 배치한다.
      </li>
      <li>
        GridLayoutManager: 항목을 <strong>그리드</strong>로 배치한다.
      </li>
      <li>
        StaggeredGridLayoutManager: 항목을 <strong>높이가 불규칙한 그리드</strong>로 배치한다.
      </li>
    </ul>
</ul>

<h3>4-3-1. 항목ㅇ르 가로 혹은 세로 방향으로 배치</h3>
<ul>
  <li>
    항목을 가로나 세로 방향으로 배치하고 싶다면 LinearLayoutManager를 사용한다.
  </li>
</ul>

```kotlin
// 1. 항목을 세로로 배치.
// layoutManager 프로퍼티에 LinearLayoutManager를 등록한 것이다.
//   - 방향은 지정하지 않았기 때문게 기본값인 세로로 설정된다.
//   - 가로로 배치를 희망할 경우 orientation 값을 LinearLayoutManager.HORIZONTAL로 지정한다.
binding.recyclerView.layoutManager = LinearLayoutManager(this)
```

```kotlin
// 2. 항목을 가로로 배치.
val layoutManager = LinearLayoutManager(this)
layoutManager.orientation = LinearLayoutManager.HOTIZONTAL
binding.recyclerView.layoutManager = layoutManager
```

<h3>4-3-2. 그리드로 배치하기</h3>
<ul>
  <li>
    항목을 그리드로 배치할 경우 GridLayoutManager를 이용한다.
  </li>
</ul>

```kotlin
// 1. 항목을 그리드로 배치.
//   - 생성자의 숫자는 그리드에서 열(column)의 개수를 의미한다. 현재는 2이다.
val layoutManager = GridLayoutManager(this, 2)
binding.recyclerView.layoutManager = layoutManager
```

```kotlin
// 2. 그리드에서 항목을 가로로 배치.
val layoutManager = GridLayoutManager(
    this, 
    3,
    // 항목이 가로로 배치되도록 설정한다.
    GridLayoutManager.HORIZONTAL, 
    false)
binding.recyclerView.layoutManager = layoutManager
```

```kotlin
// 3. 그리드에서 항목을 오른쪽부터 배치.
val layoutManager = GridLayoutManage(
    this, 
    3, 
    GridLayoutManager.HORIZONTAL, 
    // true로 설정하면 세로일 때에는 아래부터, 가로일 때에는 우측부터 항목이 채워진다.
    true)
```

<h3>4-3-3. 높이가 불규칙한 그리드로 배치하기</h3>
<ul>
  <li>
    StaggeredGridLayoutManager는 GridLayoutManager처럼 뷰를 그리드 구조로 배치하지만 각 <strong>뷰의 크기</strong>가 다르면 <strong>지그재그 형태</strong>로 배치한다.
  </li>
</ul>

```kotlin
// 1. 지그재그 그리드 형태로 배치.
val layoutManager = StaggeredGridLayoutManager(
    2,
    StaggeredGridLayoutManager.VERTICAL
)
binding.recyclerView.layoutManager = layoutManager
```

<h2>4-4. 아이템 데커레이션</h2>
<ul>
  <li>
    아이템 데커레이션은 리사이클러 뷰를 다양하게 <strong>꾸밀 때 사용</strong>한다.
  </li>
    <ul>
      <li>
        각 항목을 꾸미거나 레이아웃 매니저가 항목을 배치하기 전후로 설정할 수 있다.
      </li>
      <li>
        필수가 아니기 때문에 필요할 때 적용하면 된다.
      </li>
    </ul>
  <li>
    라이브러리에서 제공하는 아이템 데커레이션은 항목의 구분선을 출력해 주는 <strong>DividerItemDecoration</strong> 뿐이다. 
  </li>
    <ul>
      <li>
        따라서 필요에 따라 <strong>ItemDecoration을 상속받는 개발자 클래스</strong>를 만들고 직접 다양한 꾸미기 작업을 수행해야 한다.
      </li>
    </ul>
  <li>
    아이템 데커레이션은 꾸미기 작업을 하는 함수 3개를 제공한다.
  </li>
    <ul>
      <li>
        onDraw(): 항목이 배치되기 <strong>전</strong>에 호출된다.
      </li>
        <ul>
          <li>
            항목이 배치되기 전에 Canvas 객체로 각종 그림을 먼저 그릴 수 있다.
          </li>
        </ul>
      <li>
        onDrawOver(): 항목이 모두 배치된 <strong>후</strong> 호출된다.
      </li>
        <ul>
          <li>
            매개변수로 전달되는 Canvas 객체로 그림을 그리고 항목 위에 그림이 나타난다.
          </li>
        </ul>
      <li>
        getItemOffsets(): 개별 항목을 <strong>꾸밀 때</strong> 호출된다.
      </li>
        <ul>
          <li>
            항목 하나당 한 번씩 호출되어 각 항목을 꾸미는 데사용한다.
          </li>
          <li>
            매개변수로 전달되는 Rect 객체는 각 항목을 화면에 출력할 때 필요한 <strong>사각형 정보</strong>이다.
          </li>
            <ul>
              <li>
                사각형 정보를 이용해 네 방향(left, top, right, bottom)의 여백을 설정할 수 있다.
              </li>
            </ul>
        </ul>
    </ul>
  <li>
    생성한 데커레이션 객체를 리사이클러 뷰에 적용할 때에는 <strong>addItemDecoration()</strong> 함수를 이용한다.
  </li>
</ul>

```kotlin
// 1. 아이템 데커레이션 구현.
class MyDecoration(val context: Context): RecyclerView.ItemDecoration() {
    override fun onDraw(x: Canvas, parent: RecyclerView, state: RecyclerView.State) {
        super.onDraw(c, parent, state)
    }
    override fun onDrawOver(c: Canvas, parent: RecyclerView, state: RecyclerView.State) {
        super.onDrawOver(c, parent, state)
    }
    override fun getItemOffsets(
        outRect: Rect,
        view: View
        parent: RecyclerView,
        state: RecyclerView.State
    ) {
        super.getItemOffsets(outRect, view, parent, state)
    }
}
```

```kotlin
// 2. 항목이 배치되기 전에 호출되는 onDraw() 함수.
override fun onDraw(c: Canvas, parent: RecyclerView, state: RecyclerView.State) {
    super.onDraw(c, parent, state)
    c.drawBitmap(BitmapFactory.decodeResource(context.getResources(), R.drawable.stadium), 0f, 0f, null)
}
```

```kotlin
// 3. 모든 항목이 배치된 후 호출되는 onDrawOver() 함수.
override fun onDrawOver(c: Canvas, parent: RecyclerView, state: RecyclerView.State) {
    super.onDrawOver(c, parent, state)

    // 뷰 크기 계산.
    val width = parent.width
    val height = parent.height

    // 이미지 크기 계산.
    val dr: Drawable? = ResourcesCompat.getDrawable(context.getResources(), R.drawable.kbo, null)
    val drWidth = dr?.intrinsicWidth
    val drHeight = dr?.intrinsicHeight

    // 이미지가 그려질 위치 계산.
    val left = width / 2 - drWidth?.div(2) as Int
    val top = height / 2 - drHeight?.div(2) as Int
    c.drawBitmap(
        BitmapFactory.decodeResource(context.getResources(), R.drawable.kbo),
        left.toFloat(),
        top.toFloat(),
        null
    )
}
```

```kotlin
// 4. 개별 항목을 꾸미는 getItemOffsets() 함수.
override fun getItemOffsets(
    outRect: Rect,
    view: View,
    parent: RecyclerView,
    state: RecyclerView.State
) {
    super.getItemOffsets(outRect, view, parent, state)
    val index = parent.getChildAdapterPosition(view) + 1
    if (index % 3 == 0)
        outRect.set(10, 10, 10, 60)  // left, top, right, bottom
    else
        outRect.set(10, 10, 10, 0)
    
    view.setBackgroundColor(Color.LTGRAY)
    ViewCompat.setElevation(view, 20.0f)
}
```

```kotlin
// 5. 리사이클러 뷰에 아이템 데커레이션 적용.
binding.recyclerView.addItemDecoration(MyDecoration(this))
```

<h1>5. 뷰 페이저2 - 스와프로 넘기는 화면 구성</h1>
<ul>
  <li>
    뷰 페이저(ViewPager)는 <strong>스와이프 이벤트</strong>로 화면을 전환할 때 사용한다.
  </li>
  <li>
    뷰 페이저는 플랫폼 API에서 제공하지 않기 때문에 <strong>androidx 라이브러리</strong>를 이용해 개발하며 기존의 viewpager와 별개로 2019년 부터 제공된 <strong>viewpager2 라이브러리</strong>를 사용한다. 기능이 더 많기 때문이다.
  </li>
  <li>
    뷰 페이저2는 화면을 <strong>항목</strong>으로 본다. 각 항목은 순서대로 나열되어 있으며 단지 <strong>한 화면에서 항목 하나</strong>가 나온다는 개념이다.
  </li>
    <ul>
      <li>
        각 페이지에 어떤 항목이 보여져야할지를 관리하기 위해 리사이클러 뷰와 동일하게 <strong>어댑터</strong>를 사용해야한다.
      </li>
        <ul>
          <li>
            뷰 페이저2에서 사용할 수 있는 어댑터는 리사이클러 뷰에서 본 <strong>RecyclerView.Adapter</strong>를 사용하거나 <strong>FragmentStateAdapter</strong>를 사용할 수 있다.
          </li>
        </ul>
    </ul>
</ul>

```xml
<!-- 1. res/layout의 레이아웃 XML에 뷰페이저2 라이브러리 추가. -->
<androidx.viewpager2.widget.ViewPager2
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:id="@+id/viewpager"
    android:layout_width="match_parent"
    android:layout_height="match_parent" />
```

<h3>5-1-1. 리사이클러 뷰 어댑터 이용</h3>

    리사이클러 뷰에서 사용하던 RecyclerView.Adapter와 차이가 없으며 동일하게 작성하되 뷰 <strong>페이저2의 어댑터</strong>로 적용만 하면된다.

```kotlin
// 1. 뷰 페이저 2의 구현 - RecyclerView.Apater 이용.
class MyPagerViewHolder(val binding: ItemPagerBinding) :
    RecyclerView.ViewHolder(binding.root)

class MyPagerView.Adapter(val datas: MutableList<String>) :
    RecyclerView.Adapter<RecyclerView.ViewHolder>() {
    override fun getItemCount(): Int {
        return datas.size
    }

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int):
        RecyclerView.ViewHolder =
            MyPagerViewHolder(ItemPagerBinding.inflate(LayoutInflater.from(
                parent.context), parent, false))
    
    override fun onBindViewHolder(holder: RecyclerView.ViewHolder, position: Int) {
        val binding = (holder as MyPagerViewHolder).binding

    // 뷰에 데이터 출력.
    binding.itemPagerTextView.text = datas[position]
        when (position % 3) {
            0 -> binding.itemPagerTextView.setBackgroundColor(Colort.RED)
            1 -> binding.itemPagerTextView.setBackgroundColor(Color.BLUE)
            2 -> binding.itemPagerTextView.steBackgroundColort(Color.GREEN)
        }
    }
}
```

```kotlin
class MyFragmentPagerAdapter(activity: FragmentActivity): FragmentStateAdapter(activity)
```

<h1>6. 드로어 레이아웃 - 옆에서 열리는 화면 구성</h1>
<ul>
  <li>
    <strong>드로어 레이아웃(DrawerLayout)</strong>은 액티비티 화면에 보이지 않던 내용이 <strong>왼쪽이나 오른쪽</strong>에서 손가락의 움직임에 따라 밀려 나오는 기능을 의미한다.
  </li>
    <ul>
      <li>
        androidx의 레이아웃이며 서랍(drawer)처럼 열리는 메뉴를 구성할 때 사용한다.
      </li>
    </ul>
  <li>
    드로어 레이아웃을 액티비티에 적용하려면 액티비티의 <strong>레이아웃 XML 파일 구성</strong>이 중요하다.
  </li>
    <ul>
      <li>
        레이아웃 XML 파일에서 드로어 메뉴가 출력되어야 하는 부분의 태그를 <strong>DrawerLayout</strong>으로 선언한다.
      </li>
    </ul>
  <li>
    액티비티 레이아웃 XML 파일의 루트 태그가 DrawerLayout일 필요는 없지만, 대부분은 액티비티의 기본 화면이 보이고 그 위에 덮듯이 출력이 되도록 하기 때문에 <strong>루트 태그를 DrawerLayout</strong>으로 선언한다.
  </li>
  <li>
    DrawerLayout 아래에는 <strong>뷰를 두 개 선언</strong>한다.
  </li>
    <ul>
      <li>
        첫 번째 하위 태그 부분을 액티비이 화면이 출력한다.
      </li>
      <li>
        두 번째 하위 태그 부분은 안 보이다가 끌려나온다.
      </li>
    </ul>
  <li>
    레이아웃 XML만으로 드로어 레이아웃을 이용할 수 있지만 대부분 툴바 영역에 <strong>토글(toggle) 버튼</strong>을 함께 제공한다.
  </li>
    <ul>
      <li>
        드로어 메뉴 토글 버튼은 <strong>ActionBarDrawerToggle 클래스</strong>에서 제공한다.
      </li>
    </ul>
</ul>

```xml
<!-- 1. 드로어 레이아웃 구성. -->
<!-- 최상단에 drawerlayout 태그를 적용한다. -->
<adroidx.drawerlayout.widget.DrawerLayout
    xlmns:android="http://schemans.android.com/apk/res/android"
    android:id="@+id/drawer"
    android:layout_width="match_parent"
    android:layout_height="match_parent">
    
    <!-- 첫 번째로 작성한 태그가 베이스로 출력된다. -->
    <LinearLayout ... 생략 ...>
        (... 생략 ...)
    </LinearLayout>

    <!-- 두 번째로 작성한 태그는 사용자가 화면에 상호작용을 하면 drawer로 출력된다. -->
    <!--   - android:layout_gravity 속성값을 통해 화면에서 나오는 방향을 지정할 수도 있다. -->
    <!--     start는 적용되는 언어의 방향을 따른다. -->
    <TextView
        android:layout_gravity="start" />
</androidx.drawerlayout.widget.DrawerLayout>
```

```kotlin
// 2. 드로어 메뉴 토글 버튼 적용.
class DrawerActivity : AppCompatActivity() {
    lateinit var toggle: ActionBarDrawerToggle
    override fun onCreate(savedInstanceState: Bundle?) {
        (... 생략 ...)
    
    // ActionBarDrawerToggle 버튼 적용.
    toggle = ActionBarDrawerToggle(
          this, 
          // 두 번째 매개변수는 토클 버튼으로 여닫는 드로어 객체이다.
          binding.drawer, 
          // 문자열 객체로 드로어가 열렸을 때의 상태를 표현한다.
          R.string.drawer_opened, 
          R.string.drawer_closed
    )   
        // 토글 버튼으로 사용할 아이콘이 출력되게 한다.
        //   - HomeAsUp 아이콘은 왼쪽 화살표(←) 모양이다.
        //   - toggle.syncState() 함수를 설정하면 내비게이션 아이콘이 나오게 할 수도 있다.
        supportActionBar?.setDisplayHomeAsUpEnabled(true)
        toggle.syncState()
    }

    // 토글 버튼도 액션바 영역에 나오는 아이콘으로 내부에서 메뉴로 취급하기 때문에 onOptionsItemSelected()에서
    // 토글 버튼의 이벤트를 처리해 주어야 화면에 출력된다.
    override fun onOptionsItemSelected(item: MunuItem): Boolean {
        // 이벤트가 토글 버튼에서 발생하면
        if (toggle.onOptionsItemSelected(item)) {
            return true
        }
        return super.onOptionsItemSelected(item)
    }
}
```
